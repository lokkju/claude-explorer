# Frontend-derived backend API contract

This document is the **frontend's view of the backend HTTP contract**.
It was authored by reading `frontend/src/lib/api.ts`, the `frontend/src/hooks/use*.ts`
hooks, `frontend/src/contexts/*.tsx`, the per-component `fetch()` call sites,
and the mocked routes in `frontend/e2e/fixtures.ts` + the Playwright specs.
**Backend `routers/*.py` was NOT consulted.** The whole point of P1 is a
clean frontend-derived contract; verifying clauses against the backend is
what the rest of the plan does.

Every clause carries a stable ID like `BKM-201`, `PREF-PATCH-MERGE`, etc.
Tests cite these IDs in their docstrings (`PLANS/2026.05.08 BACKEND TEST PLAN.md`
"Test naming + docstring discipline").

## Clause-ID conventions

- `<RESOURCE>-<HTTP-CODE>` for primary clauses (`BKM-201`, `PREF-200`, `CFG-200`).
- `<RESOURCE>-<HTTP-CODE>-<ASPECT>` when one status covers multiple semantic
  rules (`PREF-200-MERGE`, `PREF-200-NULL`, `PREF-200-MODE`).
- `<RESOURCE>-SSE-<EVENT_TYPE>` for SSE event clauses
  (`FETCH-SSE-START`, `FETCH-SSE-COMPLETE`, `RFR-SSE-CAPTURE-DONE`).
- `<RESOURCE>-HEADER-<NAME>` for response-header clauses
  (`FETCH-HEADER-CT`, `EXPORT-HEADER-DISPOSITION`).
- `<RESOURCE>-NEG-<ASPECT>` for negative-space (must-not-leak) clauses
  (`CCIMG-NEG-LEAK`, `ATCH-NEG-FS`).

Resource prefixes used: `CFG`, `STATS`, `ORG`, `CONV`, `TREE`, `SRCH`, `EXPMD`,
`EXPMDB`, `EXPPDF`, `EXPALL`, `STAT` (fetch status), `FETCH` (start SSE),
`RFR` (refresh SSE), `PERCONV` (per-conversation refetch), `BKM`, `PREF`,
`CCIMG`, `FILES` (desktop proxy), `ATCH` (local attachment cache).

## 1. Config

### GET /api/config

- **Caller(s):** `frontend/src/lib/api.ts:90` (`api.getConfig()`); used by
  `frontend/src/hooks/useConversations.ts:109` (`useConfig()`); rendered by
  the connection-status modal (`frontend/src/components/ConnectionStatus.tsx`).
- **Query params:** none.
- **Request body:** none.
- **Expected response shape:**
  ```ts
  interface AppConfig { data_dir: string }
  ```
  (`frontend/src/lib/types.ts:188-190`).
- **Expected status code(s):** `200`.
- **Error contract:** any non-200 surfaces as `ApiError(status, await response.text())`
  (`frontend/src/lib/api.ts:23-29`); React Query retries unless 404
  (`frontend/src/lib/queryClient.ts:7-15`). Used as a connectivity probe.
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** none. Cheap probe; the `/config/stats` count was
  intentionally moved out (see `backend/tests/test_config.py:5-17`).
- **Edge cases:** must NOT include `conversation_count` (frontend test
  `frontend/e2e/fixtures-defaults.spec.ts` and `backend/tests/test_config.py:17`
  both pin this).
- **Inferred contract clauses:**
  - `CFG-200` — `GET /api/config` returns 200 with body `{data_dir: string}`.
  - `CFG-200-FIELDS` — body has exactly the `data_dir` field; `conversation_count`
    is NOT present (it lives on `/config/stats`).
  - `CFG-200-DATADIR` — `data_dir` is a non-empty string.

### GET /api/config/stats

- **Caller(s):** `frontend/src/lib/api.ts:92` (`api.getConfigStats()`);
  `frontend/src/hooks/useConversations.ts:99-107` (`useConfigStats()`);
  used in the Settings page (Settings is the only place willing to wait
  for the count).
- **Query params:** none.
- **Request body:** none.
- **Expected response shape:**
  ```ts
  interface AppConfigStats extends AppConfig {
    conversation_count: number
  }
  ```
  (`frontend/src/lib/types.ts:192-194`).
- **Expected status code(s):** `200`.
- **Error contract:** same as `/config` — surfaces via ApiError.
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** counts JSONL/JSON files on disk; documented in `useConfigStats`
  comment as "slower" — frontend caches with `staleTime: Infinity`.
- **Edge cases:** must include both `data_dir` AND `conversation_count` AS AN
  INTEGER (not stringified). Empty store → 0, not null/missing.
- **Inferred contract clauses:**
  - `STATS-200` — returns 200 with body `{data_dir, conversation_count}`.
  - `STATS-200-COUNT-INT` — `conversation_count` is `int` typed
    (e.g. `0`, not `"0"`, not omitted).
  - `STATS-200-COUNT-MATCHES-FIXTURE` — count equals the number of conversation
    files actually present on disk (no hardcoded zero — this was the
    2026-05-07 reference incident in CLAUDE-TESTING.md §7).

## 2. Orgs

### GET /api/orgs

- **Caller(s):** `frontend/src/lib/api.ts:51` (`api.getOrgs()`);
  `frontend/src/hooks/useOrgs.ts:21-26` derives three-state UI from result.
- **Query params:** none.
- **Request body:** none.
- **Expected response shape:**
  ```ts
  interface OrgsResponse {
    authenticated: boolean
    orgs: Org[]
  }
  interface Org { org_id: string; name: string | null; is_primary: boolean }
  ```
  (`frontend/src/lib/types.ts:39-48`).
- **Expected status code(s):** `200` (both authenticated and unauthenticated cases),
  `500` on corrupt credentials (per `useOrgs` doc-comment, `useOrgs.ts:11-13`).
- **Error contract:** 500 → React Query error state → "explicit error banner".
  Body shape on 500 not asserted by frontend (only triggers banner).
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** reads `~/.claude-explorer/credentials.json`.
- **Edge cases:**
  - No creds file → 200 `{authenticated: false, orgs: []}` (NOT 401).
  - Synthetic `_claude_code` org must NOT appear (`is a source, not a tenant`
    per `useOrgs.ts:7-8`).
  - Selector visibility gate: `orgs.length >= 2` AND `authenticated` —
    `useOrgs.ts:35`. (Backend just provides the list; gate is frontend.)
- **Inferred contract clauses:**
  - `ORG-200-AUTH` — creds present + parseable → 200, `authenticated:true`,
    `orgs[]` contains `{org_id, name, is_primary}`.
  - `ORG-200-NOAUTH` — creds file missing → 200, `authenticated:false`,
    `orgs:[]`. (Specifically not 401: the unauthenticated state is a
    legitimate UI mode, not an error.)
  - `ORG-500-CORRUPT` — creds file present but unparseable → 500.
  - `ORG-NEG-CLAUDECODE` — synthetic `_claude_code` UUID is filtered out
    of `orgs[]` even when present in the creds file.
  - `ORG-PRIMARY-FLAG` — exactly one org in `orgs[]` has `is_primary:true`
    when authenticated.

## 3. Conversations

### GET /api/conversations

- **Caller(s):** `frontend/src/lib/api.ts:32-49` (`api.getConversations(filters)`);
  `frontend/src/hooks/useConversations.ts:7-33` (`useConversations`);
  `frontend/src/components/Sidebar.tsx` consumes the returned list.
- **Query params** (all optional, all string-encoded):
  - `search?: string` — title/project substring (frontend ALSO filters
    client-side per `useConversations.ts:23-30`; backend may ignore).
  - `starred?: boolean` (encoded `"true"`/`"false"`).
  - `model?: string`.
  - `source?: 'all' | 'CLAUDE_AI' | 'CLAUDE_CODE'`.
  - `sort?: 'updated_at' | 'created_at' | 'name' | 'project'`.
  - `sort_order?: 'asc' | 'desc'` (snake_case in URL, camelCase in TS).
  - `include_phantom?: 'true'` (only set if true; absent = false).
  - `organization_id?: string` (UUID).
- **Request body:** none.
- **Expected response shape:** `ConversationSummary[]` with the full
  shape at `frontend/src/lib/types.ts:15-36` (notably: `uuid`, `name`,
  `summary`, `model`, `created_at`, `updated_at`, `is_starred`,
  `is_temporary`, `message_count`, `human_message_count`,
  `has_branches`, `source: 'CLAUDE_AI'|'CLAUDE_CODE'`,
  `project_path`, `project_name`, `git_branch`, optional
  `organization_id`, `organization_name`, optional `subagents[]`).
- **Expected status code(s):** `200`.
- **Error contract:** non-200 → ApiError; React Query retries 5× with
  exponential backoff (`queryClient.ts:7-15`).
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** none mutating; backend reads filesystem.
- **Edge cases:**
  - Empty corpus → `[]`, not `null` / 404.
  - `organization_id=<unknown-uuid>` → `[]`, not 404.
  - `source=CLAUDE_CODE` filters to claude-code-sourced conversations only.
- **Inferred contract clauses:**
  - `CONV-LIST-200` — returns 200 with a JSON array.
  - `CONV-LIST-EMPTY` — empty store → `[]`.
  - `CONV-LIST-FIELDS` — every element has the required `ConversationSummary`
    fields; optional fields can be `null` but the keys exist.
  - `CONV-LIST-FILTER-SOURCE` — `?source=CLAUDE_CODE` returns ONLY
    `source==CLAUDE_CODE` rows (and same for `CLAUDE_AI`).
  - `CONV-LIST-FILTER-STARRED` — `?starred=true` returns only rows where
    `is_starred==true`; `?starred=false` returns only rows where
    `is_starred==false`. Boolean is encoded as the literal string
    `"true"`/`"false"` (`api.ts:41`).
  - `CONV-LIST-FILTER-MODEL` — `?model=<id>` filters to rows whose
    `model` field exactly matches (`api.ts:42`).
  - `CONV-LIST-FILTER-ORG` — `?organization_id=<uuid>` returns only
    rows whose `organization_id` matches.
  - `CONV-LIST-FILTER-ORG-UNKNOWN` — unknown `organization_id` → `[]`.
  - `CONV-LIST-INCLUDE-PHANTOM` — `?include_phantom=true` includes
    "phantom" conversations (e.g. legacy untagged JSONs) that are
    excluded by default. Param is omitted entirely when false
    (`api.ts:45`).
  - `CONV-LIST-SORT` — `?sort=updated_at&sort_order=desc` is the default
    used by the sidebar; changing either changes order.

### GET /api/conversations/{uuid}

- **Caller(s):** `frontend/src/lib/api.ts:53-64` (`api.getConversation(uuid, leaf?)`);
  `frontend/src/hooks/useConversations.ts:35-44` (`useConversation`).
- **Path params:** `uuid: string` (conversation UUID).
- **Query params:** `leaf?: string` (URL-encoded message UUID for the active branch).
- **Request body:** none.
- **Expected response shape:** `ConversationDetail` (extends `ConversationSummary`
  with `messages: Message[]`, `current_leaf_message_uuid: string`,
  optional `file_path`, optional `compact_markers[]`) at
  `frontend/src/lib/types.ts:117-122`.
- **Expected status code(s):** `200` on hit; `404` on unknown UUID.
- **Error contract:** 404 → `throw new ApiError(404, 'Conversation not found')`
  (`api.ts:57-58`); body shape on 404 is `{detail: string}` per backend
  convention but the frontend reads only the status (the message it shows
  the user comes from the `getConversation` line).
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** none.
- **Edge cases:**
  - `?leaf=<msg-uuid>` selects an alternate branch; backend's `messages`
    array reflects that path.
  - Backend should NOT cache a stale `messages` array if the JSON file
    changed on disk.
- **Inferred contract clauses:**
  - `CONV-200` — known UUID → 200 with full `ConversationDetail`.
  - `CONV-404` — unknown UUID → 404 with `{detail: string}`.
  - `CONV-200-LEAF` — `?leaf=<uuid>` produces a `messages` array whose
    walk ends at `current_leaf_message_uuid == <uuid>`.
  - `CONV-200-DEFAULT-LEAF` — no `?leaf` → backend chooses the most-recent
    leaf; frontend trusts `current_leaf_message_uuid` to render.

### GET /api/conversations/{uuid}/tree

- **Caller(s):** `frontend/src/lib/api.ts:66-67` (`api.getConversationTree(uuid)`);
  `frontend/src/hooks/useConversations.ts:46-53` (`useConversationTree`);
  used by branch-switching UI.
- **Path params:** `uuid`.
- **Query params:** none.
- **Request body:** none.
- **Expected response shape:**
  ```ts
  interface ConversationTree {
    uuid: string
    root_messages: MessageNode[]
    active_path: string[]   // ordered list of message UUIDs
  }
  interface MessageNode { message: Message; children: MessageNode[] }
  ```
  (`frontend/src/lib/types.ts:124-133`).
- **Expected status code(s):** `200` on hit; `404` on unknown UUID.
- **Error contract:** 404 → ApiError; React Query does NOT retry 404
  (`queryClient.ts:9-11`).
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** none.
- **Edge cases:**
  - `active_path` ordering MUST match the backend's notion of the live
    branch root → leaf.
  - Branched conversations: a single node can have multiple `children`.
  - Linear conversations still return a tree (single chain of nodes).
- **Inferred contract clauses:**
  - `TREE-200` — known UUID → 200 with `{uuid, root_messages, active_path}`.
  - `TREE-404` — unknown UUID → 404.
  - `TREE-200-ACTIVE` — `active_path[-1]` equals the conversation's
    `current_leaf_message_uuid`.
  - `TREE-200-NODES` — every `MessageNode` has `message` and `children`
    (children may be `[]`).
  - `TREE-200-BRANCHED` — for a conversation with branching, at least
    one node has `children.length > 1`.

## 4. Search

### GET /api/search

- **Caller(s):** `frontend/src/lib/api.ts:69-88` (`api.search(...)`);
  `frontend/src/hooks/useConversations.ts:55-97` (`useSearch`);
  command palette / search panel.
- **Query params:**
  - `q: string` (REQUIRED; `useSearch` only fires when `length >= 2`).
  - `source?: 'all' | 'CLAUDE_AI' | 'CLAUDE_CODE'` (omitted when `'all'`).
  - `context_size?: 'snippet' | 'full'` (omitted when `'snippet'`).
  - `sort?: 'updated_at' | 'created_at' | 'name' | 'project'` (omitted when default).
  - `sort_order?: 'asc' | 'desc'` (omitted when `'desc'`).
  - `conversation_uuid?: string` — scope to one conversation.
  - `project_path?: string` — scope to project (overridden by
    `conversation_uuid` per `api.ts:82` semantics).
  - `bookmarks?: string` — comma-joined list of bookmark IDs.
- **Request body:** none.
- **Expected response shape:**
  ```ts
  interface SearchResult {
    conversation_uuid: string
    conversation_name: string
    conversation_updated_at: string
    conversation_created_at: string
    project_name: string | null
    matching_messages: MessageSnippet[]
  }
  interface MessageSnippet {
    message_uuid: string
    sender: string
    snippet: string
    match_start: number
    match_end: number
    created_at: string | null
  }
  ```
  (`frontend/src/lib/types.ts:137-153`).
- **Expected status code(s):** `200`; `422` when `q` is missing.
- **Error contract:** 422 on missing `q` (validated in `backend/tests/test_search.py:9`).
- **Expected response headers:** `content-type: application/json`.
- **Side effects:** none mutating; backend may cache.
- **Edge cases:**
  - Empty `q` (`""`) → backend currently returns `200` OR `422`
    (current backend test treats both as acceptable, `test_search.py:34-35`);
    frontend short-circuits (`debouncedQuery.length >= 2`) so neither
    case actually fires.
  - `match_start`/`match_end` are byte/character offsets into `snippet`,
    not into the full message text.
  - `bookmarks` CSV: empty list → not sent (`api.ts:84-86`).
- **Inferred contract clauses:**
  - `SRCH-200` — valid `q` → 200 with `SearchResult[]`.
  - `SRCH-422` — missing `q` → 422.
  - `SRCH-200-EMPTY` — `q` matches nothing → `[]`, not 404.
  - `SRCH-200-OFFSETS` — `match_start <= match_end <= len(snippet)`.
  - `SRCH-200-SCOPE-CONV` — `?conversation_uuid=X` returns ONLY
    rows where `conversation_uuid==X` (or `[]`).
  - `SRCH-200-SCOPE-PROJ` — `?project_path=X` filters to that project's
    conversations.
  - `SRCH-200-SCOPE-CONV-WINS` — both `conversation_uuid` and `project_path`
    set → `conversation_uuid` is authoritative
    (frontend behavior: `api.ts:82-83` doesn't strip but backend should
    treat it as more specific; verify in P3).
  - `SRCH-200-BOOKMARKS-CSV` — `?bookmarks=a,b,c` filters to bookmark IDs;
    unknown IDs → `[]`.
  - `SRCH-200-SOURCE` — `?source=CLAUDE_CODE` returns only CC-sourced
    snippets.
  - `SRCH-200-CONTEXT-SIZE` — `?context_size=full` returns longer
    `snippet` text (full message body) than the default `snippet`
    behavior (`api.ts:79`).
  - `SRCH-200-SORT` — `?sort=name&sort_order=asc` returns rows ordered
    by `conversation_name` ascending (and similarly for `created_at`,
    `updated_at`, `project`); default is `updated_at desc` (`api.ts:80-81`).
  - `SRCH-200-TOOL-USAGE` — search matches text in `tool_use.input` and
    `tool_result.content` blocks (already tested in
    `test_search_tool_usage.py`).

### GET /api/conversations/{uuid}/export/markdown

- **Caller(s):** `frontend/src/lib/api.ts:94-95` (`api.exportMarkdown(uuid, showToolCalls)`);
  triggered from the conversation header "Markdown" button (inline mode).
- **Path params:** `uuid`.
- **Query params:** `include_tools?: 'true' | 'false'` (defaults to `'true'`
  per the api function signature; passed verbatim).
- **Request body:** none.
- **Expected response shape:** Markdown text body — speaker labels
  (`**You:**`, `**Claude:**`), heading, optional `<details>` blocks for
  tool calls.
- **Expected status code(s):** `200` on hit; `404` on unknown UUID.
- **Error contract:** raw `Response` (frontend doesn't unwrap;
  treats as a download trigger).
- **Expected response headers:**
  - `content-type: text/markdown; charset=utf-8` (per
    `frontend/e2e/exports.spec.ts:67-68`).
  - `content-disposition: attachment; filename="<conv-name>.md"` (per
    `frontend/e2e/exports.spec.ts:69`).
- **Side effects:** none.
- **Edge cases:**
  - `include_tools=false` → tool blocks STRIPPED from body.
  - `include_tools=true` → tool blocks rendered as `<details>` collapsible.
  - `TOOL_PLACEHOLDER` strings stripped outside fenced code; PRESERVED
    inside fences (CLAUDE-TESTING.md §5.4 reference incident).
  - Image refs are absolute `/api/...` URLs (not bundled).
- **Inferred contract clauses:**
  - `EXPMD-200` — known UUID → 200 with markdown body.
  - `EXPMD-404` — unknown UUID → 404.
  - `EXPMD-HEADER-CT` — `content-type` matches `text/markdown`.
  - `EXPMD-HEADER-DISP` — `content-disposition` is `attachment;`
    with a filename derived from conversation name.
  - `EXPMD-INCLUDE-TOOLS-TRUE` — `?include_tools=true` body contains
    tool body (`web_search`, `web result body`, etc.).
  - `EXPMD-INCLUDE-TOOLS-FALSE` — `?include_tools=false` body does
    NOT contain tool blocks (negative-space).
  - `EXPMD-PLACEHOLDER-STRIP-OUT` — `TOOL_PLACEHOLDER_TEXT` outside
    fences is stripped.
  - `EXPMD-PLACEHOLDER-KEEP-IN` — `TOOL_PLACEHOLDER_TEXT` inside fences
    is preserved.
  - `EXPMD-HEADER-NAME` — body opens with `# <conversation_name>`.
  - `EXPMD-SPEAKERS` — body contains `**You:**` and `**Claude:**` labels.

### GET /api/conversations/{uuid}/export/markdown-bundle

- **Caller(s):** `frontend/src/lib/api.ts:98-105` (`api.exportMarkdownBundle(uuid, showToolCalls, dialect)`);
  `MarkdownExportDialog.tsx` (Bundle CommonMark / Bundle Obsidian modes).
- **Path params:** `uuid`.
- **Query params:**
  - `include_tools?: 'true' | 'false'`.
  - `dialect: 'commonmark' | 'obsidian'` (defaults to `'commonmark'`
    in the api signature).
- **Request body:** none.
- **Expected response shape:** zip archive containing `conversation.md`,
  `images/` directory, and `attachments/` directory.
- **Expected status code(s):** `200` on hit; `404` on unknown UUID;
  `422` on unknown `dialect` (`backend/tests/test_export_bundle.py:226`).
- **Error contract:** 422 expected for unknown dialect; 404 for unknown UUID.
- **Expected response headers:**
  - `content-type` is `application/zip` (or similar binary).
  - `content-disposition: attachment; filename="..."`.
- **Side effects:** none.
- **Edge cases:**
  - `dialect=commonmark` → image refs are `![alt](images/x.png)`.
  - `dialect=obsidian` → image refs are `![[images/x.png]]` (wikilinks).
  - Inline base64 images decoded to `images/<n>.png`.
  - `[Image: source: <abs-path>]` markers resolved + bundled.
  - PDF attachments bundled under `attachments/` (per
    `test_export_bundle_attachments.py`).
- **Inferred contract clauses:**
  - `EXPMDB-200` — valid request → 200 with zip body.
  - `EXPMDB-404` — unknown UUID → 404.
  - `EXPMDB-422-DIALECT` — `?dialect=foobar` → 422.
  - `EXPMDB-ZIP-MEMBERS` — zip contains `conversation.md` at root,
    plus `images/` and (when applicable) `attachments/`.
  - `EXPMDB-COMMONMARK-LINKS` — commonmark dialect produces
    `![alt](images/...)` links.
  - `EXPMDB-OBSIDIAN-LINKS` — obsidian dialect produces `![[images/...]]`
    wikilinks.
  - `EXPMDB-INLINE-IMG` — inline base64 image bytes decoded into the
    zip with deterministic filename.
  - `EXPMDB-MARKER-IMG` — `[Image: source: <path>]` markers resolved
    and bytes bundled.
  - `EXPMDB-PDF-ATTACHMENT` — PDF attachments bundled under `attachments/`.

### GET /api/fetch/refresh (SSE — Build-9 capture + fetch pipeline)

- **Caller(s):** `frontend/src/lib/api.ts:151-156` (`api.startRefresh(incremental)`);
  `frontend/src/components/fetch/FetchToast.tsx::useRefreshPipeline` —
  drives the sidebar Refresh button (Build-9 spec in CLAUDE.md).
- **Query params:** `incremental?: 'false'`.
- **Request body:** none.
- **Expected response shape:** SSE stream. Same wire shape as `/fetch/start`
  (data-only frames + `:` comments).
- **Expected status code(s):** `200` on first request; `409 Conflict`
  on a second concurrent request (per `_refresh_in_progress` flag,
  CLAUDE.md "Web UI Refresh button" section).
- **Error contract:** SSE `error` event uses the **NEW envelope**:
  ```
  data: {"type":"error", "kind":"AUTH"|"TRANSIENT"|"TERMINAL",
         "retryable":true|false, "message":"..."}\n\n
  ```
  (Per `FetchToast.tsx:8-29`.) On 409, response body is JSON
  `{detail: "in progress"}` (per `test_refresh_pipeline.py:325-327`).
- **Expected response headers:** same as `/fetch/start`.
- **Side effects:** May invoke `fetcher.playwright_capture.capture_credentials`
  in-process; on success persists `credentials.json` (atomic write,
  `0o600`) before the fetch phase. Module-level `_refresh_in_progress`
  flag is set on entry, cleared in `finally`.
- **Edge cases:**
  - Creds present and valid → no capture; events: `start` → ≥1
    `progress` → `complete`.
  - Creds missing → `capture_start` → ≥0 `capture_waiting_login` →
    `capture_done` → fetch phase.
  - Capture timeout / browser-closed → `error` event with
    `Capture failed: ...` message.
  - Post-capture fetch still 401s → ONE `error` event, NO retry-capture
    loop (Build-9 spec: "at most once per request").
  - 25-second heartbeat: `capture_waiting_login` may fire during the
    5-minute login wait.
  - Cancellation: client disconnects mid-stream → `_refresh_in_progress`
    cleared by `finally`; subsequent request returns 200.
- **Inferred contract clauses:**
  - `RFR-200` — first request → 200 SSE.
  - `RFR-409` — concurrent second request → 409 `{detail: string}`.
  - `RFR-HEADER-CT` — `content-type` starts with `text/event-stream`.
  - `RFR-HEADER-CACHE` — `cache-control: no-cache`.
  - `RFR-WIRE-DATA-ONLY` — events are `data:` frames; no `event:` lines.
  - `RFR-WIRE-COMMENTS` — `:`-prefixed keep-alive comments are emitted
    during the capture-wait phase and MUST be ignored by parsers.
  - `RFR-SSE-CAPTURE-SKIP` — creds present → no `capture_*` events;
    sequence is `start` → `progress` → `complete`.
  - `RFR-SSE-CAPTURE-START` — creds missing → first event is
    `capture_start`.
  - `RFR-SSE-CAPTURE-WAIT` — `capture_waiting_login` may appear ≥0
    times between `capture_start` and `capture_done`.
  - `RFR-SSE-CAPTURE-DONE` — successful capture → `capture_done`
    BEFORE any `start` event.
  - `RFR-SSE-CAPTURE-ERROR` — capture failure → `error` event with
    `Capture failed:` prefix; no fetch phase follows. Envelope is
    legacy plain-message (per `fetch.py:992-1000` and the
    Ambiguities table below — for now the frontend tolerates either
    envelope on capture failures).
  - `RFR-SSE-RECAPTURE-401-FAILS` — post-capture fetch still 401 →
    exactly ONE capture per request, then `error`, no `complete`.
  - `RFR-SSE-AUTH-CLASSIFY` — error events for auth class carry
    `kind:"AUTH"`, `retryable:false`.
  - `RFR-SSE-TRANSIENT-CLASSIFY` — DNS / network errors carry
    `kind:"TRANSIENT"`, `retryable:true`.
  - `RFR-SSE-TERMINATES` — stream closes after `complete` or `error`.
  - `RFR-CRED-PERMS` — credentials persisted by the capture phase have
    permissions `0o600` (per `test_refresh_pipeline.py:332-362`).
  - `RFR-FLAG-RESET` — `_refresh_in_progress` is False after stream
    completion or cancellation; a subsequent request returns 200.

### PATCH /api/preferences

- **Caller(s):** `frontend/src/hooks/usePreferences.ts:48-58` (`patchPrefs`);
  `frontend/src/contexts/FilterContext.tsx:391-401, :421-427` (filter
  migration).
- **Query params:** none.
- **Request body:**
  ```ts
  { data: Record<string, unknown> }
  ```
  Top-level keys are merged into existing data (per-key overwrite).
  An explicit `null` value clears that key.
- **Expected response shape:** the same envelope as GET, reflecting
  the merged state (`usePreferences.ts:50-58`).
- **Expected status code(s):** `200`.
- **Expected response headers:** `content-type: application/json`.
- **Side effects:**
  - Atomic write: `tmp` file with `0o600` chmod, then `os.replace`.
  - On-disk file ends with permissions `0o600`.
- **Edge cases:**
  - **Per-key overwrite (deep-merge top-level):** PATCH `{theme:"dark"}`
    on an existing blob with `{theme:"light", lang:"en"}` → final

    activeFilterIds:null}` → those keys absent in subsequent GET;
    untouched siblings preserved.

    `a` set, `b` cleared, `c` set; siblings `d, e, f` untouched.
  - **Empty body `{data:{}}`:** no-op.
  - **Concurrent PATCHes** must not corrupt the blob — final state
    has every PATCH's data.
  - **Unknown keys tolerated** (no `extra='forbid'` today; future
    plan introduces `STRICT-PREF-422`).
- **Inferred contract clauses:**
  - `PREF-PATCH-200` — returns 200 with merged envelope.
  - `PREF-PATCH-MERGE` — top-level keys merged into existing data;
    untouched keys preserved (negative-space).
  - `PREF-PATCH-NULL` — explicit `null` clears the matching key.
  - `PREF-PATCH-MULTIKEY` — `{a:X, b:null, c:Y}` applies all three
    independently; explicit-`null` on `b` does NOT clear `a` or `c`.
  - `PREF-PATCH-EMPTY` — `{data:{}}` is a no-op (state unchanged).
  - `PREF-PATCH-CONCURRENT` — N parallel PATCHes don't corrupt;
    final state contains every PATCH's data.
  - `PREF-PATCH-PERMS` — on-disk `preferences.json` has mode `0o600`.
  - `PREF-PATCH-ATOMIC-RECOVERY` — `os.replace` failure leaves the
    original file byte-identical and no `*.tmp` leaked.
  - `PREF-PATCH-UNKNOWN-KEY` — unknown keys accepted (no 422 today).
  - `PREF-PATCH-NEG-NO-CLOBBER` — PATCH on key A does NOT alter
    keys B/C/D (negative-space).

  - Invalid `org_id`/`file_uuid` shapes — backend should still
    reject before making the upstream call (verify in P3).
- **Inferred contract clauses:**
  - `FILES-200` — valid request with valid creds → 200 with image
    bytes, `cache-control` set.
  - `FILES-401` — upstream 401 → 401 `{detail: SESSION_EXPIRED hint}`.
  - `FILES-404` — upstream 404 → 404 `{detail: "image not found"}`.
  - `FILES-503-NOAUTH` — no creds → 503 with capture hint.
  - `FILES-PROXY-COOKIE` — outbound request carries `sessionKey`,
    `__cf_bm`, `cf_clearance` cookies and impersonates Chrome.

### GET /api/attachments/{conv_uuid}/{file_uuid}/{variant}

- **Caller(s):** rendered by `<img>` tags whose `src` is set by the
  backend at fetch time. Frontend mocks at
  `frontend/e2e/fixtures.ts:343-349`.
- **Path params:**
  - `conv_uuid`: conversation UUID.
  - `file_uuid`: file UUID.
  - `variant`: one of `thumbnail | preview | original | document`
    (per `backend/tests/test_desktop_attachments_full.py:274-281`).
- **Query params:** none.
- **Request body:** none.
- **Expected response shape:** binary file bytes.
- **Expected status code(s):**
  - `200` on hit.
  - `400` on path-traversal attempt or unknown variant (current code
    returns 400 OR 404 for unknown variant — see Ambiguities).
  - `404` on cache miss.
- **Error contract:** `{detail: string}`.
- **Expected response headers:**
  - `content-type`: `image/...` for image variants, `application/pdf`
    for PDFs, `text/...` for text documents.
- **Side effects:** none mutating; reads
  `~/.claude-explorer/files/<conv>/<file>/<variant>.<ext>`.
- **Edge cases:**
  - `conv_uuid="../../etc"` → MUST be rejected with 400 and no fs
    touch. (Plan P2.6 calls this out as a real gap: current code
    returns 404 only because the resolved escape target has no such
    dir under the resolved root. The plan ships a `relative_to`
    check on `file_dir` after `resolve()`.)
  - `..` in `file_uuid` → same 400.
  - `variant="../etc"` → 400 (already enforced by allowlist).
  - Variant outside the allowlist (`evil`) → 400 OR 404 (current
    backend ambiguity; see table below).
- **Inferred contract clauses:**
  - `ATCH-200` — valid request with cached file → 200 with bytes.
  - `ATCH-200-PDF` — PDF document → `content-type: application/pdf`.
  - `ATCH-200-TXT` — text document → `content-type: text/...`.
  - `ATCH-200-IMG` — image variant → `content-type: image/...`.
  - `ATCH-400-CONV-TRAVERSAL` — `conv_uuid` containing `..` → 400
    (currently 404; will be fixed in P2.6).
  - `ATCH-400-FILE-TRAVERSAL` — `file_uuid` containing `..` → 400.
  - `ATCH-400-UNKNOWN-VARIANT` — variant not in allowlist → 400
    (or 404; current `test_desktop_attachments_full.py:280-281`
    tolerates either — see Ambiguities).
  - `ATCH-NEG-FS` — path-traversal attempts MUST NOT touch the
    filesystem outside the attachments root (negative-space).
  - `ATCH-404-MISSING` — known variant + valid uuids + no cached
    file → 404.
