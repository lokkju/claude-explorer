# Offline fallback for Desktop attachment proxy (`/api/<org>/files/...`)

**Status:** Ready to implement (in the other CC session).
**Owner:** other session.
**Date:** 2026-05-29.
**Why this exists:** Part 2 claims "images keep working even when you're offline from claude.ai"
(`articles/part_2_web_app.md:480`, `articles/part_2_web_app_userdoc.md:247`). That holds for Claude
Code images but NOT for Claude Desktop attachments. This plan makes the claim true rather than
softening the article.

## Root cause

`backend/routers/files.py::_proxy` (lines ~84–167) serves Desktop attachment thumbnails/previews.
Frontend `<img src="/api/<org>/files/<file>/thumbnail">` → `get_thumbnail`/`get_preview` (`:179`,
`:189`) → `_proxy(org_id, file_uuid, variant)`.

`_proxy` is **upstream-first**:

1. `_load_session_cookies()` — raises **503** if no creds on disk.
2. `curl_requests.get(claude.ai URL, ...)` — on a **network failure (offline) this throws** and is
   caught at `:99–101`, which raises **502** unconditionally.
3. Only when the upstream call **succeeds with HTTP 404** (`:103`) does the code consult the local
   on-disk cache (`:119–141`): a metachar-guarded glob `_attachments_root().glob("*/{file_uuid}/{variant}.*")`,
   served via `FileResponse` if found.

So when you are offline, step 2 throws → 502, and the cache at step 3 is **never reached** — even
though the bulk fetcher already cached `thumbnail`/`preview`/`original` bytes on disk at fetch time
(`fetcher/bulk_fetch.py:469–489`, written under `~/.claude-explorer/files/<conv>/<file>/<variant><ext>`).

CC images are unaffected: `/api/cc-image` (`files.py:221+`) serves straight from disk with a
permanent-cache fallback, so it already works offline.

## The fix

Reach for the already-cached bytes whenever the upstream is unreachable, not only on a 404 response.

### Step 1 — extract the cache lookup into a helper (DRY)

The 404 branch (`:119–145`) already contains the exact lookup we need. Pull it into a module-level
helper so both the 404 branch and the new offline branch share it. **Preserve the Hunt #4 glob-metachar
guard verbatim** — it's a real path-traversal defense (a `file_uuid` of `**` would otherwise walk the tree).

```python
def _serve_from_local_cache(file_uuid: str, variant: str) -> FileResponse | None:
    """Return the bulk-fetcher-cached attachment for (file_uuid, variant), or None.

    Hunt #4 (API boundaries audit): reject glob metacharacters in file_uuid so a
    crafted `**`/`*`/`?`/`[]` can't escape the attachments root via the glob below.
    """
    if any(ch in file_uuid for ch in "*?[]"):
        return None
    try:
        cached = [
            m for m in _attachments_root().glob(f"*/{file_uuid}/{variant}.*")
            if m.is_file()
        ]
    except OSError:
        return None
    if not cached:
        return None
    media_type = mimetypes.guess_type(str(cached[0]))[0] or "application/octet-stream"
    return FileResponse(cached[0], media_type=media_type, headers={
        "cache-control": "public, max-age=86400, stale-while-revalidate=604800",
    })
```

Rewrite the existing 404 branch to call it:

```python
    if upstream.status_code == 404:
        served = _serve_from_local_cache(file_uuid, variant)
        if served is not None:
            logger.info("proxy_local_fallback", extra={"file_uuid": file_uuid, "variant": variant, "reason": "upstream_404"})
            return served
        raise HTTPException(status_code=404, detail="image not found upstream and no local cache")
```

### Step 2 — fall back to the cache on a network failure (the offline fix)

Change the `except` branch at `:97–101` so it tries the cache before giving up:

```python
    try:
        upstream = curl_requests.get(url, cookies=cookies, impersonate="chrome", timeout=15)
    except Exception as exc:
        served = _serve_from_local_cache(file_uuid, variant)
        if served is not None:
            logger.info("proxy_local_fallback", extra={"file_uuid": file_uuid, "variant": variant, "reason": "upstream_unreachable"})
            return served
        logger.warning("image proxy upstream fetch failed for %s: %s", url, exc)
        raise HTTPException(status_code=502, detail=f"upstream fetch failed: {exc}") from exc
```

That is the whole behavioral change: **offline + bytes cached → serve them; offline + not cached → 502 as before.**

### Decision to make before coding: also cover the no-creds (503) path?

`_load_session_cookies()` raises **503** at `:94` *before* any upstream attempt when creds are missing.
A user whose creds were deleted (but who fetched earlier) still has cached attachment bytes. Two options:

- **(Recommended, minimal)** Leave the 503 path as-is. The common offline case is "creds present, network
  down," which Step 2 covers. Smaller blast radius.
- **(Optional, more robust) Cache-first.** Move `_serve_from_local_cache(file_uuid, variant)` to the very
  top of `_proxy`, before `_load_session_cookies()` / the upstream call. Cache hit → serve immediately
  (no creds, no network, and faster online too, since file attachments are immutable / content-addressed
  by `file_uuid`). Cache miss → fall through to the existing proxy path. This subsumes Step 2 and also
  fixes the no-creds case, but it changes online behavior (serves cached bytes instead of re-fetching),
  so it needs the broader e2e sweep below. Pick this only if you want the online speedup too.

Lead with the minimal fix unless you specifically want cache-first.

## What NOT to change

- Keep the **401/403** branch (`:146–153`) returning the "session expired, re-capture" message — that's a
  reachable-but-unauthorized signal, distinct from offline, and the user needs the re-capture nudge.
- Keep the metachar guard exactly as-is (it's load-bearing security, not cleanup).
- Don't touch `/api/cc-image` — it already serves offline.
