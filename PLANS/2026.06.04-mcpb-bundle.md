# MCPB bundle for `claude-explorer mcp`

**Author:** Claude Code session 2026-06-04
**Status:** Draft — ready for execution
**Scope:** Package the existing stdio MCP server (`mcp_server.server` via `claude-explorer mcp`) as an installable Claude Desktop extension so a non-CLI user can drag a single file into Settings → Extensions and have the 5 conversation-archive tools (`list_sessions`, `list_projects`, `get_session_outline`, `get_messages`, `export_session`) live in their Claude Desktop instance.

## Why this is the right framing

The repo already ships a stdio MCP server: `mcp_server/server.py` (676 LOC, FastMCP) exposes 5 tools over a stdio transport, wired into the CLI as `claude-explorer mcp`. So the work is **not** "build an MCP server" — it's "produce a bundle artifact Claude Desktop can install in one drag-drop, with a `manifest.json` that tells the host how to launch the existing server."

## Format facts (verified 2026-06-04)

The format Anthropic published as **DXT (`.dxt`)** has been **renamed to MCPB (`.mcpb`)**. Both names refer to the same artifact:

- `.dxt` → `.mcpb` file extension
- `dxt` CLI → `mcpb` CLI (`npm install -g @anthropic-ai/mcpb`)
- Spec lives at https://github.com/anthropics/dxt (repo not yet renamed at time of writing)
- Current spec: **`manifest_version: "0.3"`** baseline, **`"0.4"`** required for the `uv` runtime type
- Claude Desktop ships **Node.js**, but **not Python** — Python servers either bundle their own deps in `server/lib/` or use the `uv` runtime so the host manages Python + dep install

The Medium article narrative will need to mention both names. The bundle file we ship is `claude-explorer.mcpb`.

## Runtime choice: **UV, not bundled-Python**

`server.type: "uv"` is the correct choice for this project. Rationale:

1. **pydantic is a compiled dep.** The spec README explicitly calls out pydantic as the canonical example of "cannot portably bundle compiled dependencies" via the traditional `server/lib/` approach. The MCP server pulls pydantic transitively through FastMCP, backend.models, and backend.config — there is no version of this bundle that avoids pydantic.
2. **UV runtime delegates Python + dep install to the host.** Claude Desktop runs `uv run --directory ${__dirname} server.py` on first launch; UV reads our bundled `pyproject.toml`, resolves transitively, compiles wheels for the user's platform, and caches them. No per-platform binary bundling for us.
3. **`manifest_version: "0.4"`** unlocks `server.type: "uv"` and we use it.

Trade-off: first launch is slower than a pre-bundled Node extension because UV is doing a resolve + install. This is acceptable for a developer-facing extension; we note it in the description and in the README.

## Things that stay OUT of the bundle (do NOT include)

The MCP path's import closure is much narrower than the full project. Exclude:

- **`fetcher/`** — mitmproxy + playwright + curl_cffi, hundreds of MB of unused capability. Capture stays CLI-side.
- **`frontend/`** — React SPA; the MCPB user never sees it.
- **`backend/routers/`** — FastAPI routers; the MCP server doesn't go through HTTP.
- **`backend/main.py`** — FastAPI app + DYLD bootstrap; not on the MCP code path.
- **`backend/exporters/pdf.py`'s weasyprint hard-dep at runtime** — `from weasyprint import HTML` is INSIDE function bodies (`backend/exporters/pdf.py:587`), so importing the module doesn't load weasyprint. **Verified 2026-06-04.** The MCPB bundle can omit `weasyprint`, `cairo`, `pango`, `libffi` entirely. PDF export is a future tool; if/when added, it ships as a separate extension or guards on weasyprint at runtime.
- **`backend/cc_watcher.py`** — long-running watchdog process. Out of MCP scope.
- **`fastapi`, `uvicorn[standard]`, `playwright`, `mitmproxy`, `curl_cffi`** — none in the MCP closure.

This pruning is non-negotiable. The transitive dep set MUST be derived by static-import analysis of `mcp_server/server.py`, not by copying `pyproject.toml`'s full `dependencies` list.

### 3. Stripped `pyproject.toml` at the bundle root

UV reads this file to resolve and install. It MUST be the bundle's `pyproject.toml`, NOT the project's. Concrete shape:

```toml
[project]
name = "claude-explorer-mcp"
version = "1.1.0"
description = "MCP server bundle for the claude-explorer conversation archive"
requires-python = ">=3.11"
dependencies = [
    "fastmcp>=3.0",
    "pydantic>=2.0",
    "platformdirs>=4.0",
    "portalocker>=2.8",
    "orjson>=3.10",
]
```

**Critical:** this dep list must be derived from static-import analysis of `mcp_server/server.py` + its transitive `backend.*` imports — NOT copied from the main `pyproject.toml`. Build-time check in `scripts/build-mcpb.py` (§5):

```python
import ast, pathlib

ALLOWED = {"fastmcp", "pydantic", "platformdirs", "portalocker", "orjson"}
FORBIDDEN = {"fastapi", "uvicorn", "playwright", "mitmproxy", "curl_cffi", "weasyprint", "watchdog"}

for src in pathlib.Path("server/").rglob("*.py"):
    tree = ast.parse(src.read_text())
    for node in ast.walk(tree):
        if isinstance(node, ast.Import):
            for alias in node.names:
                root = alias.name.split(".")[0]
                if root in FORBIDDEN:
                    raise SystemExit(f"FORBIDDEN dep in {src}: {alias.name}")
        elif isinstance(node, ast.ImportFrom) and node.level == 0:
            root = (node.module or "").split(".")[0]
            if root in FORBIDDEN:
                raise SystemExit(f"FORBIDDEN dep in {src}: from {node.module}")
```

If a future change pulls FastAPI into the MCP closure unintentionally, the build fails loudly rather than silently shipping a 400 MB bundle.

### 5. Build script: `scripts/build-mcpb.py`

Lives in the main repo, NOT in the bundle. Outputs `dist/claude-explorer-VERSION.mcpb`. Tasks:

1. Clean `build/mcpb/` work dir.
2. Static-import-walk `mcp_server.server` → produce the selective `server/backend/` copy.
3. Copy `mcp_server/` (minus `tests/`) into `server/mcp_server/`.
4. Write `server/main.py`:
   ```python
   #!/usr/bin/env python3
   from mcp_server.server import main
   if __name__ == "__main__":
       main()
   ```
5. Write the stripped `pyproject.toml` at bundle root.
6. Write `manifest.json` from a template + read `version` from `mcp_server/__init__.py` (single source of truth — wire on commit 0).
7. Copy `icon.png` (need to source — see §10).
8. Write `.mcpbignore`:
   ```
   .venv/
   __pycache__/
   *.pyc
   .pytest_cache/
   .mypy_cache/
   *.egg-info/
   uv.lock
   tests/
   ```
9. Run `mcpb pack build/mcpb/ dist/claude-explorer-${VERSION}.mcpb` (requires `npm install -g @anthropic-ai/mcpb` — first-time dev needs to run that, documented in the build script's docstring).
10. Verify the artifact: unzip into a temp dir, parse `manifest.json`, run `mcpb validate` if available.

The script is a single file, ~150 LOC, no third-party deps beyond `tomllib` (stdlib 3.11+).

### 10. Project-specific gotchas (verified)

- **Lazy-imported weasyprint is SAFE to exclude.** Top-level imports in `backend/exporters/pdf.py` don't pull weasyprint; the `from weasyprint import HTML` line is inside `create_pdf`'s function body at `backend/exporters/pdf.py:587`. The MCP server never calls `create_pdf`. Verified by grep.
- **Search index builds lazily on first query.** `backend/search_index.py:get_search_index()` is a singleton; first query against a fresh `~/.claude-explorer/search-index.sqlite` triggers full FTS5 build. Time-on-first-query expectations should be in the manifest's `long_description` (something like "First search may take a few seconds while the index initializes").
- **CC watcher and fetcher are OUT.** `claude-explorer install-watcher` and `claude-explorer fetch` remain CLI-only. The MCPB is **read-only on existing data**. The `long_description` MUST say this, so users don't install the MCPB expecting it to fetch.
- **Corrupt-config safe mode still applies.** `backend/config.py:Settings` returns `config_corrupt_reason` on a broken `~/.claude-explorer/config.json`; readers still work. The MCP server has no writers, so the `_refuse_if_config_corrupt` gate in `backend/deps.py` is never triggered. The MCPB user gets the same graceful-degradation behavior as the web UI in this case.
- **DYLD bootstrap is NOT in the closure.** `backend/main.py:_bootstrap_macos_dyld_for_weasyprint` only runs in the FastAPI app's lifespan. Since `backend/main.py` is excluded from the bundle, this isn't a concern.
- **UV first-launch needs network access.** Claude Desktop's extension sandbox MUST allow UV to reach PyPI to install deps on first run. Pin this in the test plan (§7 step 5) — if Claude Desktop blocks egress, fall back to `server.type: "python"` with a bundled venv built on the host platform, accepting the pydantic-compiled-wheel-per-platform pain.
- **macOS / Linux / Windows parity.** UV handles pydantic's manylinux/macOS/Windows wheels transparently as long as PyPI access is allowed. No platform-specific build needed.
