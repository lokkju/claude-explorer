# Plan: replace polling watcher with `watchdog` (FSEvents/inotify/RDCW)

**Date:** 2026-05-15
**Status:** planned, not yet implemented
**Owner:** rpeck
**Related:** `backend/cc_image_watcher.py`, `fetcher/cli.py:install-watcher`,
PLANS/2026.05.15-recover-cc-images-from-time-machine.md

## Context

Today's `backend/cc_image_watcher.py` polls `~/.claude/image-cache/`
every 60 seconds (recently re-tuned from 5s → 60s). Polling has two
costs:

1. **Wakeups per day** — 1,440 even when nothing happens. On a laptop
   this measurably interferes with macOS App Nap and idle sleep.
2. **Race window** — the worst-case capture latency is the polling
   interval. Even at 60s, an adversarial CC rotation that creates +
   deletes a file inside one window leaves us with no capture.
   Empirically CC's rotation runs hours/days, not seconds, so this is
   theoretical — but the *cost of missing an image* is permanent
   data loss.

OS-native filesystem-event APIs eliminate both:

| OS      | API                       | Latency    | Idle CPU |
|---------|---------------------------|------------|----------|
| macOS   | FSEvents                  | < 1s       | 0        |
| Linux   | inotify                   | < 100 ms   | 0        |
| Windows | ReadDirectoryChangesW     | < 100 ms   | 0        |

The standard cross-platform Python wrapper is
[`watchdog`](https://github.com/gorakhargosh/watchdog) — actively
maintained for ~13 years, ~3M downloads/month on PyPI, used by Black,
Django auto-reload, Flask reloader, and many others.

## Goal

Replace the polling loop in `backend/cc_image_watcher.py` with an
event-driven watcher built on `watchdog`, while preserving every
existing guarantee:

- Same destination layout (`~/.claude-explorer/cc-images/<sess>/<sess>--<N>.<sha8>.<ext>`).
- Same content-addressed dedup (no duplicate writes for the same bytes).
- Same per-process `_seen` memo for already-handled paths.
- Same `scan_once()` return type (count of newly handled paths) so
  the FastAPI lifespan task and the search-index drift co-loc keep
  working without changes.
- Same "search-index drift pass piggybacks on each watcher iteration"
  behavior — restructured (see "Search-index drift" below).
- Same env-var override (`CLAUDE_EXPLORER_CC_WATCHER_INTERVAL_SEC`)
  acts as the **backstop poll** interval (see "Backstop poll" below).

## Architecture: event-driven primary + slow poll backstop

Production-grade tools that depend on filesystem events use both:
events for latency, polling for safety against the rare OS bug
(FSEvents has a documented coalescing edge case under extreme load;
inotify has a per-watch event-queue overflow possibility).

```
                    ┌───────────────────────────┐
   FSEvents/inotify │   watchdog Observer       │
   ──────────────▶  │   (background thread,     │
   per-file event   │    ~zero idle CPU)        │
                    └────────────┬──────────────┘
                                 │ on_created / on_modified
                                 ▼
                     handle_one_path(path)
                       ├── copy_marker_image_to_cache()
                       └── search-index drift mark
                                 ▲
                                 │
   ┌─────────────────────────────┴─────────────┐
   │ Backstop poll loop                        │
   │   every BACKSTOP_INTERVAL_SEC (default    │
   │   600s = 10 min) full rglob() walk        │
   │   to catch any events the OS missed       │
   └───────────────────────────────────────────┘
```

`handle_one_path()` is the existing `scan_once()` body, refactored
to take a single path. Idempotent because of the `_seen` memo.

## Public API surface (preserved)

```python
def scan_once() -> int:
    """Run a single full poll pass. Used by:
       1. FastAPI lifespan startup (initial warm)
       2. The backstop poll loop (every 10 min)
       3. The CLI `claude-explorer warm-cc-cache` command

    so the `install-watcher`-installed launcher script remains
    backward-compatible during the rollout window."""

def start_watcher(stop_event: asyncio.Event | None = None) -> None:
    """NEW. Start the watchdog Observer + backstop poll loop.
    Used by FastAPI lifespan and by the new launcher script
    `cc-watcher.py` (which `install-watcher` writes)."""
```

## Backstop poll

The backstop catches events that the OS dropped or coalesced. Tuning:

- **Default interval: 600s** (10 min). Backstop is for correctness,
  not latency. Events handle latency.
- **Env-var override:** `CLAUDE_EXPLORER_CC_WATCHER_INTERVAL_SEC`
  (existing) is repurposed to mean "backstop interval". Existing
  user overrides keep working but apply to a different timer; the
  meaning shift is documented in the env-var help text and CHANGELOG.

## Search-index drift

Today's `scan_once()` runs the search-index drift pass at the end of
each scan (every 60s). With events, drift detection should fire on:

1. Any `~/.claude/projects/*.jsonl` file event — instant search
   index update for new/edited sessions.
2. Backstop poll — catches anything events missed.

This is a strict improvement over today: typing in a CC session
becomes searchable within ~1s instead of waiting up to 60s.

## Implementation steps

1. **Add `watchdog` to `pyproject.toml`** (`watchdog = "^4.0"`).
   Verify cross-platform wheel availability (it ships wheels for
   every supported Python on PyPI).

2. **Refactor `backend/cc_image_watcher.py`:**
   - Extract `handle_one_path(path)` from current `scan_once()` body.
   - Keep `scan_once()` as a `for path in root.rglob("*"): handle_one_path(path)` shortcut.
   - Add `start_watcher(stop_event)` that:
     - Builds a `watchdog.observers.Observer`.
     - Schedules a `FileSystemEventHandler` on
       `~/.claude/image-cache/` (recursive=True) and on
       `~/.claude/projects/` (recursive=True, JSONL-only filter).
     - Starts a daemon thread for the backstop poll loop.
     - Blocks on `stop_event` until shutdown.

3. **Update FastAPI lifespan in `backend/main.py`:**
   - Replace the existing `asyncio.to_thread(scan_once)` periodic
     pattern with a one-time `start_watcher` call inside
     `asyncio.to_thread`.
   - Initial `scan_once()` still fires at startup (warm pass).

4. **Update launcher generator in `fetcher/cli.py`:**
   - `_build_watcher_inline_script()` body changes from `while True:
     scan_once(); sleep(interval)` to a single
     `from backend.cc_image_watcher import start_watcher; start_watcher()`.
   - The launchd plist / systemd unit / Task Scheduler task ALL
     just run `<python> ~/.claude-explorer/cc-watcher.py` (no
     change to the supervisor units themselves; only the launcher
     body changes).
   - `--interval` CLI flag keeps its name but its help text shifts
     to "backstop poll interval (default: 600 = 10min); event-
     driven watcher captures most cases instantly."

5. **Tests:**
   - `backend/tests/test_cc_image_watcher_events.py` — new file.
     Use `tempfile.TemporaryDirectory` + `monkeypatch` of
     `_live_image_cache_root()` to redirect to a tmp dir; create a
     file inside; assert the event fires within 2s and the file is
     mirrored. Use `watchdog`'s in-memory `PollingObserver` for
     deterministic CI behavior (FSEvents is hard to test
     deterministically; the production code uses `Observer()` which
     auto-selects the OS-native backend).
   - Backstop poll test: `monkeypatch` the backstop interval to

   - Existing `scan_once()` tests should keep passing unchanged

6. **Migration / rollout:**

## Risks / known issues

1. **FSEvents coalescing on extreme event rates.** Mitigation:
   backstop poll. Realistic worst case in our domain (single user,
   handful of files/day) is fine.

2. **inotify per-user watch limit** (`/proc/sys/fs/inotify/max_user_watches`,
   default 8192 on most distros, often raised to 524288 by Docker /
   IDE installs). We watch two directories (recursive); should be
   well under the limit. If we hit it, `watchdog`'s
   `OSError: inotify watch limit reached` propagates clearly.
   Document the `sysctl fs.inotify.max_user_watches=524288` fix
   in the troubleshooting section.

3. **macOS sandboxed Python.** App-Store / sandboxed Python builds
   may have FSEvents access denied. Watchdog falls back to its
   PollingObserver in that case. Detect at startup; print a warning
   if we landed on the polling fallback.

4. **NFS / network-mounted home directories.** inotify doesn't
   fire on remote mounts. Pure polling fallback applies. Same
   detection + warning pattern.

5. **Watchdog dep adds ~50KB + a C extension on Linux/Windows.**
   Already a transitive dep of many tools (Django, Flask, etc.) so
   most users have it cached. Not a concern.

6. **Observer thread crash isolation.** A bad event handler can
   take down the whole Observer thread silently. Wrap each event
   handler call in a `try/except logger.exception`, same pattern as
   the existing `_seen` retry-on-error logic.

## Out of scope (future work)

- **Per-file event types beyond CREATE.** Today we only care about
  new files. If we ever need to track edits or moves, the watchdog
  event model already exposes `on_modified`, `on_moved`, etc. —
  trivial to add later.
- **Cross-machine (network) image sync.** Mirror sync between
  multiple machines is a separate problem; out of scope for this
  refactor.
- **Mirror compression / dedup beyond content addressing.** Mirror
  size is small (<<1GB for most users); not yet a problem.

## Success criteria

- All existing `backend/tests/test_cc_image_watcher*.py` tests pass
  unchanged.
- New event-driven test passes deterministically using
  `PollingObserver`.
- `claude-explorer serve` starts in same time as before (no
  blocking on Observer setup).
- Manual verification: paste an image into a CC session, observe
  the mirror copy appears in `~/.claude-explorer/cc-images/<sess>/`
  within 2 seconds (vs up to 60 seconds today).
- After backstop interval (10 min), no duplicate work in logs
  (idempotent via `_seen`).
- Energy Impact column in `top -o energy` (macOS) for the watcher
  process drops to "0.0" or near-zero idle, vs current ~0.5–1.5
  during polling.
