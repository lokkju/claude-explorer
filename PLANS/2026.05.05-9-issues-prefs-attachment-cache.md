# Plan: 9 Issues + Permanent Attachment Cache + Server-Side Preferences

## Context

The user manually tested the previous round of work (search-scope pin + focus model + image fallback) and surfaced 9 issues. They invoked `/coding`, an LLM Council triaged them, and the user answered the 10 follow-up questions — most importantly expanding scope to two new infrastructure pieces:

1. **Server-side preferences** (instead of browser localStorage) so settings work across browsers and devices.
2. **Permanent attachment cache** (CC image-cache files + Desktop `Message.files[]`) so conversations stay readable forever even after Claude Code rotates its image cache or Desktop deletes attachments.

This plan supersedes any prior content of `smooth-Meandering-cat.md` (which described a different article-coverage task). Strict TDD applies throughout: every code change requires a failing Playwright (frontend) or pytest (backend) test FIRST. No push to main without explicit user approval.

## User answers (already given)

- **Q1+Q2 (persistence):** server-side, `~/.claude-explorer/preferences.json`. Cross-browser. localStorage is ephemeral and per-browser, which the user does not want.
- **Q3 (Cmd+G→Enter focus):** Enter moves DOM focus to the bubble; Cmd+G keeps focus in the input. Confirmed.
- **Q4 (chip bug):** real bug. User had a pin set, no chip rendered. Reproduce + fix.
- **Q5 (title search):** match `name` OR `project_path` (drop `summary`). Relabel input to "Search titles and projects".
- **Q6 (Unpin):** always visible, disabled when nothing is pinned.
- **Q7 (tool placeholder):** hide entirely.
- **Q8 (PDF):** PDF stays its own button; PDF MUST render images (currently broken — confirmed by exploration).
- **Q9 (non-image attachments):** always bundle; fetch+cache permanently.
- **Q10 (14.png reload):** auto-reload missing images via permanent cache. Going forward, fetch downloads + permanently caches.
- **Plus:** "All UX flows / rules should be documented in UX.md and CLAUDE.md should refer to it."

## Key code findings (from exploration)

- **localStorage today (13 keys across 4 contexts):** `SettingsContext` (9 keys), `SourceFilterContext` (1), `SearchPanelContext` (4), `FilterContext` (2). Bookmarks already server-backed via `~/.claude-explorer/bookmarks.json` — that's the **pattern to mirror** for preferences.
- **PDF rendering is broken for images:** `backend/export.py:347–354` has an explicit comment that `WeasyPrint.HTML(string=...)` cannot fetch `/api/...` URLs — there's no session, no localhost awareness, no sessionKey. Result: blank image placeholders. No test asserts PDF contains image data.
- **Desktop attachments already partially cached:** `fetcher/bulk_fetch.py:353–514` — `ClaudeFetcher._download_file()` and `download_conversation_files()` write to `~/.claude-explorer/files/<conv-uuid>/<file-uuid>/{thumbnail|preview|original|document}` and stamp the conversation JSON with `local_thumbnail` etc. **This already exists** but the frontend image proxy doesn't use the local copy yet.
- **CC image-cache pass-through, no caching:** `backend/routers/files.py:138–200` `/api/cc-image?path=<abs>` reads directly from `~/.claude/image-cache/`. Nothing copies these to our own dir, so Claude Code's rotation breaks them.

### Phase 3 — Server-side preferences infrastructure

**Council edit:** dual-read/dual-write + migration marker + PATCH deep-merge. The naive "read localStorage, PUT, delete" path is unsafe in a two-tab race.

Mirror the bookmarks pattern (`~/.claude-explorer/bookmarks.json` + `backend/routers/bookmarks.py`).

**Backend (test-first):**
- `backend/tests/test_preferences.py` RED tests: GET returns defaults when file missing; PUT writes atomically; **PATCH deep-merge** (a stale tab can't wipe a fresh tab's keys); round-trip; permission `0o600`; unknown key tolerated; concurrent PATCHes don't corrupt; **backend-down fallback** (frontend reads localStorage if `/api/preferences` returns 5xx).
- New `backend/routers/preferences.py`: `GET /api/preferences`, **`PATCH /api/preferences`** as the primary write path (deep-merge per key). `PUT` available but discouraged. Atomic tmp→rename. Versioned envelope `{"version": 1, "data": {...}}`.

**Frontend (test-first):**
- New hook `usePreferences()` that reads on mount via TanStack Query, writes via PATCH mutation. Optimistic update.
- **Dual-read:** if server returns null/5xx for a key, fall back to localStorage. **Dual-write:** for the migration release, write BOTH stores. Land a separate later commit that removes the localStorage fallback once telemetry/manual confirms server is authoritative. **Do NOT delete localStorage in the same commit that introduces server reads.**
- **Migration marker:** before the first PATCH, write `prefs_migrated_v1=true` to localStorage. Other tabs check this flag first and skip migration. **Two-tab race test:** open two tabs simultaneously with seeded legacy localStorage; assert both converge to the same server state without corruption.

- Frontend Playwright RED tests: change a setting in browser A, reload, see it persist; clear localStorage, reload, see server-side value still applied; backend-500 → app stays usable on localStorage fallback.

## Council review (Gemini-3-Pro + GPT-5.2-Pro consensus)

The plan was reviewed by an LLM Council. Both councilors converged on 8/8 questions; minor divergence on cache layout was resolved via hybrid (split trees, no Desktop duplication). Three real safety holes were identified and folded into the plan above:

1. **PDF base64 inlining was wrong primitive** — replaced with WeasyPrint `url_fetcher` hook. Memory-safe; no MB-scale HTML strings.
2. **Migration race was unsafe** — Phase 3 now uses dual-read/dual-write + `prefs_migrated_v1` marker + PATCH deep-merge; localStorage NOT deleted in the same commit as server-read introduction.
3. **Permanent cache layout had collision risk** — split into separate trees: new `~/.claude-explorer/cc-images/<conv-uuid>/<sess>--<N>.<sha8>.png` for CC marker images; reuse existing `~/.claude-explorer/files/...` for Desktop attachments.

Other folded edits: tighter PDF test (known-fixture byte signature, not "≥1 stream"); two-test isolation for Issue 4 (`summary`-removal vs ranking); narrow-viewport test for Issue 2; backend export filtering of TOOL_PLACEHOLDER; Firefox in Phase 9 test matrix.

Phase order ROI: smallest-first wins, both councilors agreed. Phases 1–2 are isolated and safe; Phases 3–4 land before Phase 7 (Markdown dialog writes preferences) — order already correct.

## Risks

- **Phase 3 (preferences migration) is invasive** — touches every settings-reading context. Risk of regressing unrelated UI. Mitigation: one-context-per-commit; keep tests granular.
- **Phase 4 (cache fill) needs a real fetch run** — the test fixtures don't actually download from claude.ai, so end-to-end caching of Desktop attachments must be manually verified.
- **Phase 5 (PDF base64)** — embedding images as base64 inflates the HTML passed to WeasyPrint; for conversations with many images this can be slow or memory-heavy. Acceptable for v1; profile if it becomes a problem.
- **Phase 4 + Phase 5 sequencing** — PDF embedding depends on the permanent cache existing. Phase 4 must land before Phase 5 unless we want a shim that downloads on-demand at PDF time.
