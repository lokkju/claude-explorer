# F12 — Cowork source indicator is mislabeled as "Claude Desktop"

**Date:** 2026-05-29

**Owner:** the *other* CC session (code). This session owns the article follow-up (see §7) once the code lands.
**Scope:** frontend only. The source-indicator UI. No backend/data-model change required (see §6 for a related-but-separate data field that is explicitly OUT of scope).

## 1. Problem

Cowork is a first-class source: `backend/cowork_reader.py:154` tags sessions `source: "CLAUDE_COWORK"`, the type union includes it (`frontend/src/lib/types.ts:3`), and the sidebar **source dropdown** lists it as a first-class entry with its own glyph and color (`Sidebar.tsx:230-234`).

But the per-row and per-conversation **source indicators** were written as a binary `CLAUDE_CODE ? … : …` ternary with no Cowork branch. Everything that is not `CLAUDE_CODE` falls through to the **"Claude Desktop"** arm. So a Cowork session renders the blue "Claude Desktop" message glyph in the sidebar and a blue **"Desktop"** badge in the conversation header — actively wrong, and inconsistent with the dropdown the user just picked "Claude Cowork" from.

This is a real user-facing inaccuracy, not cosmetic: the source indicator is the field the article (and the user) leans on to answer *"where did this conversation come from?"* at a glance.

## 2. Canonical Cowork visual identity (already chosen — do not invent a new one)

The source dropdown already established Cowork's look. Match it exactly everywhere:

| Source | Glyph (`lucide-react`) | Color | Row tooltip | Header badge text |
|---|---|---|---|---|
| Claude Desktop (`CLAUDE_AI`) | `MessageSquare` | `text-blue-500` / `bg-blue-100 text-blue-700 dark:bg-blue-900 dark:text-blue-300` | "Claude Desktop" | "Desktop" |
| Claude Code (`CLAUDE_CODE`) | `Terminal` | `text-green-500` / `bg-green-100 text-green-700 dark:bg-green-900 dark:text-green-300` | "Claude Code" | "Code" |
| **Claude Cowork (`CLAUDE_COWORK`)** | **`Sparkles`** | **`text-purple-500` / `bg-purple-100 text-purple-700 dark:bg-purple-900 dark:text-purple-300`** | **"Claude Cowork"** | **"Cowork"** |

Source of truth for the purple `Sparkles` choice: `frontend/src/components/layout/Sidebar.tsx:230-234`.

### 3c. OPTIONAL — grouped-view header icon

## 4. Recommended design: extract one shared mapping

Three sites (3a, 3b, and arguably 3c) encode the same source→{glyph,color,label} mapping. That is exactly how the Cowork branch got missed in two of them. Strongly consider a single tiny helper so a future 5th source can never re-introduce this drift:

- e.g. `frontend/src/lib/sourceDisplay.tsx` exporting `SOURCE_DISPLAY: Record<ConversationSource, {Icon, dotClass, badgeClass, label, short}>` plus a `<SourceIcon source/>` and `<SourceBadge source/>`.
- Route 3a and 3b (and the dropdown at `Sidebar.tsx:218-235`, if you want full convergence) through it.

**Minimal-blast-radius alternative:** just add the inline Cowork arm in 3a and 3b exactly as shown above. Either satisfies F12; the helper is the cleaner, drift-proof option and is the recommended one if the other session's council agrees the refactor is in scope.

## 6. OUT OF SCOPE (flagged so it isn't a surprise later)

`MessageBubble.tsx:88` and `ConversationPage.tsx:741` map source to a **bookmark provenance string**: `conversationSource === 'CLAUDE_AI' ? 'claude_desktop' : 'claude_code'`. A Cowork bookmark is therefore stored with `source: 'claude_code'`. That is a **data-model** question (does the bookmarks schema want a `claude_cowork` provenance value?), not a source-indicator question, and changing it touches the bookmarks API + stored data. **Do not fold it into F12.** If you decide a Cowork bookmark should be tagged distinctly, that is its own small plan with its own migration thought. Noted here only so the F12 work doesn't accidentally "fix" it halfway.
