## Behaviors to cover (from UX.md § "Composable filters")

For every spec clause, at least one test. Each test annotates which clause it verifies.

### Atom semantics
- Hide Behavior + glob patterns: matching titles drop, others stay (multiple patterns OR'd).
- Show-only Behavior + glob patterns: matching titles kept, others drop.
- Regex mode for both behaviors.
- Empty-patterns atom: passes for every conversation (least-surprise).
- Disabled atom: never appears in active picker; selected-then-disabled becomes no-op.

### Group semantics (pure combinator, no Behavior)
- "all of these": every enabled member must pass.
- "any of these": at least one enabled member must pass.
- Group of two hide-atoms with `match: any` ≈ "hide if either pattern hits" — the canonical alternative to the cron-atom shortcut.
- Group of mixed hide+show-only atoms: each child contributes its own keep/drop; group ANDs/ORs over those.
- Empty group OR group whose members are all disabled / orphans: passes everything.
- **Recursive groups (Gemini council edit):** Group A `match: all` containing Group B `match: any`. Verify the evaluator traverses arbitrary depths cleanly.
- **Disabled-member-dropped-not-short-circuited (Gemini council edit):** the prior bug short-circuited `match: any` to true on a disabled member's `true` return; the correct behavior drops disabled members before the quantifier. To make these distinguishable from the OUTSIDE, use a `match: any` group with TWO members: Member 1 disabled, Member 2 a show-only atom matching only "Foo". Buggy = passes everything; correct = passes only "Foo". This is the only test that distinguishes the two semantics observably.
- **Stale activeId (Gemini council edit):** seed `/api/preferences` with `filters.activeId` pointing at a non-existent FilterId. App must load, treat as no-op (no filter active), every conversation visible.

### Active filter
- The sidebar `<Select>` shows "All conversations" plus every enabled filter alphabetical.
- Selecting "All conversations" deactivates filtering AND persists `activeId: null` (or equivalent) — assert via PATCH body inspection (GPT council edit).
- Selection persists across `page.reload()`. Two assertions per test (GPT council edit): intercept the PATCH on selection AND assert the picker reflects the persisted value on re-mount.
- A "Manage filters…" item lives at the bottom of the dropdown and opens the modal *without* changing the active filter. Assert: the picker's displayed value text is unchanged after closing the modal (GPT council edit; guards a subtle bug where `activeId` mutates on modal close).
- Picker offers "Manage filters…" even when ZERO filters exist (GPT council edit, spec-implied).

### Manage Filters modal — atom editor
- Behavior radio at top: "Hide matches" and "Show only matches".
- Mode radio: "Glob" and "Regex".
- Patterns textarea (one per line).
- Plain-English summary line below the controls. Hide + ≥1 pattern: contains "Hides" and "match any of"; show-only: contains "Shows only" and "match any of"; empty: a "no patterns" message.
- **Behavior radio updates the summary line in real time, no save needed (GPT council edit).**
- **Mode radio switch re-evaluates the active filter against current conversations within the same session — not just on reload (GPT council edit).** I.e., selecting Regex with a regex pattern that matches a title that the previous Glob pattern wouldn't have, the sidebar updates immediately.
- Name auto-fill from first usable pattern (≥3 alphanum after metachar strip), debounced.
- Auto-fill stops once the user manually types into Name.
- Clearing Name resumes auto-fill.
- Enabled toggle on row + in editor (synced **in BOTH directions** — toggle in row updates editor, toggle in editor updates row; GPT council edit).
- **Trash icon present on every row of the left list, reachable, and not clipped off the right edge.** This is the test that catches the user's just-reported regression. Three assertions (Gemini council edit): `toHaveCount(N)`, `toBeVisible()`, AND `toBeInViewport()` (the latter catches CSS clipping where the button is "found" but pushed off-screen). Plus a bounding-box check (GPT council edit): `await deleteBtn.boundingBox()` and `await row.boundingBox()`; assert `dx + dw <= rx + rw + epsilon` (epsilon ~1px for sub-pixel rounding).
- "Used by:" line under the Name input lists referencing groups when present, hidden when not. The line text includes ALL referencing group names (GPT council edit).

### Manage Filters modal — group editor
- Match radio: "all of these" and "any of these" (no AND/OR jargon).
- Member chips, each with an `×` to remove.
- "Add member" Select excludes (a) self and (b) candidates that would create a cycle.
- Disabled members render with "(disabled)" suffix.
- Plain-English summary line below members ("Hides..." / "Shows only..." per behavior of children).
- **Trash icon on every row** (same as atom).

### Delete UX
- Unreferenced filter: trash click reveals an inline Confirm/Cancel; Confirm calls remove.
- Referenced filter: trash click reveals an inline "Used by: <group names>. Remove it from those groups first." block; deletion is blocked.
- Deleting the currently-active filter clears the active selection (no stale `activeId`).

### Cycle defense
- "Add member" Select must not list a candidate whose addition would close a cycle.
- A prefs blob that already contains a cycle (manually edited) loads without crashing; the runtime evaluator returns "passes" for the visited node.

### Migration
- v1 → v2: atoms with `polarity: 'include'` → `behavior: 'show-only'`; `polarity: 'exclude'` → `behavior: 'hide'`. Groups unchanged. `_migratedV2: true`. Idempotent across mounts.
- Pinned-style legacy state (savedFilters + activeFilterIds) → "Default (migrated)" group containing previously-pinned atoms; `_migratedV1: true`; legacy keys nulled in PATCH.
- **Migration PATCH body assertion (Gemini council edit):** intercept the PATCH via `page.route('**/api/preferences', ...)`, parse the body, assert it explicitly contains tombstone keys `savedFilters: null` AND `activeFilterIds: null` (omitting them is not enough — the backend's per-key overwrite leaves them on disk).
- **Don't assert `match` on the "Default (migrated)" group (GPT council edit):** UX.md doesn't specify `all` vs `any`. Assert the OBSERVABLE outcome (previously-pinned atoms grouped under that name, the group is the active filter, and behavior matches the expected filtering result on the seeded conversations) rather than the field value.
- Banner appears once when `_migratedV1 && !migrationBannerDismissed`; dismiss persists across reload via PATCH.
  - PATCH body must explicitly set `filters.migrationBannerDismissed: true` (key path may differ in impl; assert via key-substring match if the spec is ambiguous).
  - Dismiss button: don't hard-require a specific aria-label; assert `getByRole('button', { name: /dismiss|close|×/i })` (GPT council edit).
- Fresh install (no legacy state, no `_migratedV1`) never sees the banner.

### Selector philosophy

Use semantic queries throughout. The spec doesn't promise data-testids, so the tests don't rely on them. Examples:

- Active picker: prefer `getByLabel(/filter/i)`. shadcn's `<Select>` trigger may render as `role=button` rather than `role=combobox` (GPT council flag), so the `getByRole('combobox')` query is unreliable across implementations. Allowed fallback (the only contract-implicit testid): `data-testid="active-filter-select"`.
- Behavior radio: `page.getByRole('radio', { name: /hide matches/i })`. If the impl uses a custom segmented control (button with `aria-pressed`), that is itself a spec violation worth surfacing — log and report.
- Match radio: `page.getByRole('radio', { name: /all of these/i })`.
- Patterns textarea: `page.getByRole('textbox', { name: /patterns/i })`.
- Name input: `page.getByRole('textbox', { name: /^name/i })`.
- "Add member" Select: `page.getByRole('combobox', { name: /add member/i })` with the same shadcn-Portal-wait pattern.
- Trash button: `page.getByRole('button', { name: /^delete/i })` (matches `aria-label="Delete <filter name>"`).
- Enabled toggle: `page.getByRole('checkbox', { name: /enabled/i })`.

When a spec clause prescribes specific text ("Hides conversations whose titles match any of:"), assert with `toContainText(/<key phrase>/i)` rather than exact-equals — keeps the test robust to copy edits while still catching real drift.

**Portal animation discipline (Gemini council edit):** shadcn `<Select>` and `<DropdownMenu>` mount option lists in a Portal with mount animations. Always `await trigger.click()` THEN `await expect(page.getByRole('option', { name: ... })).toBeVisible()` THEN `.click()`. Raw `.click()` on options can race the animation and miss the target.

### Mocking

Always seed `/api/preferences` via `mockBackend({ preferences: {...} })`. For each test, build the smallest legitimate `filters` blob the spec implies, plus 3-5 conversations whose titles span the patterns under test. Don't introspect or duplicate existing test setups; build clean fixtures from the spec.

### Phase 3 — Run, triage, label

```
cd frontend && npx playwright test e2e/spec-filters-*.spec.ts --reporter=line
```

Run ALL the new specs in one pass. Categorize each failure:

- **Real spec violation**: the implementation deviates from UX.md. Fix the implementation. (Trash icons disappearing is here.)
- **Spec ambiguity**: the spec is clear but the test interpreted a phrase too tightly (e.g. exact copy of the summary line). Update the test to a looser, still-meaningful assertion.
- **Test infra bug**: typo, wrong `mockBackend` setup, race condition the spec doesn't dictate. Fix the test.

Surface each failure in the format:
```
spec-filters-<file>.spec.ts:<line>  CLAUSE: "<UX.md sentence>"  CATEGORY: violation|ambiguity|infra|gap
```

If a category is *violation*, the linked clause is the contract that needs honoring; the fix lands in the relevant `frontend/src/...` file.

**Fourth category — `gap` (Gemini council edit):** the test failed because the impl does X, the test expected Y, and `UX.md` is entirely silent on which is correct. Don't fix the code; don't fix the test alone; surface the gap to the user with a recommendation. (Example candidate: what happens if the active filter is deleted from another tab and a server-state sync pulls the new prefs.) These produce DECISIONS, not commits.

## Risks

- **Spec drift from copy edits.** UX.md may have wording I'm asserting too tightly on. Mitigation: prefer regex matchers over exact-equals, and prefer key-phrase matches ("Hides", "match any of") over full sentences.
- **Selector ambiguity for the picker.** The picker is one of three `<Select>` controls in the sidebar, all with `role=combobox`. Pin via the `data-testid="active-filter-select"` (the one test-id the spec implicitly mandates by naming the picker structure).
- **Migration tests need pre-seeded prefs.** `mockBackend({ preferences: { savedFilters: [...], activeFilterIds: [...] } })` lets the app run its v1 migration on the seeded blob and the test verifies the post-migration shape via subsequent PATCH bodies and visible state. If the seed shape is wrong, the migration won't run; that's a test bug not a spec violation.
- **The "delete" Phase 3→4 loop must terminate.** If a violation fix uncovers a second violation, fix one at a time; commit logically grouped fixes; don't bundle.
- **Don't accidentally read app code in Phase 2.** The constraint is human discipline; mitigation is using a clearly-labeled allowlist of files (UX.md, plan files, fixtures.ts) and running `grep` only against them.
