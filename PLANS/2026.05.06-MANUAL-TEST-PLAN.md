## §1 — Server-side preferences (P3) — cross-browser test

**The flagship infrastructure change today.** Settings, sort
preferences, theme, keyboard mode, source filter, etc. now persist
server-side in `~/.claude-explorer/preferences.json` instead of
browser localStorage. Open the SAME app in two different browsers and
your settings should follow.

### §1.1 — Single-browser persistence

1. Open http://localhost:5173 in your usual browser.
2. Open Settings (gear icon or Cmd+,).
3. Change theme to **Dark**, keyboard mode to **Vim**, click "Bundle
   images as a zip", switch dialect to **Obsidian**.
4. Hard-reload the page (Cmd+Shift+R).
5. **Expected:** all four settings persist. Theme is still Dark. Vim
   mode still on.

### §1.2 — Cross-browser persistence (the real test)

1. With the settings from §1.1 still set in browser A, open
   http://localhost:5173 in a DIFFERENT browser (Chrome ↔ Safari, or
   a private/incognito window).
2. **Expected:** the second browser opens with theme=Dark, vim mode,
   the bundle settings — even though it has a different localStorage.

### §1.3 — Inspect the on-disk file

```bash
cat ~/.claude-explorer/preferences.json | python3 -m json.tool
```

Should show `{"version": 1, "data": {"theme": "dark", "keyboardMode":
"vim", "markdownBundleImages": true, "markdownDialect": "obsidian",
...}}`.

### §1.4 — Backend-down fallback

1. With both browsers open and settings synced, **stop the backend**:
   `lsof -ti :8000 | xargs -r kill -INT`.
2. Reload either browser.
3. **Expected:** app still loads. Theme/vim mode still applied (read
   from localStorage mirror). You'll see a connection-error toast for
   conversation data — that's expected.
4. Restart the backend (see Setup); the connection error clears.

**Pass:** preferences travel between browsers, survive backend outage,
and live in `~/.claude-explorer/preferences.json` on disk.

## §2 — Permanent attachment cache (P4)

Two separate caches:

- **`~/.claude-explorer/cc-images/<conv-uuid>/<sess>--<N>.<sha8>.png`** — Claude Code image-cache markers (`[Image: source: ...]`) get COPIED here at fetch time, so they survive when CC rotates the originals.
- **`~/.claude-explorer/files/<conv-uuid>/<file-uuid>/{thumbnail|preview|original|document}`** — Desktop attachments (every kind, not just images).

### §2.1 — CC image survives rotation

1. Open a Claude Code conversation that has an image marker.
2. Confirm the image renders.
3. Find the source path in the marker (open inspector, look at the
   `<img>` `src` — it'll be `/api/cc-image?path=<abs>`).
4. **Delete the original** from `~/.claude/image-cache/<sess>/<N>.png`
   (simulating CC rotation).
5. Hard-reload the conversation.
6. **Expected:** the image still renders (resolved from
   `~/.claude-explorer/cc-images/`).

### §2.2 — Desktop attachments cached

1. Open a Claude Desktop conversation that has an image OR a non-image
   attachment (PDF, txt).
2. Verify the file shows up.
3. Inspect the file path:
   `ls ~/.claude-explorer/files/<conv-uuid>/<file-uuid>/`. Should
   contain `thumbnail`, `preview`, `original`, or `document` depending
   on kind.

### §2.3 — Auto-retry on transient failure

1. Open a conversation with images.
2. Throttle network to "Slow 3G" in DevTools (so first fetch
   occasionally 404s).
3. Reload the conversation a few times.
4. **Expected:** images that fail on first load auto-retry with a
   cache-busting `?retry=1` URL before falling back to the broken-image
   tile. You should NOT see the dashed "Image not in cache" tile for
   images that exist on disk.

### §2.4 — Friendly fallback copy

1. Find a conversation with a CC image marker pointing at a path that
   really doesn't exist on disk anywhere (live OR cache).
2. **Expected:** dashed-border tile with `ImageOff` icon and the text
   "Image not in cache: <filename>". Hovering shows a tooltip
   explaining "Original was rotated by Claude Code; this image was not
   present at fetch time, so we couldn't cache it."

## §8 — Tool placeholder hidden (P1.3)

Anywhere a message text contains the literal string
`This block is not supported on your current device yet, please update
to use the latest version` (Claude Code's TOOL_PLACEHOLDER), it should
be silently stripped from the rendered viewer, the Markdown export, AND
the PDF export.

1. Find a conversation that contains a tool block — most CC sessions
   with tools used.
2. Verify in the viewer: NO instance of "not supported on your current
   device" appears.
3. Export the conversation as Markdown (Inline) and grep:
   `unzip -p ~/Downloads/foo.md | grep -c "not supported"` should be 0.
4. Export as PDF; open it; visually scan for the message.

### §8.1 — Fenced-block badge survives

The frontend keeps the badge "Tool call or artifact not captured in
export" inside fenced code blocks. So the placeholder is stripped
OUTSIDE fences but the friendly badge inside fences stays.

1. Find a CC session with a tool_use block.
2. Verify: in the viewer, you see the friendly badge inside a code
   block. NO "not supported" text leaks elsewhere.
