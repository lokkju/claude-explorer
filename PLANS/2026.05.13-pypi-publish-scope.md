# PyPI publish scope — `claude-explorer`

## Goal
Turn the install path from "git clone + uv sync + 5 commands" into
`uvx claude-explorer serve`. One line, no clone, no dev environment.

## TL;DR
**~half day (4-6 hours) of focused work.** Name is available, license
file is missing, and we need a build hook to bundle the React app
into the wheel. After the first release, ongoing publish is a tag push.

## Work breakdown

### Phase 1 — Build configuration (2-3 hours)

**1.1 Add LICENSE** (15 min)

Pick a license. MIT is most permissive and common for unofficial CLI
tools. Add `LICENSE` at repo root. Reference in `pyproject.toml`:

```toml
[project]
license = "MIT"
license-files = ["LICENSE"]
```

**1.2 Add a frontend-build hook** (1-2 hours)

Two approaches; recommend (A):

**(A) Custom Hatch build hook.** Migrate from setuptools to hatchling.
Add a `hatch_build.py` that runs `npm install && npm run build` in
`frontend/` before wheel assembly, then `include = ["frontend/dist/**"]`
gets the built bundle into the wheel. Builder needs `node`/`npm`;
end-user does not.

```toml
[build-system]
requires = ["hatchling", "hatch-fancy-pypi-readme"]
build-backend = "hatchling.build"

[tool.hatch.build]
include = [
  "backend/**/*.py",
  "fetcher/**/*.py",
  "mcp_server/**/*.py",
  "frontend/dist/**",
]
exclude = ["**/tests/**", "**/__pycache__/**"]

[tool.hatch.build.hooks.custom]
path = "hatch_build.py"
```

`hatch_build.py`:
```python
from hatchling.builders.hooks.plugin.interface import BuildHookInterface
import subprocess, sys, pathlib

class FrontendBuildHook(BuildHookInterface):
    def initialize(self, version, build_data):
        frontend = pathlib.Path(self.root) / "frontend"
        if (frontend / "dist" / "index.html").exists():
            return  # already built
        subprocess.run(["npm", "ci"], cwd=frontend, check=True)
        subprocess.run(["npm", "run", "build"], cwd=frontend, check=True)
```

**(B) Stay with setuptools.** Use `setup.py` (legacy) with a custom
`build_py` that runs npm. More boilerplate, harder to maintain. Skip.

**1.3 Verify the wheel contains the bundle** (30 min)

```bash
uv build
unzip -l dist/claude_explorer-0.1.0-*.whl | grep -E "index.html|assets/"
```

Should list `frontend/dist/index.html` and `frontend/dist/assets/*.js`.

**1.4 Fix the static-file serving path** (15 min)

Wherever the backend currently mounts `frontend/dist/` — it's mounting
a path relative to the project root, which works during `uv run` from
the repo but breaks in an installed wheel. Update to use
`importlib.resources` or `Path(__file__).parent` resolution to find
the bundle inside the installed package.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Frontend bundle path resolution breaks in installed wheel | Use `importlib.resources.files("backend") / "../frontend/dist"` or move dist into the `backend` package at build time |
| `npm` not available on builder's machine | Build hook fails loud; document `npm` as a dev/release prerequisite in CONTRIBUTING.md |
| WeasyPrint system libs not present | Document `brew install pango cairo libffi` / `apt install libpango...` in README install section. Won't catch every Linux distro but covers macOS and Debian/Ubuntu |
| Playwright Chromium not installed | On first Refresh click, the backend detects missing Chromium and shows a one-liner instruction in the toast: `uvx --from claude-explorer playwright install chromium`. Alternative: auto-install on first capture (slow first-run UX but zero friction) |
| Trademark concern on "claude-" name | Add disclaimer to README: *"This is an independent project, not affiliated with or endorsed by Anthropic. 'Claude' is a trademark of Anthropic."* |
| Version bump policy | Manual for V1. Adopt `hatch-vcs` post-V1 to derive version from git tags |

## Revisions (2026-05-13 follow-up answers)

### 1. End users vs builders re: npm

The wheel that lands on PyPI contains the **pre-built** React bundle as
static files. The end user (running `uvx claude-explorer serve`) needs
only Python + uv. **No npm. No node.** They download static JS/CSS that
FastAPI serves like any other web asset.

`npm` is only required for two paths:
- **Builders/maintainers** running `uv build` before publish (the build
  hook compiles the React app, then bundles the output into the wheel).
- **Contributors** doing `pip install git+https://...` directly from
  GitHub. That path runs the build hook on their machine and needs
  Node. We document this in CONTRIBUTING.md as the intentional
  contributor path; end-users are explicitly told to use PyPI.

This is the standard model for any Python package shipping a web UI
(Django admin, JupyterLab, Streamlit components — all build their JS at
release time, none make end-users install Node).

### 2. License — keep enterprise options open

**Recommendation: Apache-2.0 + a CLA (Contributor License Agreement).**

- **Apache-2.0** over MIT: clearer patent grant (matters in enterprise
  procurement conversations) and explicit handling of contributions —
  but still permissive enough that adoption isn't gated.
- **CLA on every PR**: contributors agree their code can be relicensed
  by the project owner. This is what lets you later:
  - Ship an enterprise edition under a separate commercial license.
  - Dual-license (Apache for community, commercial for enterprise).
  - Re-license to BSL/Elastic/SSPL if the market shifts.

Without a CLA, every contributor's commits are locked to the original
license — relicensing requires getting every contributor to re-sign,
which is impossible at scale.

**How to add a CLA cheaply**: [CLA Assistant](https://cla-assistant.io)
on GitHub. Free. Bot comments on every PR with a one-click sign flow.
Common stack used by GitLab, Elastic, Confluent — proven path.

**Alternative model**: open-core. Keep `claude-explorer` MIT/Apache
and build a separate proprietary `claude-explorer-enterprise` repo that
plugs into the open core via documented extension points. No CLA
needed because the enterprise repo is your code top-to-bottom. This is
the GitLab / Sentry / Mattermost pattern. More complex architecturally
but gives the cleanest legal split.

**Sidestep alternative**: stay MIT/Apache, build enterprise features
as a SaaS product (closed source, hosted) on top of the open client.
The client stays free; the value-add is the hosted backend. Sentry's
historical model.

For V1: **Apache-2.0 + CLA Assistant**. Both decisions are reversible
later if you adopt open-core or SaaS.

### 4. Contributor instructions — new `CONTRIBUTING.md`

Add `CONTRIBUTING.md` at repo root. Outline:

```markdown
# Contributing to claude-explorer

Thanks for the interest. This is a solo-maintained project; PRs are
welcome but please open an issue first for anything bigger than a
typo so we can agree on the approach before you write code.

## Prerequisites
- Python 3.11+ (uv will bootstrap if missing)
- Node.js 20+ + npm (for the React frontend build)
- macOS system libs: `brew install pango cairo libffi` (for PDF export)
- Linux system libs: `apt install libpango-1.0-0 libcairo2 libffi-dev`

## Repo setup
```bash
git clone https://github.com/rpeck/claude-explorer
cd claude-explorer
uv sync --extra dev
cd frontend && npm install && cd ..
uv run playwright install chromium
```

## Running locally (dev mode)
- Backend (with auto-reload): `uv run uvicorn backend.main:app --reload --port 8000`
- Frontend (separate dev server): `cd frontend && npm run dev`
- Tests:
  - Backend: `uv run pytest backend/tests -q`
  - Vitest: `cd frontend && npm run test:run`
  - Playwright: `cd frontend && npx playwright test`

## Code style
- Python: ruff + pyflakes (CI enforces zero warnings on changed files)
- TypeScript: tsc --noEmit clean, eslint via vite-plugin
- Testing discipline: see `CLAUDE-TESTING.md` for black-box / bidirectional
  rules and the Playwright "deterministic settle barrier" pattern
- Article changes: see voice notes in `PLANS/articles/PROCESS/`

## Pull request process
1. Open an issue describing the change (skip for typos / docs).
2. Fork, branch, commit.
3. Run all three test suites locally before pushing.
4. On PR open, the CLA Assistant bot will ask you to sign the CLA
   (one-time, takes 30 seconds).
5. PR review focuses on: test coverage, voice consistency for any
   article/doc changes, no silent regressions.

## What we don't accept
- PRs that paste official Anthropic source code or API responses
  beyond the minimal samples already in test fixtures.
- PRs that bypass the credential capture / fetch model (e.g.,
  scraping claude.ai HTML); we deliberately use only the same
  endpoints the official clients use.
- Feature additions without tests.
```

### 5. MCP server as first-class in the description

The `mcp_server/` package exists and exposes 5 FastMCP tools
(`list_sessions`, `list_projects`, `get_session_outline`,
`get_messages`, `export_session`) that let Claude itself — or any
MCP-aware agent — query your conversation archive programmatically.
That's not a side feature; it's one of the three primary surfaces this
project provides (web UI, exports, MCP).

**Updated project description** (replacing the prior one-liner):

> *"Local-first toolkit for your Claude history: a browseable web
> archive, exporters (Markdown + PDF), and an MCP server — covering
> both Claude Desktop conversations and Claude Code sessions, indexed
> for fast search and queryable by Claude itself via MCP."*

**Shorter punchy variants for PyPI summary (≤200 chars)**:

- *"Local archive + web viewer + MCP server for Claude Desktop and
  Claude Code conversations. Browse, search, export, and let Claude
  query its own history."*

- *"Your Claude history, local and queryable. Web UI, Markdown/PDF
  exports, and a built-in MCP server for both Claude Desktop and
  Claude Code sessions."*

Use the second on PyPI (more concrete + ends with the primary use
cases). Use the first as the README headline.

**Article impact**: Part 2 currently doesn't surface the MCP server
at all. If MCP is first-class, it deserves either:
- A short subsection in Part 2 mentioning *"and there's also an MCP
  server — covered in Part N"*, OR
- A dedicated Part (3 or later) for the MCP server with its own tour.

I'd recommend the dedicated part — the MCP server's value prop is
distinct (Claude querying Claude's own history) and deserves its own
narrative arc. Adding a one-liner to Part 2 is fine in the meantime so
readers know it exists.
