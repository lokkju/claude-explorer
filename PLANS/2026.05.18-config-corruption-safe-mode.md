# 2026-05-18 — Corrupt `config.json` safe-mode follow-up

## Context

Commit `ed9dca9` (`fix(config): catch JSONDecodeError on corrupt
config.json so the app still boots`) fixed the immediate crash class:
a single-byte corruption in `~/.claude-explorer/config.json` no longer
prevents `claude-explorer serve` from booting. The reader now catches
`(JSONDecodeError, OSError, TypeError)`, logs a warning, and
`continue`s to the next candidate config path.

The `/code-audit` Round-2 Red-Teamer (Gemini-2.5-Pro filling for
quota-exhausted GPT-5.2) flagged a residual risk the immediate fix
does NOT address:

> *"On parse failure you fall through to the default `data_dir` and
> `claude_dir`. If the user's intended `data_dir` lived somewhere
> custom — say an external SSD or a synced cloud folder — the next
> fetch silently writes to `~/.claude-explorer/conversations/` instead
> of the user's real location. They don't lose data, but they orphan
> the existing archive AND start building a parallel archive in the
> wrong place. The user has no signal beyond a single log line that
> nothing reads."*

The Council CTO synthesis deferred the mitigation as out-of-scope for
a `≤25 LOC implementation-only` fix (it crosses module boundaries:
`config.py` + every writer + CLI surfaces + UI banner). This plan
files it as a dedicated follow-up so the broader sweep doesn't lose
the thread.

## Failure scenarios this plan addresses

1. **Silent data-dir orphaning** (Red-Teamer's primary case): user's
   `config.json` has `{"data_dir": "/Volumes/SSD/claude-archive"}`,
   a partial save corrupts the JSON, server boots with defaults,
   `claude-explorer fetch` writes to `~/.claude-explorer/
   conversations/`. User notices weeks later when the SSD archive
   stops growing.

2. **Silent migration mid-flight**: a lifespan-migration handoff
   between `legacy_home_dir()` and `canonical_home_dir()` that
   partially completes leaves a corrupt config on one side; the
   reader falls back to defaults instead of recovering.

3. **Permission-denied vs corruption confusion**: a chmod accident
   makes the config unreadable; the current code logs the same
   warning as a parse failure, masking the distinction the user
   needs to take the right action.

## Out of scope

- **Auto-repair**. Repairing a partial JSON is best-effort at best and
  silently overwriting a user-edited file is worse than the boot
  crash we just removed. Refuse to write until the user acts.
- **Schema migration**. Separate concern. The plan in
  `PLANS/2026.05.13-pypi-publish-scope.md` already tracks schema
  versioning.
- **Generic config-file framework**. This plan is scoped to
  `config.json` specifically.

## Design

Three layers, each gateable. Land in this order so a partial ship
leaves the system in a strictly safer state than today.

### Layer 1 — `Settings.is_corrupt` flag

`backend/config.py:Settings` gains a single field:

```python
@dataclass
class Settings:
    ...
    config_corrupt_reason: str | None = None
```

Populated by the parse loop when ANY config-path read fails. Stays
`None` when every present config parses cleanly OR when no config
file exists (the no-config case is legitimate, not corrupt). Logged
at WARNING with the path and underlying exception.

The flag is descriptive (a human-readable string), not just a bool —
the UI needs to render *why* the config is corrupt, not just *that*
it is.

### Layer 2 — Writer gating

Every code path that WRITES to disk under `settings.data_dir` checks
the flag first and refuses if set:

- `backend/routers/fetch.py` — the fetch pipeline (capture + fetch
  conversations).
- `backend/routers/bookmarks.py` — bookmarks file writes.
- `backend/routers/preferences.py` — preferences file writes.
- `fetcher/cli.py` — `claude-explorer fetch`, `claude-explorer
  install-watcher` (when it writes the launcher template).
- `backend/cc_watcher.py` — the writable cache paths
  (`summary_cache.sqlite`, `search-index.sqlite`).

Gating shape:

```python
def _refuse_if_config_corrupt(settings: Settings) -> None:
    if settings.config_corrupt_reason:
        raise HTTPException(
            status_code=503,
            detail=(
                "Config file is corrupt; refusing to write to avoid "
                f"orphaning your archive. Reason: "
                f"{settings.config_corrupt_reason}. "
                "Fix or remove ~/.claude-explorer/config.json and "
                "restart."
            ),
        )
```

Reads remain unconditional — the user can still browse what's already
on disk even when the writer gate is closed. The 503 status preserves
the "service degraded" semantics for the frontend retry logic.

### Layer 3 — UI banner

`frontend/src/components/ConnectionStatus.tsx` (or a new sibling
`ConfigCorruptionBanner.tsx`) consumes a new `/api/config` response
field:

```python
class AppConfig(BaseModel):
    data_dir: str
    config_corrupt_reason: str | None = None   # NEW
```

When non-null, render a persistent (non-dismissible) banner at the
top of the app shell:

> ⚠️ **Config file is corrupt** — `~/.claude-explorer/config.json`
> didn't parse. Writes are disabled until you fix it.
> Reason: `<config_corrupt_reason>`. [Open file] [Read docs]

Non-dismissible because dismissing it would silently re-enable the
data-orphaning failure mode. The user must act on the underlying
file to remove the banner.

## Tasks (DAG)

```
T1 — Settings.config_corrupt_reason field + populate
  │
  ├── T2 — Writer gate helper + apply to 5 writer sites
  │       │
  │       └── T3 — Backend tests: every writer 503s when corrupt
  │
  └── T4 — /api/config response field + UI banner
          │
          └── T5 — Frontend tests: banner renders when reason set;
                  writer UI affordances (Refresh button, etc.)
                  disable/grey when reason set
```

T1 is the prerequisite for every other task. T2 and T4 can land in
parallel after T1. T3 and T5 are the test slabs for their respective
implementations.

## Tests

Bidirectional pairs throughout:

- `test_config_clean_does_not_set_corruption_flag` — every valid
  config (canonical only / legacy only / both present and matching)
  leaves `config_corrupt_reason` None.
- `test_config_corrupt_sets_reason_with_path_and_exception` — for
  each of `{JSONDecodeError, OSError, TypeError}`, the flag is
  populated with the path and a one-line summary.
- `test_writer_routes_503_when_corrupt` — fetch, bookmarks PATCH,
  prefs PATCH, install-watcher write all return 503 when flag set.
- `test_writer_routes_200_when_clean` — sibling positive case.
- `test_read_routes_unchanged_when_corrupt` — /api/conversations,
  /api/search, /api/conversations/{uuid} all still 200 — reads
  remain available so the user can recover their data.
- `test_api_config_carries_corruption_reason` — round-trip from
  Settings field → /api/config JSON response.
- `test_banner_renders_when_reason_present` (vitest) — bidirectional
  pair with `test_banner_absent_when_reason_null`.

Per the rewritten `/code-audit` skill's Phase-4 blast-radius rule,
this touches >3 files and changes interface contracts (the new
`/api/config` field + the new 503 response from writer routes), so
the implementation belongs in its own dedicated PR with explicit
sign-off — not bundled under a routine commit.

## Migration / compatibility

- **Existing installs**: the new field defaults to None on existing
  configs. No migration needed.
- **External API consumers**: the new `/api/config` field is
  optional; older clients that ignore unknown fields continue to
  work. The new 503 from writers is a behaviour change — external
  scripts that write through the API (e.g. third-party tools that
  POST to /api/preferences) will see 503 where they previously saw
  500. Documented in the release notes.
- **The MCP server** in `mcp_server/server.py` calls
  `backend.search.search_conversations` directly (Python import) and
  doesn't touch writers, so the gate doesn't affect MCP behavior.

## Acceptance

- Boot with a hand-corrupted `~/.claude-explorer/config.json`:
  - Server starts.
  - `/api/config` returns `config_corrupt_reason` populated.
  - UI shows the banner.
  - `claude-explorer fetch` returns a clean error (not a stack
    trace) with the recovery instruction.
  - `/api/conversations` still returns the existing archive (reads
    unaffected).
- Repair the file: refresh the app, banner disappears, writers work
  again. No restart required (settings lru_cache cleared by the
  existing `_settings_refresh` flow — confirm or wire if missing).

## Verification

- `uv run pytest backend/tests` green including the new test slab.
- `cd frontend && npm test -- --run` green including the banner tests.
- Manual: corrupt the config file mid-session, observe the UI banner
  appearing without a restart (this validates the cache-clear path).

## Risks

- **Cache-clear path may not exist** for the corruption flag —
  Settings is lru_cached and the corruption check runs at construction
  time. The UI banner needs a way to RE-CHECK after the user fixes
  the file. Either (a) clear the lru_cache on every /api/config call
  (cheap, ~10µs), or (b) add an explicit "Recheck config" affordance
  in the banner. Pick during T1 design.
- **Refusing-to-write breaks the install-watcher install flow** if
  the install command also writes the launcher template. Audit
  whether the install path runs through the writer gate; if yes,
  the install command needs an explicit `--force-write-despite-
  corrupt-config` flag or the gate should exempt the launcher
  template (which lives outside `data_dir`).
- **External orchestration scripts** that previously saw 500 on
  corrupt config and retried may now see 503 forever (since the
  gate is sticky until the user fixes the file). Acceptable — the
  retry was incorrect behavior; failing fast is better.
