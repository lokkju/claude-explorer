# Backend test gap matrix (frontend-derived)

This file is the dual-view companion to
`PLANS/2026.05.07-frontend-api-contract.md`:

- **View 1: Per-endpoint summary.** 24 rows (25 with the dual-attachment
  routing — same shorthand used in P1.1). Navigational; lets a reviewer
  pick an endpoint and see the planned tests at a glance.
- **View 2: Clause × test matrix.** ~75 clauses against ~30 test files.
  Each cell is `✓ <test>:[Confidence]` / `partial: <note>` / `✗` / `n/a`.

**Confidence rubric** (applied to every `✓` cell):
- `high` — asserts semantics + value + negative-space (e.g. PATCH preserves
  unrelated keys; 404 detail content; SSE event order + payload shape).
- `medium` — asserts shape + some values; doesn't fully exercise contract
  (e.g. 200 + body has key X but value not pinned).
- `low` — asserts only status / existence; could still pass with a corrupted
  body or hardcoded sentinel value. **Flag for P3 upgrade.**

The five systemic gaps named in BACKEND TEST PLAN's Context section
should be visible in this matrix:

1. **Zero SSE-full-contract tests** — current `test_refresh_pipeline.py`
   asserts event presence but NOT termination, NOT header content-type,
   NOT exhaustive payload shape; `/fetch/start` has zero SSE tests.
2. **Zero concurrency tests** — `test_concurrent_refresh_returns_409`
   sets the flag manually rather than racing two requests.
3. **Zero security-adjacent tests** — path traversal coverage exists
   for `/api/cc-image` but not for `/api/attachments`; permission-bit
   coverage is partial.
4. **Almost-zero migration legacy seeds** — backend is a passthrough,
   but no test pins that explicitly with a v1-shape seed.
5. **~10 weak status-only files** — see View 2 `low`-confidence cells.
