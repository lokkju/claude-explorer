- **Principle:** B (file-size), with secondary C (anti-spaghetti)
- **Evidence:** `frontend/src/routes/ConversationPage.tsx` grew from 1696 ‚Üí 1773 (+77).
  The gate is ~1000. The file owns: virtualizer mount, source-badge rendering,
  header toolbar (copy + PDF + markdown buttons), a 5-stage scroll-to-highlight
  orchestration (lines ~774‚Äì905), bracket-key compact-marker nav, prelude toggle,
  several effects that each carry their own multi-line eslint-disable rationale.
  The diff also added another disable-line comment for `react-doctor/no-render-in-render`
  around `renderBubbleRow` (~line 1473) ‚Äî a real signal that the file is straining
  its current shape.
- **Why it matters:** every new feature on this page (Cowork badge, sandbox-path
  subline, virtualizer recovery) lands in the same already-overloaded module. The
  disable-line comments are individually well-justified, but their growing density
  is the smell ‚Äî each new feature now requires "where does this go in the
  1700-line file" judgement.
- **Suggested remedy:** before the next feature, extract three boring modules:
  `ConversationHeader` (source badge + meta + toolbar), `useScrollToHighlight`
  (the multi-stage orchestration with its 6+ deps), and `useBracketCompactNav`
  (the [/] handler now using `useEffectEvent`). Each is mechanical and
  behavior-preserving.
- **Blocks approval:** decomposition **question**, per rubric B ‚Äî never an
  automatic reject. Accept with "decomposition deferred to next polish pass" or
  schedule it.

### [P1] Source-badge ternary duplicated across two render sites

- **Principle:** F (canonical layer & reuse)
- **Evidence:** the three-way `CLAUDE_CODE / CLAUDE_COWORK / (else) Desktop`
  badge ternary appears in both `ConversationPage.tsx:1125-1140` and
  `ConversationList.tsx:813-820`. Same three branches, same icons
  (`Terminal` / `Sparkles` / `MessageSquare`), same color triple.
  Sidebar.tsx uses a parallel `<SelectItem>` for the filter dropdown ‚Äî
  different surface but same mapping.
- **Why it matters:** F12 was the second time we shipped this badge
  (binary ‚Üí ternary). The next source (or rename, or visual tweak) is a
  three-file diff with three review surfaces and three opportunities to drift.
  The "purple Sparkles for Cowork" comment on each site is itself evidence ‚Äî
  the maintainer is documenting the invariant per copy rather than encoding
  it once.
- **Suggested remedy:** one `<SourceBadge source={...} variant="row"|"header" />`
  component owning the icon + color + label triple. Sidebar's dropdown stays
  separate (different a11y shape); the two body renderers collapse to one call.
- **Blocks approval:** yes (P1) unless explicitly deferred.

### [P1] "Phase 1 a11y nested label" pattern repeated ~10 times with the same suppression comment

- **Principle:** F (canonical layer & reuse), with secondary D (anti-magic)
- **Evidence:** the `<label className="...border... oxlint-disable-next-line
  react-doctor/label-has-associated-control"> <RadioGroupItem .../>
  <div>...</div></label>` shape appears 3√ó in SettingsPage.tsx (Theme),
  2√ó in SettingsPage.tsx (Keyboard Mode), 3√ó in SettingsPage.tsx (Export mode),
  and 3√ó in MarkdownExportDialog.tsx. Every site repeats the same multi-line
  rationale comment for the same suppression. `control-has-associated-label`
  suppressions on nested-checkbox labels appear another 4√ó (Settings + Sidebar
  + Dialog + ConversationPage).
- **Why it matters:** the disable-line comments grow every time we add an
  option. Each one is correct in isolation, but the canon now lives in 10
  comment blocks instead of one place. A future a11y refactor (say, switching
  from Radix to a custom radio) requires touching 10 sites with the same
  change.
- **Suggested remedy:** one `<RadioOptionCard value={...} active={...}
  title={...} description={...} icon={...} />` (and a sibling
  `<CheckboxRow label={...} />` for the nested-checkbox shape). Encapsulate
  the suppression once. Same as the Source-badge pattern ‚Äî repeated structure
  begs an abstraction.
- **Blocks approval:** yes (P1) unless explicitly deferred.

### [P1] Orphan preference keys removed in code but not from existing users' server-stored prefs

- **Principle:** E (boundary cleanliness, drift edge)
- **Evidence:** `markdownBundleImages` and `markdownDialect` are deleted from
  `SettingsContext`, `SettingsPage`, and `MarkdownExportDialog` (good ‚Äî kills
  the dead-write bug). But any user who previously ran the broken code still
  has those keys in their stored `~/.claude-explorer/preferences.json`. The
  new code never writes them, never deletes them, and never reads them. They
  become invisible cruft.
- **Why it matters:** silent abandonment, not a runtime bug today ‚Äî but it
  forecloses a future where the *same* key name is reused. A V2 "bundleImages"
  tri-state radio would be ambiguous: is `true` the old boolean or a
  new-meaning string? Cleaner now than archaeology later.
- **Suggested remedy:** in `SettingsContext` (or a one-shot migration effect),
  PATCH `{ markdownBundleImages: null, markdownDialect: null }` once on first
  load of the new code, then mark migration done via the same sentinel
  pattern `FilterContext._migratedV1` uses. Or document the intentional ignore
  in `usePreferences` docs.
- **Blocks approval:** lightest P1 of the four ‚Äî accept-with-justification
  ("low-risk silent ignore, drop in next polish") is reasonable.
