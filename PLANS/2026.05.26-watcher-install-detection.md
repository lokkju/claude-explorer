# Detect-and-warn when the CC image-cache watcher isn't installed

**Status:** plan. Backend detection helper (`backend/watcher_status.py`)
+ dedup'd permanent-loss warning shipped 2026-05-26 as the
prerequisite. This plan covers the user-facing surfaces that consume
the detection result.

**Problem.** Without `claude-explorer install-watcher`, the CC
image-cache watcher only runs while `claude-explorer serve` is up.
Claude Code's `cleanupPeriodDays` rotates `~/.claude/image-cache/<sess>/<N>.png`
on its own schedule. Any image written + rotated during a "serve is
down" window is permanently lost to the explorer — the JSONL keeps
the `[Image: source: ...]` marker, but neither the live cache nor
the permanent cache (`~/.claude-explorer/cc-images/`) has the bytes.

On 2026-05-26 the user reported a flood of "CC image referenced by
conv X not on disk" warnings. Root cause: install-watcher had never
been run; `~10` image slots across that conversation fell into the
gap between Time Machine snapshot boundaries.

**Design principle.** The user must be told **at most once per session**
that the watcher isn't installed — louder than an INFO log line in a
file they don't tail, quieter than a modal that interrupts their
flow. The signal must persist across page reloads (it's a "configure
once" issue) but must be dismissable (it's not a runtime crisis like
[[corrupt-config-safe-mode]]).

## Surfaces

### 1. Backend startup log line

`backend/main.py`'s lifespan startup calls
`watcher_status.is_watcher_installed()` and logs once:

```
WARNING  backend.main: CC image-cache watcher not installed.
WARNING  backend.main: Run 'uv run claude-explorer install-watcher' to
WARNING  backend.main: prevent permanent image-cache data loss during
WARNING  backend.main: backend downtime. See PLANS/2026.05.26-watcher-install-detection.md.
```

When installed: one INFO line ("CC image-cache watcher: installed").
No spam, surfaces in supervised-job logs for diagnostic clarity.

### 2. API endpoint `/api/health/watcher`

New endpoint mirroring the existing `/api/config` corruption surface
(`backend/routers/config.py` precedent). Returns:

```json
{
  "installed": false,
  "platform": "darwin",
  "install_command": "uv run claude-explorer install-watcher",
  "docs_url": "/PLANS/2026.05.26-watcher-install-detection.md"
}
```

Read-only, cheap (cached probe), no auth gate. Same `lru_cache` clear
on the endpoint that `/api/config` uses for `Settings`, so a user who
runs install-watcher mid-session sees the banner clear on the next
poll without restarting the backend.

### 3. Frontend persistent banner

Pattern: clone `ConfigCorruptionBanner.tsx` (already in the codebase
per CLAUDE.md "Corrupt-config safe mode"). One important difference:

| Banner | Severity | Dismissible | Persistence |
|---|---|---|---|
| ConfigCorruptionBanner | data-loss-imminent | NO | until file is fixed |
| WatcherMissingBanner (new) | data-loss-during-downtime | YES (per-session via localStorage) | until install-watcher run |

The dismiss-key in localStorage: `watcher-missing-dismissed-v1`.
**Not** `-permanent` — bump the suffix if we want to re-show a
dismissed banner after a future release (e.g., after a watcher
config change requires reinstall). Re-show triggers also include:
the user opens a conversation that fires ≥3 permanent-loss
WARNINGs in one viewer session (live monitored via existing console
assertion fixture; passes a hint to React).

Wire-up:
- `frontend/src/components/layout/AppShell.tsx` mounts the banner above
  the conversation route, below the sidebar.
- Fetches `/api/health/watcher` on app boot, polls every 5 min.
- Renders text: "Image-cache watcher not running. Some Claude Code
  screenshots may be permanently lost during downtime. **Install
  watcher →**" with the install command in a copy-to-clipboard
  affordance.

### 4. Settings → "Diagnostics" tile

Long-tail surface for users who dismiss the banner and forget about
it. The Settings page already has a Diagnostics section (precedent:
the SQLite busy_timeout / FTS5 schema-version readout). Add a
"Background services" tile:

```
Background services
✓ Search index: schema v12, 260,473 messages indexed
✗ CC image-cache watcher: not installed
   Install: uv run claude-explorer install-watcher
   (Run this once per user account. The watcher auto-starts at login
   on macOS/Linux/Windows.)
```

Always visible (no dismiss). Color-coded.

**User decisions on 2026-05-26 (during Phase 2/3 implementation):**

- Q1 "Banner when no missing-marker events?" → **Show regardless.**
  Banner is preventative, not reactive. Implemented: no event-count
  gate; `useWatcherHealth` polls and `WatcherMissingBanner` renders
  whenever `installed === false`.
- Q2 "`claude-explorer serve` terminal warning?" → **Yes.**
  Implemented: `cli/main.py:serve` checks `is_watcher_installed()`
  and echoes a 4-line stderr banner before launching uvicorn.

## Test plan (TDD-strict; RED-first)

**Backend (Phase 2):**

```python
def test_lifespan_logs_warning_when_watcher_uninstalled(caplog, monkeypatch):
    monkeypatch.setenv("CLAUDE_EXPLORER_WATCHER_INSTALLED", "0")
    watcher_status.invalidate_cache()
    # Boot the app via TestClient; assert exactly one WARNING record
    # with the "install-watcher" string in the message and the docs
    # URL in the body.

def test_lifespan_logs_info_when_watcher_installed(caplog, monkeypatch):
    monkeypatch.setenv("CLAUDE_EXPLORER_WATCHER_INSTALLED", "1")
    watcher_status.invalidate_cache()
    # Boot; assert exactly one INFO record and zero WARNING records
    # mentioning the watcher.

def test_health_endpoint_returns_install_status(client, monkeypatch):
    monkeypatch.setenv("CLAUDE_EXPLORER_WATCHER_INSTALLED", "0")
    watcher_status.invalidate_cache()
    r = client.get("/api/health/watcher")
    assert r.status_code == 200
    assert r.json()["installed"] is False
    assert "install-watcher" in r.json()["install_command"]

def test_health_endpoint_reflects_mid_session_install(client, monkeypatch):
    """Bidirectional: user runs install-watcher mid-session, the
    endpoint reflects the new state within one poll."""
    monkeypatch.setenv("CLAUDE_EXPLORER_WATCHER_INSTALLED", "0")
    watcher_status.invalidate_cache()
    assert client.get("/api/health/watcher").json()["installed"] is False
    monkeypatch.setenv("CLAUDE_EXPLORER_WATCHER_INSTALLED", "1")
    # The endpoint MUST invalidate the cache on each call (or rely
    # on a TTL ≤ 60s). Pin this behavior — without invalidation,
    # the banner would lie about an install that just happened.
    assert client.get("/api/health/watcher").json()["installed"] is True
```

**Frontend (Phase 3):**

E2E in `frontend/e2e/watcher-missing-banner.spec.ts`:
- Mock `/api/health/watcher` to return `{installed: false}` → banner
  visible with install command + dismiss button.
- Click dismiss → banner hidden, localStorage `watcher-missing-dismissed-v1=1`.
- Reload → banner stays hidden.
- Mock endpoint to return `{installed: true}` → banner stays hidden
  regardless of localStorage (the install solved it; don't gaslight).
- Force ≥3 permanent-loss events in one conversation view (mock the
  conversation detail response with three `[Image: source: ...]`
  markers pointing at fake-missing paths) → banner re-shows even
  with dismiss flag set.

Console assertion: zero unexpected errors/warnings during the toggle
flow per CLAUDE-TESTING §5.15.

## Out of scope

- Auto-install of the watcher (CLAUDE.md: "no CLI for normal ops" —
  this is the edge case where the CLI IS the only option, by design).
- Modal/blocking UI. Banner is the right severity tier.
- Mid-process re-detect on a 1s timer. 5min poll is fine; install-watcher
  is a one-shot user action with an explicit "loading…" affordance.
- Telemetry on banner show/dismiss rates. V2.

## Open questions

- Should the banner appear when the user has NEVER seen the explorer
  loss (i.e., they have zero conversations with missing markers)?
  **Default: yes.** Prevention > reaction. They installed the app;
  showing one persistent banner during onboarding to ask them to
  install-watcher is reasonable.
- Should `claude-explorer serve` print a one-line warning at process
  start when the watcher isn't installed? **Default: yes** (Phase 2
  covers the structured logger; this is the human-readable terminal
  surface).
