## Detailed divergences

### Case 2 (❌) — Argful slash commands carry `is_command_marker=True`

**Spec invariant X1 (violated):**
> `is_command_marker=True` AND `slash_command_args is not None and slash_command_args.strip() != ""` MUST NOT both be true.

**Code reality (`backend/claude_code_reader.py:297`):**
```
"is_command_marker": True,
```
This line is INSIDE `_collapse_local_command_triplets`, set unconditionally for every collapsed run regardless of whether the run is argless or argful. The collapser DOES already distinguish via the private `_argless_marker` flag (L311) for the prelude pass, but it does NOT use that distinction to gate `is_command_marker`.

**Consequence:**
- The `Message.is_command_marker` field is `True` on argful markers like `/coding Real prompt here.`. Any downstream consumer that reads `is_command_marker` as "this is a marker badge, suppress full bubble" would suppress the user's real prose.
- The frontend `MessageBubble` currently does NOT branch on `is_command_marker` to suppress the bubble — it renders the body unconditionally via the `slash_command` badge above (L204-206). So the user-visible damage is currently limited.
- BUT: any future feature (export filtering, search-result hiding, accessibility "skip markers" affordance) that reads `is_command_marker` will treat argful markers as noise. This is the user's "recent bug" pattern.

**Tests:** there is no test pinning that `is_command_marker == False` for argful markers. The existing `test_argful_command_marker_carries_args_as_text_and_slash_command` asserts the badge name and text body but does NOT assert `is_command_marker == False`. **Missing invariant test.**

**File:line evidence:** `backend/claude_code_reader.py:297`, plus
`test_local_command_triplet_collapse.py:393` (only asserts `is True`,
never asserts `is False` for argful).

### Case 1 / 4 (⚠️) — Export pipeline ignores `is_prelude` and `is_command_marker`

**Spec contracts (violated):**
- Case 1 "Export contract": argless markers excluded from markdown/PDF.
- Case 4 "Export contract": prelude markers excluded from all exports.

**Code reality (`backend/export.py:302`):**
```python
for message in conversation.messages:
    if message_has_visible_content(message, include_tools):
        lines.append(message_to_markdown(message, include_tools))
```
`message_has_visible_content` (L159-182) checks `text`, `content`,
images. It does NOT consult `is_prelude` or `is_command_marker`.

**Consequence:**
- Markdown bundle exports (CommonMark and Obsidian) include `"Session: /exit"` as a "You:" message at the top.
- Inline single-file Markdown exports leak the same.
- PDF export (via `conversation_to_html`, L309) inherits the same iteration; likely same leak (audit didn't read the full HTML pipeline, but it uses the same Message list).
- **`assistant_canned_response_consumed`** is also not consulted — exports include the marker but DON'T include the absorbed "No response requested." (because it was dropped by `_fold_canned_assistant_responses_into_marker` before export, which is fine).

**Tests:** no test asserts exports exclude markers/preludes. The
existing export tests (`test_export_bundle.py`, `test_export_pdf_images.py`) appear to focus on content correctness and image embedding, not marker filtering. **Missing test.**

**File:line evidence:** `backend/export.py:302`,
`backend/export.py:159-182` (visibility helper),
`backend/export.py:944` (PDF body renderer also iterates).

### Case 6 (⚠️) — Thinking blocks have no frontend renderer

**Spec contract:** "When shown: render as a distinct, muted collapsible block immediately preceding the main assistant text."

**Code reality (`MessageBubble.tsx:499-534`):**
```ts
switch (block.type) {
  case 'text': ...
  case 'tool_use': ...
  case 'tool_result': ...
  case 'image': ...
  default: return null
}
```
There is **no `case 'thinking':`**. Thinking content blocks fall through to `default: return null` and are silently dropped from the viewer.

**Consequence:**
- Backend `_extract_searchable_text` (search.py L135-140) DOES include thinking content in the FTS5 index when `include_tool_calls=True`. So a user searches for text inside a thinking block and gets a hit — but clicking the hit lands on a message that, in the viewer, shows NO thinking content (because it's not rendered). The result is a search "ghost" where the snippet says X but the bubble shows nothing matching.
- The viewer's "Tools" toggle currently has no impact on thinking display because thinking isn't rendered at all.

**Tests:** no frontend test pins the contract for thinking blocks. `test_search_includes_slash_command.py` doesn't cover thinking. **Missing invariant test on both sides.**

**File:line evidence:** `frontend/src/components/message/MessageBubble.tsx:499-534`.

### Case 5 / 6 (⚠️) — Search-toggle filter is per-message, not per-match

**Spec invariant X7 (refined):**
> A message M whose ONLY match site is inside `tool_calls`/`tool_results` MUST appear in search results when `showToolCalls=True` and MUST NOT when `showToolCalls=False`.

**Code reality (`backend/search.py:308-427`):**
The linear-scan path uses a cached `_extract_searchable_text` projection that toggles `include_tool_calls=False` to OMIT tool/thinking blocks from the haystack. Then `pattern.finditer(text)` runs on the filtered text. If `text` is empty after filtering, the message is skipped (L387-388: `if not text: continue`). This satisfies the spec invariant for the linear-scan path.

**FTS5 fast path (`backend/search.py:430-520`):** Step 1 calls `idx.query(...)` which returns matches based on the FULL projection (the index was built with `include_tool_calls=True`). The fast path then walks matched conversations and re-runs the snippet/regex loop with the toggle applied. **The re-run uses the SAME `_extract_searchable_text` with the user's `include_tool_calls` flag (L283-287)** — so the per-message regex on the projection IS toggle-aware, and a message whose match was only in tool content will produce no snippet hit and be dropped.

**Caveat (documented at search.py:259-266):** the FTS5 step 1 LIMIT 5000 may evict plain-text matches if 5000+ tool-only matches rank higher. **This is a known accepted drift.**

**Verdict:** the implementation matches the spec invariant in practice for typical corpora. The drift is documented and risk-accepted.

### The greedy `<command-args>` regex bug (confirmed)

**User's report:** for `/exit` markers dated 2026-04-21 and 2026-04-24 in session `76fe578b-…`, the marker text rendered as
`'</command-args>\n<command-name>/exit</command-name>...'`.

**Code (`backend/claude_code_reader.py:102`):**
```python
_LOCAL_CMD_ARGS_RE = re.compile(r"<command-args>(.*)</command-args>", re.DOTALL)
```

**Confirmed mechanism:** the regex is greedy (`(.*)`), single-shot, and `re.DOTALL`. The `_extract_local_command_name_and_args` function (L136-157) runs `_LOCAL_CMD_ARGS_RE.search(text)` on the raw row text. If a single CC JSONL row contains **multiple concatenated triplets** in its `message.content` string (e.g., two `<command-name>/exit</command-name>\n<command-args></command-args>` blocks glued together by a CC streaming-merge bug), the greedy match spans from the FIRST `<command-args>` to the LAST `</command-args>`. The captured args text includes the intermediate `</command-args>\n<command-name>/exit</command-name>` substring — exactly matching the user's observation.

**Why `_local_command_kind` (L105) doesn't catch it:** the kind classifier only checks `startswith("<command-name>")`. A multi-block row still starts with `<command-name>`, so it classifies as `command`. The collapser then extracts name+args from THE WHOLE ROW with the greedy regex.

**The existing test that "covers" this case (`test_marker_args_with_literal_close_tag_in_body_survives_greedy_regex` at `test_local_command_triplet_collapse.py:650-687`)** explicitly DESIGNS for the user-pasted single-block case ("I'm debugging a parser. </command-args>...") — it pins the greedy choice as intentional for that case. **It does NOT cover the multi-block concatenation case the user actually hit.**

**Test gap:** no test feeds a row whose text is two glued-together triplets and asserts the result is two separate parsed `(name, args)` tuples (or, fallback: that args is empty).

**Fix approach per Phase 1 spec § "The greedy regex bug":** replace single-shot greedy regex with a state-machine parser that bounds the args body to the next `<command-name>` tag if one appears before `</command-args>`. **Not implementing here per user instruction (Phase 3 says no fixes).**

## Implementation-fragility notes (not bugs, but at-risk)

- **Duplicate plumbing.** `_parse_message` (store.py:89) hand-copies four CC-only flags from `raw` to the Pydantic constructor. The CC-reader is the only producer of these flags, so a future producer addition (e.g., `is_thinking_present: bool`) must update both ends. A `**{k: raw.get(k, default) for k in CC_ONLY_FLAGS}` pattern would be safer but is a refactor, not a fix.
- **`_argless_marker` private flag.** Set in `_collapse_local_command_triplets` (L311) and read in `_flag_leading_prelude_markers` (L499). The Pydantic-default-`extra='ignore'` stripping is invariant — if a future Pydantic upgrade changes this default to `extra='allow'` or `extra='forbid'`, the private flag either leaks to the API or causes validation errors. Test at `test_local_command_triplet_collapse.py:740` pins the current behavior.
- **Search-index slash_command projection (search.py:96-98).** The slash command name is appended to the searchable text. The FTS5 tokenizer is `unicode61` which splits on `/`, so `/coding` indexes as the token `coding`. A query for the literal substring `/coding` works on the linear-scan path (regex match on raw text) but on the FTS5 path it depends on the prefix-search behavior. Documented but worth a black-box test.
