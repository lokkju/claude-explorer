# Plan: Backend test enhancement (frontend-spec-driven gap closure)

## Context

A research subagent just produced a frontend-derived API contract spec by
reading ONLY `frontend/src/**` — 24 endpoints, ~50 contract clauses,
explicit list of SSE event types, all `usePreferences` keys, the full
migration tombstone-keys story, and the image-fallback chain. The spec
reflects what the frontend EXPECTS (not what the backend DOES); that gap
is exactly the bug-hunting surface.

Inline audit of the existing 30 backend test files (~4476 LOC, ~155
tests) reveals five systemic discipline gaps relative to
`CLAUDE-TESTING.md` §5:

1. **SSE: zero full-contract tests.** Greps for `aiter_lines`,
   `httpx.stream`, `event: start|progress|complete|capture_*`,
   `text/event-stream` — **all empty**. `/api/fetch/start` and
   `/api/fetch/refresh` are the most user-visible parts of the API
   (the sidebar Refresh button) and have effectively zero unit-test
   coverage of the stream contract. `test_refresh_pipeline.py` is
   network-bound and ignored by default per yesterday's test sweep.
2. **Concurrency: zero tests.** Greps for `asyncio.gather`,
   `asyncio.Lock`, `409 Conflict`, `race`, `concurrent` — empty.
   `_refresh_in_progress` lock is contract; untested. Atomic-write
   recovery untested.
3. **Security-adjacent: zero tests.** Greps for `0o600`, `st_mode`,
   `../`, `symlink`, `path_traversal` — empty. `/api/cc-image?path=...`
   and `/api/attachments/.../...` take paths-from-user-input;
   credentials.json + preferences.json should be `0o600`. Untested.
4. **Migration legacy seeds: zero tests.** Greps for `polarity:
   'include'|'exclude'`, `savedFilters: null`, `_migratedV1`,
   `_migratedV2` — empty in `backend/tests/`. The frontend orchestrates
   the migrations, but a seeded-legacy backend smoke would catch
   deserialization regressions and the backend's per-key-overwrite
   contract that nulls tombstone keys.
5. **Weak (status-only) tests: ~10 files** never examine the response
   body — `test_cc_image_watcher.py`, `test_claude_code_title.py`,
   `test_compact_markers.py`, `test_export_*.py` (most), `test_search_*.py`,
   `test_store_layout.py`. The grep for tests that don't reference
   `.json()`, `.content`, `.text`, or `content_type` returns 10
   files. Many of those tests would still pass if the response body
   were corrupted.

Plus endpoint coverage holes:

- `/api/conversations/<uuid>/tree` — no dedicated test file.
- `/api/orgs` 500-on-corrupt-credentials — only 5 tests in
  `test_orgs.py`; need to verify the 500 path the frontend treats as
  a distinct UI state.
- `/api/bookmarks` — only 5 tests; missing DELETE 404, PATCH 404,
  POST 409-on-duplicate.
- `/api/fetch/conversation/<uuid>` — 401/404/503 friendly-detail
  responses partially in `test_fetch_errors.py` but body shape
  undertested.
- `/api/fetch/status` — `credentials_age_days` null vs nonneg
  boundary not exercised.

This plan targets those gaps with discrete, parallelizable
deliverables. Each task ships a specific failure-mode coverage that
matches a frontend contract clause.

## Council review (2026-05-07)

Two councilors reviewed in parallel. Both returned APPROVE-WITH-CHANGES with mostly orthogonal additions and one preserved disagreement on P2 ordering. The plan below folds all agreed edits in. The disagreement is recorded with both arguments verbatim so downstream readers (including Ultraplan) have full argument context.

### Gemini-3-Pro · data-model + Python-backend seat

Load-bearing edits (accepted):

1. **Pydantic `model_config = ConfigDict(extra='forbid')`** retrofit on all input models. Argument: this is a self-contained local tool with no external clients; if the frontend sends unknown fields, we WANT a 422 to instantly catch drift. Highest-leverage single change to prevent future gap accumulation. Folded as new P6.
2. **Reprioritize P2** to Security → Migration → Atomic-Write → Concurrency → SSE. Argument: "Security and data integrity always trump feature bugs. A path traversal vuln or permission-bit drift is catastrophic; SSE event dropping is a sidebar UX glitch." Folded with the GPT compromise (see disagreement below).
3. **Colocate security tests in endpoint-specific files** (`test_cc_image.py`, `test_attachments.py`) rather than a generic `test_security_paths.py`. Argument: "Tests should document the endpoint's full contract; colocate failure modes with the feature." Folded — with GPT's carveout that shared helpers live in a small `backend/tests/_security_helpers.py`.

Bug bets Gemini will accept being wrong about:

- SSE event-order regressions: **1-3** (likely missing `cache-control: no-cache`, or failing to terminate on abort).
- Permission-bit drift on preferences.json: **1** (heavy bet that atomic temp-file creation inherits `0o644` from system umask instead of explicit `0o600`).
- Path-traversal vuln on `/api/cc-image`: **0** (FastAPI `Path` + `pathlib.resolve` usually handles it; still test it).
- Migration deserialization regressions: **1-3** (legacy models in strict Pydantic parsers almost always fail on edge cases).

Other Gemini positions:

- "Frontend may overstate contracts": when a test surfaces drift on status code (e.g., frontend assumes 201 on bookmark POST, backend returns 200), assert the BACKEND'S deterministic reality and update the spec — don't contort the backend to match a frontend assumption unless the HTTP semantics are genuinely wrong.
- Atomic-write recovery test scope: monkeypatch `os.replace` to raise; do NOT over-engineer kernel-level fsync races or cross-mount tmp scenarios. pytest is for Python-level exception boundaries, not kernel reordering.
- Migration backend scope: backend's role is "round-trip the legacy shape cleanly + accept explicit-null tombstone PATCH". Do NOT add a defensive normalizer that strips legacy keys silently — that masks bugs.
- `BACKEND-TEST-GAPS.md`: dated snapshot in `PLANS/`, not a living root-level doc. Use as burn-down chart for this plan, then let it die. `CLAUDE-TESTING.md` is the living discipline doc.

### PRESERVED DISAGREEMENT — P2 ordering

> **Gemini's position:** strict serial Security → Migration → Atomic-Write → Concurrency → SSE. "A path traversal vuln or permission-bit drift is catastrophic; SSE event dropping is a sidebar UX glitch."
>
> **GPT's position (mild pushback):** "From UX-risk, SSE is the most user-visible (Refresh button); from safety-risk, security is. Test work IS parallelizable; security and SSE are both load-bearing; punishing SSE staffing for the order argument is a false economy. Keep Gemini's order for *merge-gating* and as the ranked priority for which fixes go first if multiple bugs surface — but don't block SSE work behind migration if staffing allows."

**Convergence (both councilors accepted in round 2):**

- Adopt Gemini's priority RANKING (Security > Migration > Atomic-Write > Concurrency > SSE) for: (a) which fix lands first when multiple bugs surface, (b) which task gets the senior reviewer, (c) merge-gating order if conflicts.
- Run task EXECUTION in parallel where capacity allows. Independent test files don't merge-conflict; SSE shouldn't be staffing-starved.
- Both councilors signed off:
  - Gemini: *"Priority-rank-but-parallel-staffing is a pragmatic resolution. Security and migration data-integrity regressions strictly block the merge queue and are prioritized for fixes ahead of SSE UX glitches; parallel test authoring is an acceptable economy of engineering resources."*
  - GPT: *"'Priority ranking without execution serialization' is still meaningful: it governs merge-gating, review attention, and bug-fix ordering when multiple failures surface at once. Meanwhile, parallel execution keeps time-to-signal low on SSE, which is user-visible and tends to surface surprising contract drift; delaying it doesn't make the system safer, it just delays discovery."*

The disagreement is recorded for downstream readers; the resolution is the compromise above.

## Goal

After this plan lands:

- Every contract clause in the frontend-derived spec has at least one
  strong-assertion backend test.
- Every SSE endpoint has a full event-order/types/payload/termination
  test using `httpx.AsyncClient` + `aiter_lines`.
- Every locked / atomic-op endpoint has a deterministic concurrency
  test.
- Every path-from-user-input endpoint has a path-traversal /
  symlink / extension-allowlist test.
- Every credential or preference file write asserts `0o600`.
- Every migration has a legacy-seed test that exercises the
  deserialization code path users will hit.
- Every weak status-only test is upgraded to assert the body
  contract.
- New `BACKEND-TEST-GAPS.md` ships as the persistent gap matrix; it
  becomes the spec for future test additions.

## Cross-cutting requirements (apply to every new/rewritten test)

Per the council:

- **Naming:** `test__<endpoint>__<scenario>__<contract_outcome>` (double-underscore separator). Example: `test__post_bookmarks__duplicate__returns_409`.
- **Docstring cites the contract clause ID(s)** from `PLANS/2026.05.07-frontend-api-contract.md` (the spec doc P1.1 produces). One line per clause; don't paste the full clause text.
- **Spec-driven discipline header** (the comment block above) at the top of every NEW test file. Existing files being strengthened in P3 don't need it retroactively but do gain a docstring citing the clause.
- **Rich assertion messages required on contract-class assertions** (event order/type, payload value/schema, negative-space). Not every `assert`. Example: `assert response.status_code == 201, f"POST /api/bookmarks expected 201 (Created) so the frontend can rely on body.id; got {response.status_code} ({response.text!r})"`.
- **Bidirectional verification:** P2 tests REQUIRE explicit revert-and-fail evidence in the commit message ("Verified test fails when X is reverted with message: ..."). P3 tests use canary sampling — one per file is sufficient.

## Phases (DAG)

Phases that touch disjoint files run in parallel. Each task is one
commit; multiple commits per phase OK. **Priority ranking** for
merge-gating + senior-reviewer assignment + bug-fix ordering follows
the council compromise: P0 → P2 (Security → Migration → Atomic-Write →
Concurrency → SSE) → P3 → P4 → P5 → P6. **Execution** parallelizes
where capacity allows; SSE work is not blocked behind migration.

## Risks

- **SSE tests + asyncio + httpx**: the test must own the lifespan; use `httpx.AsyncClient(transport=ASGITransport(app=app))` rather than the synchronous `TestClient`. Set `asyncio_mode = "auto"` in `pyproject.toml` if not already set.
- **Concurrency test flakiness**: never use `asyncio.sleep` as a barrier. Use `asyncio.Event` set/cleared by a monkeypatched generator step. The test should be deterministic on a 1-core CI runner.
- **Path-traversal tests must not actually leak**: assert REJECT (4xx + no leak), don't assert content from outside the data dir on success — that would mean the impl is broken AND the test allows it. Stick to `assert response.status_code != 200` AND `assert "passwd" not in response.content` AND the underlying file system access never happened (mock or `tmp_path` to confirm).
- **Permission bits on macOS may surprise**: `os.umask` and the user's environment can affect computed bits. Ensure the impl explicitly `os.chmod(path, 0o600)` after `os.replace` and the test asserts the post-rename mode.
- **Migration tests need real `tmp_path`**: `lru_cache.cache_clear()` AND `monkeypatch.setenv("CLAUDE_EXPORTER_DATA_DIR", str(tmp_path))` per `CLAUDE-TESTING.md` §5.1.
- **`BACKEND-TEST-GAPS.md` becomes a maintenance burden** if it lives forever. Plan for it: it's a snapshot from 2026-05-07; future agents update it ONLY when adding/removing endpoints or contract clauses, not on every test commit. Add a banner saying so.
- **The frontend-derived spec may overstate contracts** (frontend assumes things the backend doesn't actually promise — e.g., specific status codes, deterministic header values). When a test surfaces drift here, classify per `CLAUDE-TESTING.md` §3 Phase-3 categories: violation / ambiguity / infra / spec-gap. Use the council to decide which.
