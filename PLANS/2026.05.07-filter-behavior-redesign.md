# Plan: Filter Behavior redesign (v2 of composable filters)

## Context

CF1+CF2+CF3 shipped a composable graph (atoms with `polarity: 'include'|'exclude'`, groups with `match: 'all'|'any'`). The mathematically-clean predicate-composition model survived first contact with reality for about ten minutes before the user hit the predicted UX failure mode head-on:

> "I want to exclude any titles that match the two atomic patterns... it's not obvious how to do that. If I have two patterns, cron1 and cron2 and want to exclude all titles that match either of those, how do I do it?"

In v1 the user has to: create atom A (exclude cron1), atom B (exclude cron2), group "all of these" with [A, B], activate the group. Five clicks across two screens to express what they think of as one filter. The "exclude + any" warning we shipped catches *the wrong* combination but does not surface *the right* path.

## Council synthesis (2026-05-07, second pass)

- **Gemini-3-Pro:** "two-section atom" (atom owns includes[] + excludes[] simultaneously; groups stay polarity-less). One atom can express "show meetings, hide cancelled" in one row.
- **GPT-5.2-Pro:** "Behavior on every node" (atoms and groups both carry `behavior: 'hide' | 'show-only'`). One atom = one pattern list, OR'd, with Behavior chosen at the top. The cron case is now one atom.

I went with GPT's model + the optional addendum that Behavior also lives on groups. Reasons:
- Closer to the user's spoken mental model ("hide cron1 OR cron2").
- "Behavior" is a single radio-with-summary the user reads in 2 seconds, not a two-textarea atom editor.
- Groups can now express "hide combinations" directly without forcing the user to think about negating an inner predicate.
- Loses the ability to express "show meetings, hide cancelled" in *one* atom — but that's recoverable as `Group(show-only, all-of, [show-only meetings, hide cancelled])`. Two filters, but each is unambiguous.

## Data model (v2)

```ts
type Behavior = 'hide' | 'show-only'

interface BaseFilterNode {
  id: FilterId
  name: string
  enabled: boolean
  behavior: Behavior
}

interface AtomFilter extends BaseFilterNode {
  type: 'atom'
  patterns: string[]    // OR'd
  mode: 'glob' | 'regex'
  target: 'title'
}

interface GroupFilter extends BaseFilterNode {
  type: 'group'
  match: 'all' | 'any'
  childIds: FilterId[]
}
```

### Predicate

```ts
function evaluate(node: FilterNode, text: string, state: FiltersState, visited = new Set<FilterId>()): boolean {
  if (visited.has(node.id)) return true
  visited.add(node.id)

  let inner: boolean
  if (node.type === 'atom') {
    if (node.patterns.length === 0) inner = false  // empty atom matches nothing
    else inner = node.patterns.some((p) => patternMatches(text, p, node.mode))
  } else {
    const children = node.childIds.map((id) => state.nodes[id]).filter((c): c is FilterNode => Boolean(c) && c.enabled)
    if (children.length === 0) inner = false       // empty/all-disabled group matches nothing
    else {
      const fn = node.match === 'all' ? children.every : children.some
      inner = fn.call(children, (c) => innerMatches(c, text, state, new Set(visited)))
    }
  }
  // Apply behavior at the LAST step. inner=true means "matched"; behavior decides keep vs drop.
  return node.behavior === 'show-only' ? inner : !inner
}
```

`innerMatches` is the pre-behavior predicate of a child node (recursively `evaluate` the child but considering only its inner match, NOT its own behavior — wait no, actually we DO want each child's behavior to apply when composing). Let me re-think:

When a group has children, each child is evaluated through its own behavior; the group's match-all/any then composes the boolean RESULT of each child evaluation; the group's behavior then determines keep/drop based on that aggregated result.

```ts
function evaluate(node, text, state, visited): boolean {
  if (cycle) return true
  let raw: boolean   // "did this filter's pattern set / children match?"
  if (atom) raw = patterns.some(matches)
  else {
    const childPasses = children.map(c => evaluate(c, text, state, ...))  // each child returns its keep/drop
    raw = node.match === 'all' ? every(childPasses) : some(childPasses)
  }
  return node.behavior === 'show-only' ? raw : !raw
}
```

But then a Group{behavior:'hide', match:'any', children:[hideAtom1, hideAtom2]} composes weirdly: each hideAtom returns "true=keep" already; the group's any-of composes those keeps; then the group's hide negates again. Double-negation creeps back.

The cleanest decomposition: every node has a `matches` (raw match query) and a `passes` (keep/drop decision). Compose ONLY raw matches inside groups; behavior at the very top.

```ts
function rawMatches(node, text, state, visited): boolean {
  if (cycle) return false
  if (atom) {
    if (patterns.length === 0) return false
    return patterns.some(matches)
  }
  // group: match-aggregator over children's RAW matches (ignore child behavior here)
  const children = enabledChildren(node, state)
  if (children.length === 0) return false
  const fn = match === 'all' ? every : some
  return fn(children, c => rawMatches(c, text, state, visited))
}

function passes(node, text, state): boolean {
  if (!node.enabled) return true   // disabled = no-op
  const m = rawMatches(node, text, state, new Set())
  return node.behavior === 'show-only' ? m : !m
}

function applyActiveFilter(text, state): boolean {
  if (!state.activeId) return true
  const node = state.nodes[state.activeId]
  if (!node) return true
  return passes(node, text, state)
}
```

Cleaner: child behavior is IGNORED inside a group's composition. The group sees raw matches only. The user's mental model is: "this group matches when (all/any of) its children's PATTERNS match"; behavior is layered on top exactly once at the active filter.

Trade-off: an enabled group that contains an enabled child no longer reuses the child's behavior. If you want "hide cron AND hide standup combined under one umbrella," you set the group's behavior=hide, match=any, children=[cron-atom, standup-atom]. The atoms' own behaviors don't matter inside the group. Slightly weird but vastly simpler.

For users who want to *use* an atom both standalone and as part of a group, the atom's standalone behavior still applies when the atom is the active filter. Inside a group, behavior=hide-cron-by-itself becomes irrelevant; the group-level behavior governs.

I think this is acceptable. Document it: "Behavior applies once, at the top. Inside a group, member filters contribute their patterns; the group's behavior decides hide/show."

## Migration (v1 → v2)

1. Atoms: `polarity: 'include'` → `behavior: 'show-only'`; `polarity: 'exclude'` → `behavior: 'hide'`. Drop `polarity`.
2. Groups: add `behavior: 'show-only'` (least-surprise — the v1 `match: 'all'` group of `exclude` atoms became "all excludes pass", semantically equivalent to "hide if any matches" → would be `behavior: 'hide', match: 'any'`. But this requires inspecting the children. SAFER: just promote groups to `behavior: 'show-only'` and let users adjust if needed.) Document this choice in the migration banner v2 copy: *"Filters now have a Behavior (Hide / Show only). Groups defaulted to Show only — flip the toggle if you intended Hide."*
3. Sentinel: `_migratedV2: true` (separate from `_migratedV1` so we can track both phases).
4. PATCH the new shape, keep the legacy `_migratedV1: true` flag (don't null it; v1 ran successfully).

```
Patterns (one per line):
[ cron1                              ]

```

- Behavior radio at the TOP of the editor — it's the highest-impact decision.
- Single textarea, OR'd patterns.
- Inline summary line under patterns rephrasing what the filter does in plain English.
- Mode + patterns are below behavior.

### Manage Filters modal — group editor

```
Editing: [Name input]
Type: ( Atom )  ( • Group )

Behavior: ( • Hide matches )  ( Show only matches )

  [Scan Gmail x]  [Boring summaries x]
Add member: [select ▾]  [Add]

```

The "exclude + any" warning is GONE — Behavior at the top makes the user's intent explicit and the warning's failure mode is no longer reachable.

Unchanged. Still single Select with "All conversations" + every enabled filter + "Manage filters…".
