# V2 capture: read session cookie directly from Claude Desktop's encrypted storage

> **Status:** plan-only. Targets a post-V1 release (1.1.0 or 2.0.0, see §"Release strategy"). Created 2026-06-12 from an LLM-Council brainstorm that surfaced this option while addressing the Windows ARM64 mitmproxy install bug. Council notes preserved at end of doc.

## Context

There are TWO capture flows in v1.0.7:

**Default (`claude-explorer capture`)** — **Playwright headless browser.** Opens a Chromium window via Playwright, the user logs into claude.ai normally, the browser session yields the `sessionKey` cookie which gets written to `~/.claude-explorer/credentials.json`. This is the path most users take.

**Fallback (`claude-explorer capture --proxy`)** — **mitmproxy man-in-the-middle.** For users who can't log into claude.ai (e.g., SSO lockout) but Claude Desktop is still authenticated. Starts mitmproxy on a local port, user launches Claude Desktop with `--proxy-server`, mitmproxy intercepts the API headers and pulls the cookie out. Rarely used; an SSO-recovery escape hatch.

Both flows have install-and-operational costs that V2 storage-read could eliminate:

1. **Playwright is huge.** ~400 MB Chromium browser binary downloaded via `playwright install chromium` after the pip install. Long install time. Not a blocker (browsers are well-trodden territory) but real install-experience cost on every platform.
2. **mitmproxy has the v1.0.7 install bug.** mitmproxy v11.x ships Rust components (`mitmproxy-rs`, `aioquic`, etc.) with no prebuilt wheels for Windows ARM64 — a growing platform segment (Copilot+ PCs, Surface Pro X). Forces users to install Visual Studio Build Tools + Rust toolchain (~6 GB, ~45 min compile). Captured in [`2026.06.11-cross-platform-install-test-results.md`](2026.06.11-cross-platform-install-test-results.md). v1.0.8 PEP 508 marker makes it conditional on Win ARM64 (fix applied 2026-06-12) but the dependency remains on every other platform.
3. **mitmproxy operational friction.** Requires CA cert trust on Windows (in `Cert:\CurrentUser\Root`), launching Claude Desktop with `--proxy-server` + `--ignore-certificate-errors` flags, then keeping mitmproxy running while logging in.
4. **mitmproxy + Store-app incompatibility.** Claude Desktop on Windows is now distributed as a Microsoft Store packaged app (UWP-wrapped Win32 at `C:\Program Files\WindowsApps\Claude_*\app\Claude.exe`). Store apps live in a locked-down `WindowsApps\` directory (admin/SYSTEM ACL) and may ignore Chromium command-line flags entirely under AppContainer. This likely breaks `C6-W` in the cross-platform install matrix.

The V1.0.8 fix unblocks Win ARM64 installs but leaves both capture paths themselves unchanged.

## Proposed approach

Claude Desktop is an Electron app. It stores cookies in a Chromium-style SQLite database at predictable per-platform paths. We can **read the `sessionKey` cookie directly from that file**, decrypt it using the OS's native credential-store API, and write `credentials.json` ourselves — no proxy, no CA trust, no app relaunch, **and no Playwright browser launch**.

The headline V2 payoff isn't just removing mitmproxy — it's potentially removing **both** Playwright AND mitmproxy from the default install. claude-explorer's install footprint drops by ~400 MB (Chromium binary) plus the mitmproxy Rust dependency tree. Capture goes from "launch headless browser, prompt user to log in, ~30-60 sec" to "read a SQLite file, decrypt, ~1 sec."

### Per-platform paths and decryption

| Platform | Cookies path (most likely) | Local State path | Decryption |
|---|---|---|---|
| Windows | `%APPDATA%\Claude\Network\Cookies` | `%APPDATA%\Claude\Local State` | DPAPI via `ctypes` (`CryptUnprotectData`); the AES-256 master key is in `Local State`, itself DPAPI-protected |
| macOS | `~/Library/Application Support/Claude/Cookies` | `~/Library/Application Support/Claude/Local State` | Keychain via `keyring` (the password's stored entry name is per-Chromium-version) |
| Linux | `~/.config/Claude/Cookies` | `~/.config/Claude/Local State` | `gnome-keyring` / `kwallet` via `keyring`; fallback to plaintext if neither is running |

All paths and column names are **claims requiring verification** — Electron versions and Chromium versions shift these. The V2 implementation MUST start with an empirical survey on a real Claude Desktop install per platform before writing any decryption code.

### High-level flow

```
$ claude-explorer capture          # default; --method=storage
1. Locate Claude Desktop's cookie SQLite file for this platform (see paths above)
2. Open read-only with sqlite3 module
3. SELECT host_key, name, encrypted_value FROM cookies WHERE host_key LIKE '%claude.ai%'
4. For each row, decrypt encrypted_value using the platform-appropriate API
5. Find the row where name == 'sessionKey' → extract value
6. Locate org_id (probably also a cookie; or derive from Authorization header in a follow-up auth.claude.ai request using the sessionKey)
7. Write credentials.json with atomic 0o600 mode, same shape as today
8. Done in ~1 sec, no proxy, no relaunch
```

Critical: the file is held open by Claude Desktop while the app is running. We may need to copy it to a temp location first (Chromium does the same on its own internal cookie reads).

## Phased implementation

### Phase 1 — Spike (single platform)
- Pick one platform (macOS first — paths and Keychain semantics are most stable). Verify the cookie path, schema, and decryption work on a live Claude Desktop install.
- Write a standalone Python script (under `scripts/spikes/v2-cookie-read.py`) that prints the decrypted sessionKey without integrating into the CLI.
- Document any deviations from the table above in the plan results.

### Phase 2 — Cross-platform decryption
- Add Windows DPAPI + the AES-256 + GCM unwrap path (Local State + Cookies double-decrypt).
- Add Linux `keyring` path with a plaintext fallback.
- Each platform gets a unit test that mocks the OS credential-store API and asserts the decryption pipeline returns the expected plaintext for a fixed encrypted blob.

### Phase 3 — CLI integration
- Add `claude-explorer capture --method=storage` alongside existing `--method=browser` (current default, Playwright) and `--method=proxy` (mitmproxy fallback). Today's `--proxy` flag becomes an alias of `--method=proxy` for back-compat one release.
- Make `--method=storage` the **default** on platforms where it works.
- Order of fallbacks when the default fails: storage → browser (Playwright) → proxy (mitmproxy) → manual paste. Each step gets a clear error explaining why it's falling through and what the user needs to do.
- If Claude Desktop isn't installed at all, `--method=storage` exits immediately with "Claude Desktop required for storage capture; use `--method=browser` instead."

### Phase 4 — Remove Playwright + mitmproxy from default install
- Once `--method=storage` is the default AND has telemetry / user feedback showing it works for >95% of capture invocations on every supported platform, deprecate both browser and proxy paths from the default install.
- Move `playwright` and `mitmproxy` to `claude-explorer[fallback-capture]` extras. Default install drops by ~400 MB.
- Power users who want browser/proxy fallback can opt in: `pipx install claude-explorer[fallback-capture]`.
- Eventually drop both entirely once storage-read is bulletproof; closes the Win ARM64 install bug and the playwright-browser-install friction for everyone.

## Risks and unknowns

1. **Claude Desktop schema/path drift.** Electron major-version bumps could shift the storage shape. Mitigation: auto-detect by probing a small set of candidate paths; log which one matched; fall back to mitmproxy if none.
2. **Encryption-scheme shifts.** Chromium has rotated cookie-encryption strategies (v10 plaintext → v11 AEAD via OS key store → v12 with per-profile keys). Need to test against the version Claude Desktop currently bundles AND track upgrades.
3. **"Reaching into another app's storage" optics.** Technically the user owns both apps and both files, but reading another app's encrypted credential store is unusual. Worth raising with Anthropic Security before shipping; they may prefer a cleaner story (e.g., a Claude Desktop CLI flag that exports its own creds: `Claude.exe --export-credentials > creds.json`).
4. **File-lock contention on Windows.** Claude Desktop may hold the SQLite file with `FILE_SHARE_READ` only. Copy-then-read pattern is the standard workaround.
5. **OS keychain prompts.** On macOS, Keychain may prompt the user the first time `claude-explorer` reads the Claude-app entry. UX is OK once authorized; first-run experience needs eyeballing.
6. **Anti-virus / SmartScreen reaction on Windows.** A new third-party app reading another app's encrypted storage may trip Defender heuristics. Test on a clean Windows install before shipping.

## Why this is V2, not V1

- Cross-platform implementation is non-trivial (three OS APIs, three Electron-version probe strategies, three test matrices).
- The current mitmproxy approach is **shipping and working** for most users. V1 doesn't need to fix it; only Windows ARM64 install needs the immediate PEP 508 fix.
- Storage-read is **strategic**, not tactical. Worth doing right vs. shipping a half-working version.
