## Part 2 — E2E release-gate failure: analysis + recommended fix

### Symptom
The `v1.0.2` Release run (<https://github.com/rpeck/claude-explorer/actions/runs/26978874381>) failed the gate on the **Playwright (fixture mode)** job: **54 failed / 386 passed / 1 flaky**. The Python job passed. Publish never ran. Almost every failure is the same: `Error: Console errors during test (see CLAUDE-TESTING.md §5.15)` with the body being many `Failed to load resource: the server responded with a status of 500 (Internal Server Error)` (~929 total).

### Why it surfaced now (it's pre-existing, since ~June 2)
`main`'s **Tests** workflow has failed on every push since 2026-06-02. The older runs died fast on the e2e job's **"Verify fixtures are up to date"** step (the stale `is_temporary` fixtures), which `exit 1`s *before* the Playwright tests run — so the e2e tests hadn't actually run on `main` in days. Commit `1a992c90` fixed the freshness check, the Playwright tests finally ran, and these §5.15 console-error failures appeared. They are **not** caused by the Cowork fix or the fixture regeneration (the e2e tests use in-memory `makeSummary` mocks, not `backend/tests/fixtures`).

### Root cause
- The e2e webServer is **Vite only** (`frontend/playwright.config.ts` → `webServer.command: 'npm run dev'`, i.e. `vite`). There is **no backend**. `frontend/vite.config.*` proxies `/api` → `http://localhost:8765`, which isn't running, so any un-intercepted `/api/*` request returns **500** from Vite's dead proxy.
- Tests rely on `mockBackend()` in `frontend/e2e/fixtures.ts` to `page.route('**/api/...')`-intercept everything. There's a fail-loud catch-all `page.route('**/api/**', …→ 500)` (~`fixtures.ts:419`) so any *un-mocked* endpoint also 500s.
- The §5.15 console-error capture is an **`{ auto: true }`** fixture (~`fixtures.ts:760-790`) that records every `console`/`pageerror` and **throws after the test** if any error was seen (the throw is at ~`fixtures.ts:776`). So a single startup 500 fails the whole test.
- The app makes these startup calls on load: `/api/config`, `/api/orgs`, `/api/bookmarks`, `/api/preferences`, `/api/conversations`, `/api/health/watcher`. When any of them fire **before/outside** an installed route, they 500.

### Three distinct failure modes (don't assume one fix)
1. **Navigate-before-mock.** `beforeEach` calls `page.goto('/')`/`reload()` *before* `mockBackend()` (which runs in the test body). e.g. `e2e/keyboard-shortcuts.spec.ts:289`, `e2e/article-coverage-gaps.spec.ts` (the `Alt+<`/`Alt+>` describe). The startup calls during that navigation hit the dead proxy.
2. **Coverage gap even when mocked first.** `e2e/theme.spec.ts:4` and `e2e/settings.spec.ts:4` call `mockBackend()` *before* navigating and **still fail** — so the app is hitting an endpoint `mockBackend()` (no-arg defaults) doesn't install → catch-all 500. Likely a recently-added startup endpoint (`/api/orgs` or `/api/health/watcher` from the Cowork-multi-org / watcher work).
3. **Inconsistent `mockBackend` call conventions.** Some specs call `mockBackend(page)` (page as positional arg — e.g. `e2e/url-navigation.spec.ts:145`, `e2e/compact-markers.spec.ts:150`); others `mockBackend({ conversations, details })` (e.g. `e2e/keyboard-shortcuts.spec.ts:298`); others `mockBackend()`. These are not the same signature — at least one set is calling it wrong. Verify the actual `MockBackendFn` signature in `fixtures.ts` (~L89 type, ~L382 fixture).

### Affected specs (~30; from run 26978874381)
`url-navigation`, `sidebar-filters`, `compact-markers`, `redownload-conversation`, `cowork-multi-org`, `bookmarks`, `keyboard-shortcuts`, `fixtures-defaults`, `search-compact-auto-expand`, `connection-status`, `keyboard-navigation`, `spec-filters-modal-atom-editor`, `per-bubble-tools`, `per-block-copy-hover`, `image-attachments`, `conversations`, `search-pin-scope`, `article-coverage-gaps`, `theme`, `spec-filters-atom-behavior`, `spec-conversation-list-virtualized`, `spec-filters-group-combinator`, `search`, `markdown-export-dialog`, `spec-filters-modal-group-editor`, `spec-filters-migration-v1-v2`, `sidebar-behavior`, `settings`, `search-sidebar-scope-propagation`, `conversation-detail` (all under `frontend/e2e/`).

### Fix options considered
- **A — fixture-level baseline mock** (install default routes for the startup endpoints in an `{auto:true}` fixture before any navigation). Fixes all 54 in one place, but **adds more mocking** and touches shared infra used by all 113 specs. **The author (rpeck) is against broad mocking and rejected this.**
- **B — per-spec** (move `mockBackend(...)` before the `beforeEach` navigation, or replace `goto('/')`+`setItem`+`reload` with `page.addInitScript(() => localStorage.setItem(...))` to avoid the premature navigation entirely). Lower blast radius, but ~30 specs and modes 2–3 would mean *adding* mock coverage — again, more mocking.
- **C — RECOMMENDED (matches the author's preference: minimize mocks).** Stop intercepting `/api` with `page.route` and instead run a **real backend in fixture mode** as the e2e webServer (the project already supports a fixture-mode backend — see `mcp_server/tests/conftest.py` isolation and `scripts/generate_e2e_fixtures.py` which writes `backend/tests/fixtures/`). Point `playwright.config.ts` `webServer` at a backend started against those fixtures (e.g. `CLAUDE_EXPLORER_DATA_DIR=backend/tests/fixtures/... uvicorn backend.main:app`), drop most of `mockBackend()`, and keep `page.route` only for *targeted* cases (forced errors, SSE, image-404 paths). This is a real e2e rearchitecture but removes the whole class of "un-mocked endpoint → 500" failures and the §5.15 noise, and aligns with "minimize mocks to targeted cases."

### Key files
- `frontend/e2e/fixtures.ts` — `mockBackend` fixture (~L382-720), catch-all 500 (~L419-425), per-endpoint routes (config L460, orgs L471, preferences L483, bookmarks L541, health/watcher L445, search L514, …), §5.15 console capture + throw (~L760-790).
- `frontend/playwright.config.ts` — `webServer.command: 'npm run dev'` (~L50-52).
- `frontend/vite.config.*` — `/api` proxy → `:8765` (~L16-18).
- `.github/workflows/test.yml` — the `e2e` job (`Verify fixtures are up to date` step runs `scripts/generate_e2e_fixtures.py` + `git diff`; then `npx playwright test`).
- `CLAUDE-TESTING.md` §5.15 — the console-error rule these failures trip.
- `scripts/generate_e2e_fixtures.py` — the deterministic fixture generator (source of truth for `backend/tests/fixtures/`).

### To unblock the release independently of this
The Cowork fix is fully covered by the Python suite. The 76fe578b session's recommendation to the author was to **scope the release gate to the Python suites** (backend + fetcher + mcp_server) so `1.0.2` can ship, and treat this e2e overhaul as its own task (option C). As of this writing the author has not chosen that; they want this e2e work done in the `a70251a5` build session.
