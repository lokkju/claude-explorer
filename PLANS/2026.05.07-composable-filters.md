# Plan: Composable named filters

## Context

The current filter system has two live bugs and a dead end:

1. **Pin doesn't persist behaviorally across restart.** Each filter has a `pinned: boolean`, but a separate `activeFilterIds: string[]` actually drives whether the filter applies. Toggling Pin writes `pinned: true` to disk; the seeding logic that copies `pinned` filters into `activeFilterIds` runs once on first mount under a `useRef` guard, only if the array is empty. Live `~/.claude-exporter/preferences.json` has `pinned: true` on the user's "Scan Gmail" filter and `activeFilterIds: []` — the silent failure mode.
2. **Exclude filter not applied** for the same reason — the filter is pinned but not active.
3. **No way to model "different filters for different uses"** (Home / Client A / Client B).

The user's stated direction: *"All filters are named, and filters can be composed of ANDs and ORs of other named filters. I want a simple model that can be used to compose more complex filters in a way that's easy for users to understand."* So we replace the flat-list-of-toggles model with a **composable graph**: every saved filter is named; some are *atoms* (a pattern + polarity + mode) and some are *compounds* (AND or OR of other named filters). The "active" state becomes a single named filter (which may be a compound) — composition replaces enumeration.

## Council synthesis (2026-05-07)

Two councilors ran in parallel:

- **Gemini-3-Pro (data-model seat):** atoms vs compounds, predicate composition (AND = `every`, OR = `some`, atom-include = matches, atom-exclude = !matches), no explicit NOT (polarity covers it), reference children by ID, validate cycles at save time + runtime visited-set defense, persist a single graph blob. Migration: promote existing `pinned: true` filters into `activeIds`; strip `pinned`. Empty-pattern atom = always passes (least surprise).
- **GPT-5.2-Pro (UX seat):** rename to "Views" was considered but rejected here in favor of keeping the existing term *Filter* (the user is already fluent in it). Two-pane Manage Filters modal: list left, editor right. Group filter uses **chip-based membership editor** (no drag-drop dep). Single "Active filter: [select]" dropdown in sidebar (replaces today's chip rail). Don't bring back a "Quick access" star in v1 — the Pin concept burned the user once and earns no second chance under another name. Avoid AND/OR jargon in UI: label "Match all of these filters" / "Match any of these filters". Block deletion of a referenced filter; show "Used by: …" inline.

Convergence: keep "Filter" as the noun; data model as Gemini specified; UI as GPT specified, minus the star. Single active filter in v1 (UI shows one selector); the data layer can support multiple active in a future v2 with no schema change.

## Data model

```ts
// frontend/src/lib/filterEngine.ts (extended)

type FilterId = string

interface BaseFilterNode {
  id: FilterId
  name: string
  enabled: boolean    // disabled filters never apply and don't appear
                      // in the active picker or group-member picker
}

interface AtomFilter extends BaseFilterNode {
  type: 'atom'
  patterns: string[]
  polarity: 'include' | 'exclude'
  mode: 'glob' | 'regex'
  target: 'title'     // expand to other targets later
}

interface GroupFilter extends BaseFilterNode {
  type: 'group'
  match: 'all' | 'any'   // 'all' = AND; 'any' = OR (no AND/OR jargon in UI)
  childIds: FilterId[]
}

type FilterNode = AtomFilter | GroupFilter

// Persisted graph
interface FiltersState {
  nodes: Record<FilterId, FilterNode>   // O(1) lookup, cycle-safe
  activeId: FilterId | null              // exactly-one active in v1; null = "All"
}
```

Persistence under one preference key, e.g. `"filters"`, written through the existing `usePreferences` PATCH path. Top-level overwrite means we always send the full graph blob — fine because it's small.

### Evaluator (predicate composition)

```ts
// All recursive paths thread a `visited: Set<FilterId>` so a cycle in the
// graph (which we ALSO prevent at save time) cannot stack-overflow.
//
// IMPORTANT (Gemini council fix): the `enabled` check is applied at the
// group's child-filter step, NOT as an early-return inside evaluate().
// If we early-returned `true` for a disabled node, a `match: 'any'`
// group containing one disabled member would pass for every conversation
// (`some()` short-circuits on the first true). The right semantic is
// "disabled members are removed before the group's quantifier runs".
function evaluate(
  node: FilterNode,
  text: string,
  state: FiltersState,
  visited = new Set<FilterId>(),
): boolean {
  if (visited.has(node.id)) return true            // cycle: no-op
  visited.add(node.id)
  if (node.type === 'atom') {
    if (node.patterns.length === 0) return true    // empty atom = passes
    const hit = node.patterns.some((p) => patternMatches(text, p, node.mode))
    return node.polarity === 'include' ? hit : !hit
  }
  // group: drop orphans AND disabled children before applying the
  // quantifier. THIS is where 'enabled' gates evaluation.
  const children = node.childIds
    .map((id) => state.nodes[id])
    .filter((c): c is FilterNode => Boolean(c) && c.enabled)
  if (children.length === 0) return true           // empty/all-disabled group = passes
  const fn = node.match === 'all' ? children.every : children.some
  return fn.call(children, (c) => evaluate(c, text, state, new Set(visited)))
}

// Top-level wrapper for the active filter: if the active node is itself
// disabled, treat as "no filter active" (all conversations pass).
function applyActiveFilter(text: string, state: FiltersState): boolean {
  if (!state.activeId) return true
  const node = state.nodes[state.activeId]
  if (!node || !node.enabled) return true
  return evaluate(node, text, state)
}
```

Empty-pattern atom, empty-group, all-disabled group all pass. Disabled is gated at the GROUP level (children dropped before quantifier) and at the ROOT level (active filter that's disabled = no-op). Cycles short-circuit. Orphan child IDs are silently filtered.

## UI design

### Sidebar — replace the chip rail

Single shadcn `<Select>`:

```
┌────────────────────────────────────────┐
│  Search titles and projects...          │
├────────────────────────────────────────┤
│  Filter: [ All conversations  ▾ ]       │
│           ├── All conversations         │
│           ├── ────────────              │
│           ├── Scan Gmail                │
│           ├── Hide work-day chores      │
│           └── Client A                  │
└────────────────────────────────────────┘
```

The picker replaces today's three concepts (Pin checkbox + chip rail + activeFilterIds) with one. Selection = the active filter; "All conversations" = no filter. Disabled filters don't appear. The user's mental model is: *"Which view of my conversations do I want right now?"*

### Manage Filters modal — two-pane

```
[Manage filters]                                     [ + New filter ]

┌─────────────────────────────┬──────────────────────────────────────┐
│ Search [...]                │ Editing: <name input>                │
│                             │ Type:  ( • Atom )  (   Group   )      │
│ [scroll]                    │ Enabled: [toggle]                     │
│  Scan Gmail        [Atom]   │──────────────────────────────────────│
│  Boring summaries  [Atom]   │ ATOM editor (when type=atom)          │
│  Hide chores       [Group]  │   Polarity: ( include  /  exclude )   │
│  Client A          [Group]  │   Mode:     ( glob     /  regex   )   │
│  Receipts          [Atom]   │   Patterns (one per line):            │
│                             │   [ textarea                       ]  │
│                             │                                       │
│                             │ GROUP editor (when type=group)        │
│                             │   Match: ( • all of these  /  any of) │
│                             │   Members:                            │
│                             │     [Scan Gmail x] [Boring summaries x]│
│                             │   Add member: [select ▾] [Add]        │
│                             │                                       │
│                             │ Used by: <group names, if any>        │
└─────────────────────────────┴──────────────────────────────────────┘
                                                     [Cancel] [Save]
```

- "Add member" `<select>` lists every other filter except (a) self, (b) any descendant that would create a cycle. Disabled filters DO appear here so you can re-enable them inside a group, but show as "(disabled)" suffix.
- "**Used by**: <group names>" line shown directly under the name field (not buried in the footer) — high-salience location matches the moments the user is most likely to delete or rename. Block delete with a popover that surfaces the same list inline.
- Saving validates: cycle check, empty-name check, orphan check.
- The label "all of these"/"any of these" replaces AND/OR — no boolean jargon.

#### Group editor — "exclude + any" warning (UX councilor edit)

A `match: 'any'` group whose members are ALL `polarity: 'exclude'` atoms passes for nearly every conversation, because most items don't match at least one of the excludes. That's almost never what users want. Detect the combination at edit time and render an inline warning under the Match radio:

> ⚠ This combination usually matches almost everything. Consider switching to **Match all of these**, or change some members to **Include**.

Optional: a one-line "Logic summary" rendered under the Group settings showing the boolean expression in plain English ("Keeps items that match: NOT *invoice* OR NOT *standup*"). Defer to v2 if it's noisy; the warning alone catches the actual footgun.

#### Atom editor — name prefill from first pattern (user-requested edit)

When the user types into the Patterns textarea AND hasn't manually edited the Name input yet, auto-fill Name from the first non-empty pattern with glob/regex meta-chars (`*`, `?`, `[…]`, `^`, `$`, leading/trailing whitespace, escapes) stripped. Constraints:

- **Sentinel is client-only ephemeral state** (a `useRef<boolean> | useState<boolean>` in the modal). Do NOT persist `userEditedName` to the schema; once the filter is saved, the name is just a string.
- **Resume on clear**: if the user clears the Name input back to empty, auto-fill resumes (until they save). Set `userEditedName=false` when the input is empty.
- **Don't auto-update while the Name input is focused.** Users typing into Name shouldn't see it shift under their cursor.
- **Debounce** the auto-fill effect ~300ms after the Patterns textarea changes — avoids visible jumpiness.
- **Multiple patterns**: prefer the first non-empty line after stripping metachars. If the result is blank or under 3 alphanumeric characters (e.g. just `*` or `[a-z]`), fall through to the next line; if all lines fail, fall back to placeholder `New filter`.
- **Placeholder text** when name is empty: `Name (auto-filled from first pattern)`.

### Migration banner

On first mount after the new code lands, if the prefs blob has the old shape (`savedFilters[]` + `activeFilterIds[]`), run the migration synchronously **and** render a dismissible banner above the conversation list:

> "Filters are now composable. Your previously-pinned filters are grouped under **Default (migrated)** — your active filter. Click _Manage filters_ to review. [Dismiss]"

No interrupting toast, no mandatory click-through. Banner state persisted under `filters.migrationBannerDismissed` so it doesn't re-appear after the next reload.

## Migration

On first load with the new code, when reading prefs:

1. If `data.filters` (the new shape) is present, use it. Done.
2. Otherwise, look for legacy `data.savedFilters` + `data.activeFilterIds`:
   - For each legacy filter, build an `AtomFilter` with the same id/name/patterns/polarity/mode. Set `enabled = true`. Drop `pinned`, `target` (already 'title').
   - Build a `GroupFilter` with id `default-migrated`, name `"Default (migrated)"`, match `'all'`, childIds = the IDs of legacy filters where `pinned === true`. Set `enabled = true`.
   - If at least one filter was pinned, set `activeId = 'default-migrated'`. Else `activeId = null`.
3. PATCH the new shape AND explicitly null the legacy keys in the same payload: `{filters: {...}, savedFilters: null, activeFilterIds: null}`. The backend's per-key overwrite leaves untouched any key not present in the payload, so omitting them is NOT enough — we must overwrite with `null` to clear them.

The migration is idempotent: running twice produces the same result. We keep a sentinel flag (`filters._migratedV1: true`) inside the new blob so we never overwrite the user's edited state by re-running.

## Risks

- **Migration data loss.** Mitigation: the legacy keys stay in the prefs blob until the migration sentinel flips. If something goes wrong, the user can revert to a previous build and the legacy state is intact.
- **Group cycles via direct file edit.** Mitigation: visited-set evaluation already handles it; runtime never crashes.
- **UI complexity creep.** Mitigation: v1 caps at single-active and one level of nesting at a time (groups can reference groups; the editor only renders one level deep at a time, but the evaluator recurses arbitrarily). v2 can add a tree visualizer if users ask for it.
- **The user values least surprise.** Drop AND/OR jargon in UI. Migration banner is dismissible, not modal. Default state is "All conversations" — no filter applied — until the user picks one.

## Out of scope

- Drag-and-drop tree builder.
- Multi-active filters layered into a session-only ad-hoc compound.
- Filtering on fields other than `name` (project_path, summary, etc.). The atom's `target: 'title'` field stays in the schema for future use.
- Saving session-only "scratchpad" filters separate from the persistent named filters.
