## Defect (fix this)

**`claude-explorer --version` prints `0.1.0` regardless of the installed version.**

- Location: `cli/main.py:27`

  ```python
  @click.group()
  @click.version_option(version="0.1.0")      # <-- frozen literal
  def main() -> None:
  ```

- It's a hardcoded literal, so it has **never** matched the real version and never will. On the `1.0.4` wheel it prints `main, version 0.1.0` (note: also reports the program name as `main`, Click's default, because `prog_name` isn't set).
- Impact: a "Production/Stable"-classified public package that misreports its own version to `--version`. Confusing for users filing issues ("which version are you on?") and just wrong for V1 polish.

### Proposed fix (validated locally, then reverted)

Let Click read the version from installed package metadata so it can never drift from `pyproject.toml`:

```python
@click.group()
@click.version_option(package_name="claude-explorer", prog_name="claude-explorer")
def main() -> None:
```

`click.version_option(package_name=...)` resolves the version via `importlib.metadata`. After the change, `claude-explorer --version` prints:

```
claude-explorer, version 1.0.4
```

### Regression test (TDD — I confirmed it goes red against the current literal, green after the fix)

Drop in `fetcher/tests/test_cli_version.py` (matches where the other `cli/main.py` CLI tests live, e.g. `test_cli_fetch_wiring.py`):

```python
"""Regression test: ``claude-explorer --version`` must report the real
installed package version, not a frozen literal.

Pre-fix shipping bug (found 2026-06-04 while smoke-testing the published
1.0.4 wheel from PyPI): ``cli/main.py`` hardcoded
``@click.version_option(version="0.1.0")``, so ``claude-explorer
--version`` printed ``0.1.0`` on *every* release. The fix lets Click read
the version from installed package metadata (``package_name=``), which
can never drift from ``pyproject.toml``.
"""

from __future__ import annotations

from importlib.metadata import version

from click.testing import CliRunner

from cli.main import main


def test_version_flag_reports_installed_package_version() -> None:
    expected = version("claude-explorer")
    result = CliRunner().invoke(main, ["--version"])
    assert result.exit_code == 0
    assert expected in result.output


def test_version_flag_is_not_the_frozen_literal() -> None:
    # The shipping bug was a hardcoded "0.1.0" that never tracked the
    # package. Guard against that literal returning while the real version
    # differs.
    pkg = version("claude-explorer")
    result = CliRunner().invoke(main, ["--version"])
    assert result.exit_code == 0
    if pkg != "0.1.0":
        assert "0.1.0" not in result.output
```

- **Green** (after the one-line fix): both pass; `uv run claude-explorer --version` → `claude-explorer, version 1.0.4`.

Run: `uv run pytest fetcher/tests/test_cli_version.py -n0 --tb=short`. This file lands under `fetcher/tests`, which is already in `testpaths` and the CI/release gate, so it's covered automatically.

## Cosmetic note (NOT a bug — your call)

The MCP server's `serverInfo.version` reports **`3.4.0`**, name `"Claude Session Explorer"`. That's **FastMCP's own library version**, not ours — `mcp_server/server.py:39` calls `FastMCP("Claude Session Explorer", instructions=...)` with **no** `version=` argument, so FastMCP falls back to reporting itself. Harmless and standard framework behavior. If you want `serverInfo.version` to reflect the package, pass `version=importlib.metadata.version("claude-explorer")` to the `FastMCP(...)` constructor. Low priority; leave it unless you want the polish.
