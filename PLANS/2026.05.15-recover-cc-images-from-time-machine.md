# Recover Claude Code image-cache files from Time Machine

## Goal

Restore the screenshot attachments Claude Code deleted out of
`~/.claude/image-cache/<sess>/<N>.png` by walking every available Time
Machine backup and copying the most-recent surviving version of each
file. One-shot recovery utility; same shape as the
`/tmp/recover-claude-projects.py` script we ran for `~/.claude/projects/*.jsonl`
(claude-sessions ID `482c65de-d1b9-4fcc-83e8-d2d2e1390cfe`), retargeted at the
image cache instead of session transcripts.

## Background

- Claude Code stores screenshot attachments at
  `~/.claude/image-cache/<sess>/<N>.png`. Files are numbered per
  session, e.g. `~/.claude/image-cache/4f3e.../3.png`.
- Claude Code's startup runs a cleanup sweep that deletes files in
  `~/.claude/image-cache/` (and `~/.claude/projects/`, `~/.claude/tasks/`,
  `~/.claude/shell-snapshots/`, `~/.claude/backups/`) older than
  `cleanupPeriodDays` in `~/.claude/settings.json`. Default is 30 days,
  minimum is 1.
- Going forward we bump `cleanupPeriodDays` (or set it to e.g.
  `36500`) in `~/.claude/settings.json`, AND we rely on the existing
  `claude-explorer` CC image watcher (`backend/cc_image_watcher.py`)
  to mirror new images into `~/.claude-explorer/cc-images/` before
  Claude Code rotates them. This plan is for the one-time hole we
  already have in the live cache.
- The Time-Machine destination on this machine is the
  `M3 Max Backups 5T` volume; backups are mounted under
  `/Volumes/.timemachine/<volume-UUID>/<machine-UUID>/<snapshot>/...`.
- macOS SIP requires **Full Disk Access** for the terminal/process
  reading TM snapshots. Without it, the script gets
  `Operation not permitted` even with `sudo`. The user must add
  Terminal (or iTerm2, or Ghostty, whichever shell hosts the script)
  to *System Settings → Privacy & Security → Full Disk Access*
  before running.

## Critical files and paths

- Source: every snapshot under `tmutil listbackups`, specifically
  `<snapshot>/Users/rpeck/.claude/image-cache/<sess>/<N>.png`.
- Target (live): `/Users/rpeck/.claude/image-cache/<sess>/<N>.png`.
- Target (explorer mirror, optional second pass):
  `/Users/rpeck/.claude-explorer/cc-images/<sess>/<sess>--<N>.<sha8>.<ext>`
  (this is the format the explorer's watcher already writes).
- Script: persist to `scripts/recover_cc_images.py` in this repo so
  it survives `/tmp` cleanups and can be re-run later. (The previous
  `recover-claude-projects.py` lived in `/tmp/` and is gone.)
- Audit log: `~/.claude-explorer/recover-cc-images.log` — one line
  per restored file with timestamp, source snapshot path, target
  path, file size, sha8.
- Plan reference: prior session
  `482c65de-d1b9-4fcc-83e8-d2d2e1390cfe`, original script
  `/tmp/recover-claude-projects.py` (likely gone; can recover from
  claude-sessions MCP if needed for reference).

## What the script does

### Phase 1 — enumerate

1. Run `tmutil listbackups`. Get the list of snapshot paths. Sort
   ascending by timestamp (oldest first) so a later overwrite wins.
2. For each snapshot path, build
   `<snapshot>/Users/rpeck/.claude/image-cache/`. Skip if it does not
   exist in that snapshot (some early snapshots predate Claude Code
   being installed).

### Phase 2 — index

3. Walk every present snapshot's image-cache tree. For each file,
   record `(sess, basename, snapshot-path, mtime, size)`. The
   `(sess, basename)` pair is the recovery key.
4. Reduce to one row per `(sess, basename)`: pick the snapshot with
   the most recent mtime, on the theory that the freshest version is
   most likely the version we last saw in-app.

### Phase 3 — diff against live cache

5. Walk the current live cache at `~/.claude/image-cache/`. Anything
   that already exists locally is skipped (don't clobber, don't
   re-stat the user's working state).
6. Also walk `~/.claude-explorer/cc-images/`. Anything the explorer
   already mirrored (key by basename suffix) is also skipped — the
   explorer's hash-suffixed copy is canonical and we shouldn't add a
   redundant restore.

### Phase 4 — restore (default: dry run)

7. For each key not in either live location, `cp` from the chosen
   snapshot to the live image cache, preserving mtime
   (`shutil.copy2`).
8. Mirror into the explorer's CC image dir as well, using the same
   naming convention the watcher uses (`<sess>--<N>.<sha8>.<ext>`),
   so the recovered files become first-class in the archive even if
   Claude Code rotates them again before the user opens those
   sessions in the UI.
9. Append every action to the audit log.

The default is **dry-run**: print every move and the totals, change
nothing. `--apply` commits the moves.

## CLI shape

```bash
# Dry run (default): list every file that would be restored.
uv run python scripts/recover_cc_images.py

# Commit the restore.
uv run python scripts/recover_cc_images.py --apply

# Only mirror into the explorer's cc-images dir (don't touch the
# live ~/.claude/image-cache, in case the user prefers to keep that
# pristine and let the explorer be the only archive).
uv run python scripts/recover_cc_images.py --apply --explorer-only

# Restrict to one session UUID.
uv run python scripts/recover_cc_images.py --apply --session 4f3e1...

# Use a non-default backups source (testing).
uv run python scripts/recover_cc_images.py --backups-root /Volumes/...
```

## Implementation notes

- Single Python file, stdlib-only (`pathlib`, `subprocess`,
  `shutil`, `hashlib`, `argparse`). Same constraint as the prior
  projects-recovery script: no `uv` deps needed at recovery time.
- For each backup, `tmutil listbackups` is the authoritative
  enumerator. Parsing `/Volumes/.timemachine/...` directly would
  also work but is brittle across macOS versions.
- The `<sha8>` suffix in the explorer's cc-image filename is the
  first 8 hex chars of `sha256(file bytes)`; replicate that
  exactly (`hashlib.sha256(path.read_bytes()).hexdigest()[:8]`) so
  the mirrored files look identical to what the live watcher would
  have produced.
- Be paranoid about same-basename-different-bytes collisions: two
  snapshots can each have `<sess>/3.png` with different content (CC
  reuses the integer per session as it writes new screenshots). The
  recovery picks the newest-mtime version. If a user wants every
  historical version we ever had, add a `--all-versions` flag that
  writes each one under `<sess>/<N>.<snapshot-date>.png`.
- Restore mtime so the recovered files don't immediately look "new"
  to the explorer's watcher; the watcher's idempotency keys on
  basename + sha, not mtime, so this is cosmetic but tidy.

## Risks and rollback

- **Full Disk Access not granted**: script bails on first
  `PermissionError` reading a snapshot, prints the System Settings
  path to grant access, and exits non-zero. No partial state.
- **Wrong file picked among versions**: dry-run mode shows the
  chosen snapshot date next to every key; user can spot a
  surprising "freshest" pick (e.g. a `3.png` whose newest version
  is months older than the session's last activity) and re-run with
  `--all-versions`.
- **Live cache already has the file**: skipped by construction
  (Phase 3 step 5). The user can `rm` a file from live cache and
  re-run if they want to overwrite from backup.
- **Disk space**: image cache is small (a few MB to maybe a couple
  GB across all snapshots' contributions); script logs total bytes
  in dry-run.

## Verification

After `--apply`:

1. `find ~/.claude/image-cache -type f -name '*.png' | wc -l` is
   bigger than before by exactly the dry-run's "would restore" count.
2. `find ~/.claude-explorer/cc-images -type f | wc -l` is bigger
   by the same count (if we used the default both-target mode).
3. `claude-explorer serve`, open any session that had missing
   images, confirm thumbnails now render in the UI.
4. Audit log lists every restore.

## Open decisions for the user before running

- Default destination(s): live cache only, explorer cache only, or
  both? The plan above proposes "both" because it survives a
  subsequent CC cleanup pass. If you want only one, say which.
- Bump `cleanupPeriodDays` in `~/.claude/settings.json` to a large
  value (e.g. `36500`) before running, so the next CC startup
  doesn't undo the restore.
