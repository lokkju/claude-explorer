# Plan: Speed up full-text search — alternatives + recommendation

## Context

Full-text search is currently 0.8–2.3 seconds per query on Ray's
corpus. Slow enough to feel laggy when iterating queries.

**Empirically measured today:**
- Corpus: **1,222 files / ~1.5 GB**
  - 4 Desktop conversation JSONs (16 MB)
  - 1,218 Claude Code session JSONLs (1.5 GB)
- Latency:
  - "cron" (~few results) — **2.3s**
  - "python" (~4.5 MB of results) — **0.78s**
  - non-matching query — **0.94s**
- Frontend: 200ms debounce + 60s TanStack staleTime (already optimal)

**Architecture today** (from `backend/routers/search.py:14-57` →
`backend/search.py:134-295`):
- `/api/search` calls `ConversationStore().search_conversations()` on every
  request.
- Walks **every** JSON + JSONL on disk via `store.get_all_conversations_raw()`.
- For each conversation: iterates **every** message, calls
  `_extract_searchable_text()` (flattens text + tool_use + tool_result
  + thinking), runs `pattern.finditer(text)` (case-insensitive regex).
- Caching that exists:
  - `FileCache` (`backend/cache.py:26-152`): mtime-keyed conversation
    parse cache. Saves the JSON load on warm cache.
  - Per-message memo: `msg["__search_text__"]` survives because the
    conversation dict instance is cached.
- Caching that **doesn't** exist:
  - **No inverted index.** Every query does a full linear scan.
  - **No query-result cache.** Identical queries re-execute the loop.

**Where the time actually goes:**
- Cold cache: I/O + JSON parse dominates (~1–2s for 1,218 JSONL).
- Warm cache: regex scan over ~10⁵ messages dominates.
- Broad queries: result-payload serialization adds 100–500ms.

## Alternatives (with plusses and minuses)

### Option A — SQLite FTS5 (RECOMMENDED)

Use Python's stdlib `sqlite3` with the FTS5 virtual-table module.
A single-file index at `~/.claude-explorer/search-index.sqlite`,
populated incrementally as conversations land and CC JSONL files
grow. Backend search endpoint rewrites to a SQL query against the
index.

**Plusses:**
- **Stdlib.** Zero new runtime dependencies. FTS5 is built into
  `sqlite3` in every modern macOS/Linux Python build (verify once at
  startup, fall back if missing).
- **Fast.** Typical FTS5 query on this corpus: **5–50 ms** (40–500x
  speedup over today). BM25 ranking out of the box; built-in
  `snippet()` and `highlight()` functions.
- **Persistent.** Index survives restarts; no warm-up cost on
  `claude-explorer serve`. Cold-start latency stays sub-10s like
  today.
- **Incremental.** Insert/update on:
  - conversation fetch (already a write path; just add an INSERT)
  - CC JSONL detection (extend the existing CC image watcher to
    also detect new lines and reindex affected sessions)
- **Composable filters work at SQL layer:** WHERE clauses for
  source / project / bookmarks / single-conversation scope.
  Trivial pagination via LIMIT/OFFSET.
- **Battle-tested.** SQLite FTS5 is what GitHub uses for code
  search fallback, what 1Password uses for vault search, what
  countless macOS desktop apps use.

**Minuses:**
- **Initial index build is slow** (one-time 30–120s on a 1.5 GB
  corpus). Mitigate by running it as a background task during
  lifespan, just like the auto-warm CC pass we already have. Search
  falls back to the current linear scan until the index is ready.
- **Schema migration risk.** Adding a new searchable field later
  (e.g., conversation summary) requires a re-index. Version the
  schema; provide a one-line CLI to rebuild (`claude-explorer
  reindex-search`).
- **Two writers can corrupt the file.** SQLite is fine with a
  single writer + many readers; we control the writer (backend
  process), so this is not a real risk for our deployment.
- **Index drift if a file changes outside our writer.** Mitigate
  with mtime tracking — store the mtime per indexed file, re-scan
  on mismatch.

**Effort:** ~1.5 days. New `backend/search_index.py` (~200 lines),
~50-line patch to `search.py` to query the index, ~30-line lifespan
hook to build/maintain it.

### Option B — Whoosh (pure-Python FTS library)

Drop in [Whoosh](https://github.com/whoosh-community/whoosh) — a
pure-Python search library with an inverted index, BM25 ranking,
and faceting.

**Plusses:**
- **Pure Python**, no C deps; pip install only.
- **Higher-level API** than raw SQL; easier to add features like
  faceting, suggestions, fuzzy match later.
- Familiar Lucene-style query language (`title:foo AND body:bar`).

**Minuses:**
- **Slower than SQLite FTS5** in benchmarks (typically 2–5× slower
  for the same queries on similar corpora).
- **Larger on-disk index** (Whoosh stores tokens + positions; SQLite
  FTS5 is denser).
- **Maintenance question mark.** Last upstream release was 2022; it
  works but isn't actively developed. Forks exist
  (`Whoosh-Reloaded`) but adoption is low.
- **One more dep** to lock and ship.

**Effort:** ~1 day. Slightly less code than Option A because Whoosh
hides more, but the abstraction also gives you less control when
something goes wrong.

---

### Option C — Tantivy (Rust-backed, via `tantivy-py`)

Use [tantivy-py](https://github.com/quickwit-oss/tantivy-py),
Python bindings to Tantivy (a Lucene-class search engine in Rust).

**Plusses:**
- **Fastest of the three indexed options.** Sub-millisecond for
  most queries on this corpus.
- **Modern, actively developed**, used in production by Quickwit.
- Rich query language; supports phrase, fuzzy, and field-weighted
  ranking.

**Minuses:**
- **C/Rust dependency.** Wheels exist on PyPI for macOS/Linux x86_64
  + arm64 + Linux musl, but we'd need to verify install pain on the
  Python versions we support.
- **Overkill for 1.5 GB.** SQLite FTS5 is already faster than the
  user can perceive on this corpus.
- **Larger on-disk footprint** than SQLite FTS5 for small corpora
  (Lucene-style segmenting overhead).
- One more dep that could break on a Python upgrade.

**Effort:** ~1.5 days, similar to Option A, but with extra time
debugging install issues for whatever environment the user lands in.

---

### Option D — ripgrep subprocess wrapper

Shell out to `rg --json` for the actual scanning. Keep current code
shape, just replace the inner regex loop.

**Plusses:**
- **Trivial to implement** (~30 lines).
- ripgrep is **10–100× faster than Python regex** on raw bytes
  scans.
- No index to maintain — always fresh by definition.

**Minuses:**
- **Requires `rg` binary** on the user's PATH. We'd need to bundle
  it or fail loudly. (`brew install ripgrep` is a one-liner but
  it's another dep the user has to remember.)
- **Loses structured search.** `rg` searches raw bytes; we'd be
  matching against JSON serialization (so queries for `python`
  would also match `"model": "python-..."` keys, etc.). To fix
  this we'd have to extract text first, which removes the speed
  win.
- **No ranking.** ripgrep just lists matches in file order.
- **Doesn't fix the result-payload size issue.** Broad queries
  still return everything.
- **Doesn't help cold-start.** ripgrep still has to read every byte
  on every query (no index).

**Effort:** ~0.5 day, but you trade away most of the architectural
wins.

---

### Option E — Optimize current Python only (no new tech)

Keep the linear scan but make it faster:
- Build a **tiny in-memory inverted index** at backend startup
  (token → list of (conv_uuid, message_uuid)). Rebuild on file
  changes via the existing watcher. No persistence.
- Add **early termination** (stop scanning a conversation after
  first match if `context_size=snippet`).
- Parallelize the file walk with `multiprocessing`.

**Plusses:**
- **No new dep.** Pure Python.
- **No persistence to manage.** Rebuild on every restart.

**Minuses:**
- **Slow rebuild on every backend restart.** ~30s of CPU at
  startup. Lifespan would need to wait or have a fallback.
- **RAM cost.** ~50–100 MB resident for a 1.5 GB corpus index.
- **Worse algorithmic ceiling.** SQLite FTS5 is faster AND smaller
  AND persistent; we'd be hand-rolling a worse version.
- **Code-maintenance burden.** Hand-rolled tokenization,
  case-folding, accent stripping; SQLite FTS5 has all this for
  free.

**Effort:** ~1 day, but keeps technical debt forever.

## Recommendation: **Option A — SQLite FTS5**

Best balance of performance, simplicity, and zero-dep footprint.
The 5–50 ms latency is "imperceptible" in UX terms (well under the
human-perception threshold of ~100 ms). It uses the stdlib so we
don't bloat the install. The schema is simple enough that future
field changes are tractable.

### Sketch of the work

**1. Schema** (`backend/search_index.py` — new):
```sql
CREATE VIRTUAL TABLE messages USING fts5(
    conv_uuid UNINDEXED,
    message_uuid UNINDEXED,
    sender UNINDEXED,
    created_at UNINDEXED,
    title,           -- conversation title repeated per message (cheap, helps title boosting)
    body,            -- flattened searchable text (text + tool_use + tool_result + thinking)
    tokenize = "porter unicode61 remove_diacritics 1"
);

CREATE TABLE indexed_files (
    path TEXT PRIMARY KEY,
    mtime REAL NOT NULL,
    indexed_at INTEGER NOT NULL
);

CREATE TABLE schema_version (version INTEGER NOT NULL);
```

**2. Index lifecycle**
- **Initial build:** background task in FastAPI lifespan (alongside
  the existing CC auto-warm). Walks every JSON/JSONL, inserts rows.
  Logs progress every 100 files. Idempotent.
- **Incremental updates:**
  - On `claude-explorer fetch`: re-index the conversations that
    just landed (already a hot write path).
  - On CC JSONL change: extend the existing watcher
    (`backend/cc_image_watcher.py`) to also tail JSONL files and
    upsert affected rows. mtime check short-circuits no-op cases.
- **Drift safety:** on every search query, do a fast O(n_files)
  mtime check against `indexed_files`; mismatches trigger a
  background re-index of just those files.

**3. Query path** (`backend/search.py:search_conversations`):
- Translate user query into FTS5 MATCH expression. For now: simple
  AND-of-terms with prefix wildcard on the last token (matches
  search-as-you-type behavior). Phrase queries get quoted.
- SQL:
  ```sql
  SELECT conv_uuid, message_uuid, sender, created_at,
         snippet(messages, 5, '<mark>', '</mark>', '…', 16) AS snippet,
         bm25(messages) AS rank
  FROM messages
  WHERE messages MATCH ?
    AND (? IS NULL OR conv_uuid = ?)        -- conversation_uuid scope
    AND (? IS NULL OR conv_uuid IN (...))   -- bookmarks scope
  ORDER BY rank LIMIT 200;
  ```
- Group by `conv_uuid` server-side, build the existing
  `SearchResult` shape, return.
- **Fallback:** if the index isn't ready yet (initial build still
  running), fall back to the current linear-scan code path. So
  search never goes "down".

**4. Migration / rebuild**
- Schema version check at startup; rebuild from scratch on version
  bump.
- New CLI: `claude-explorer reindex-search` (manual override; the
  background path covers normal use).

### Critical files

**New:**
- `backend/search_index.py` — schema, build, upsert, query
  helpers, schema-version migration.
- `backend/tests/test_search_index.py` — unit tests for upsert,
  drift detection, fallback, query translation.

**Modified:**
- `backend/search.py` — `search_conversations()` queries the index
  when available, falls back to linear scan when not.
- `backend/main.py` — lifespan spawns
  `build_or_update_index_async()` alongside the existing CC warm
  + watcher.
- `backend/cc_image_watcher.py` — extend `scan_once()` to also
  upsert any JSONL whose mtime changed since last index pass.
- `fetcher/cli.py` — new `reindex-search` command (manual override
  only).

**Reused:**
- `_extract_searchable_text()` from `backend/search.py` — same
  flattening logic for the index body column.
- `FileCache` pattern — the index acts like a smarter, persistent
  version of FileCache for search.
- Watcher loop pattern — same lifespan integration as CC watcher.

### Verification

1. **Latency goal:** all 3 measured queries above ("cron",
   "python", non-matching) drop to <50 ms after first warm-up.
   Add a benchmark test that fails if any query exceeds 200 ms on
   Ray's corpus (CI can skip; local-only).
2. **Correctness:** the existing
   `backend/tests/test_search*.py` suite passes unchanged
   (results must match the linear-scan code byte-for-byte for the
   same inputs).
3. **Drift:** modify a conversation JSON externally (touch
   mtime), confirm next query reflects the change within one
   watcher cycle.
4. **Cold-start:** delete `~/.claude-explorer/search-index.sqlite`,
   restart backend, confirm:
   - Search works immediately (linear-scan fallback).
   - Background task logs progress.
   - Within ~60s, latency drops to <50 ms.
5. **Schema bump:** change the schema version constant, restart,
   confirm full rebuild.

### Risks

- **FTS5 not in user's sqlite3 build.** Detect at startup with
  `sqlite3.connect(":memory:").execute("CREATE VIRTUAL TABLE x USING fts5(c)")`;
  if it fails, log + skip indexing + use linear scan. CLAUDE.md
  documents the install (macOS Homebrew Python ships FTS5 by
  default).
- **JSONL "tail" detection.** CC sessions are append-only files
  that grow as the user works in CC. The watcher extension needs
  to track byte-offset per file and re-index incrementally, not
  re-parse the whole file. Worst case: re-parse on mtime change
  (slow but correct).
- **Index size.** ~50 MB for the current corpus is fine. If a user
  has 100× the data (~150 GB), the index might be ~5 GB — still
  manageable, but document the rebuild cost.
- **Race between initial build and query.** Mitigate with a
  feature flag / "index ready?" check; queries fall back to linear
  scan until ready.
