# Feature request: branch-aware querying in the `claude-sessions` MCP server

**Status:** DEFERRED — revisit after the Medium series (Parts 1–7) ships. Captured now so the motivating use case isn't lost.
**Raised:** 2026-06-05, while reviewing Part 3 (the MCP-server article). The article's active-branch caveat prompted the question "so how do I work with branches through the server?" — and the honest answer today is "you can't."
**Owner when picked up:** the `a70251a5` "Claude Explorer functionality" build session (code), with article follow-up in the writing session.

---

## 1. The motivating ask

A reader (and the author) will naturally want to interrogate the *shape* of a session, not just its live path. The concrete example that motivated this:

> *"Find me the branch points where I had to clarify my prompt to correct a misunderstanding by Claude."*

The intent: surface the moments of friction — where the user edited/resent a prompt because Claude went the wrong way — as a first-class signal. Those forks are a high-value retrospective target (they mark exactly where instructions were ambiguous or the model misread intent), and they're the raw material for the kind of CLAUDE.md tuning loop Part 3 already describes.

## 2. Why the server can't do this today (verified 2026-06-05)

Every MCP tool reads a **flattened active branch**, and no tool exposes branch structure:

- `backend/store.py:get_conversation` (~L856–862) runs `resolve_active_branch(chat_messages, leaf_uuid)` (`backend/store.py:211`) and sets `conversation.messages = [_parse_message(m) for m in branch]`. The walk is leaf→root, so **off-branch (abandoned) messages are discarded before any MCP tool sees the data.** You can't reach them by position or even by UUID through the tools — they're simply not in `conversation.messages`.
- `mcp_server/server.py:get_session_outline` (L520) returns `position / message_uuid / sender / summary / char_count / tool_count / timestamp` — **no branch info, no `has_branches`.**
- `mcp_server/server.py:list_sessions` (L374) returns no `has_branches` either.
- `get_messages` (L557) and `export_session` (L622) both index into the active-branch `conversation.messages` only.
- The `has_branches` boolean exists on `ConversationDetail` but is **never returned to the client by any tool** (its only use in the MCP layer is pass-through when slicing for export, `mcp_server/server.py:661`). For **Claude Code it is hard-coded `False`** (`backend/store.py:~847`: "CC has no true edit-branches"; post-compaction duplicate UUIDs would otherwise produce false positives).

**Net:** the MCP surface exposes zero branch information. Writing the example up as a working capability would be capability theater — which Part 3 §6 explicitly warns against. The article keeps only its honest one-line caveat ("the outline follows the session's active branch … this server picks the live path rather than trying to hand you a tree"). No article change was made for this; that caveat stays until this feature lands.

## 3. The capability already exists in the backend — this is wiring, not new core logic

The hard part (parsing the tree, handling placeholder roots, circular-ref safety, post-compact CC quirks) is already solved and shipping in the web app:

- `backend/store.py:build_message_tree` (L243) — full parent→children tree, iterative BFS, circular-ref safe, handles the synthetic placeholder root.
- `backend/store.py:get_conversation_tree` (L1024) — returns `ConversationTree { root_messages: build_message_tree(...), active_branch: resolve_active_branch(...) }`.
- `ConversationTree` / `MessageNode` models in `backend/models.py`.
- REST route `GET /api/conversations/{uuid}/tree` (`backend/routers/conversations.py:125`, summary "Get the full message tree for a conversation (all branches)"), with a size guard ("too many messages for tree visualization").
- Frontend "View branches" UI: `frontend/src/components/branch/TreeViewModal.tsx`, `TreeView.tsx`, `BranchIndicator.tsx` (button gated on `has_branches`).

So the MCP work is mostly: expose a slice of this through a read-only tool, in a token-frugal shape consistent with the outline-first philosophy.

## 4. Important scoping nuance: Desktop vs Claude Code

- **Claude Desktop** conversations have true **edit-branches** (edit/resend a prompt → fork). The motivating example only literally applies here. `has_branches` is real for this source.
- **Claude Code** sessions have **no true edit-branches** (hard-coded `has_branches=False`). When you "correct a misunderstanding" in CC you just send the next message — a normal sequential turn, already fully visible in the outline. The branch tool would be a **no-op / empty result** for CC sessions, and that's correct behavior, not a bug.
- **Intent vs mechanism:** for CC (where the long sessions live), the *intent* behind the example — "find where I corrected Claude" — is **already answerable today** as a content search over the active branch for correction-shaped turns ("no", "that's wrong", "I actually meant…"). Part 3's tuning-loop section already demonstrates exactly this ("zoomed in on the stretches where I'd corrected it"). Any docs for the new feature should draw this line clearly so users don't expect branch magic where a content search is the right tool.

## 5. Design options (decide when picked up)

Token-frugality is the through-line of the whole server; whatever we add must not tempt a client to pull a giant tree into context.

1. **`get_branch_points(session_id)`** — return only the forks: for each message with >1 child, the parent's position/UUID, a one-line gist, each child's first-message gist, and which child is on the active branch. Lightweight, directly answers the motivating ask, and reads like the outline (skim-first). **Leading candidate.**
2. **`get_session_tree(session_id)`** — mirror `get_conversation_tree`: full `root_messages` tree + `active_branch`. Most faithful, but heaviest; needs the same size guard as the REST route and risks the context-blowup the outline philosophy avoids. Possibly gate behind an explicit `include_full_tree` arg.
3. **Extend `get_session_outline`** with an optional `include_branches: bool = False` that annotates each row with `has_alternate_children` / sibling pointers. Keeps one entry point but complicates the outline's clean shape.

Cross-cutting:
- Add `has_branches` to `list_sessions` and/or `get_session_outline` output so a client can cheaply decide whether branch tools are even worth calling (skip entirely for CC / unbranched Desktop sessions).
- Let `get_messages` / `export_session` address off-branch messages — either via a `leaf_uuid` / branch selector arg (the store's `get_conversation(uuid, leaf_override=...)` already supports a leaf override, `backend/store.py:917`) or by allowing UUIDs that resolve outside the active branch. Without this, you can *find* a branch point but not *read* the road not taken.

## 6. Open questions for the revisit

- Which tool shape (§5.1 vs 5.2 vs 5.3)? Default lean: **`get_branch_points`** + add `has_branches` to existing outputs + a `leaf_override` path for reading off-branch.
- Token budget: branch tools add to the fixed per-conversation tool-definition tax Part 3 quantifies (4,681 chars / ~1,200–1,600 tokens for five tools). Adding a sixth/seventh tool raises that. Worth measuring and possibly gating behind a config/opt-in.
- Reuse the REST size-guard threshold for any full-tree path.
- CC behavior: return an empty/`has_branches=false` result cleanly; document that this is expected, not an error.

## 7. Acceptance sketch (for when this is built)

- A branched **Desktop** fixture: `get_branch_points` returns each fork with parent position + per-child gists + active-branch marker.
- An **unbranched / CC** fixture: returns empty, `has_branches=false`, no error.
- Reading an **off-branch** message succeeds via the chosen mechanism (leaf override or UUID).
- `has_branches` appears in `list_sessions` / outline output and matches the web app's value for the same session.
- Token-cost delta of the new tool definition(s) measured and recorded (mirror Part 3's 4,681-char measurement).
- Article follow-up: turn the motivating example into a real worked use case in Part 3 (or a short follow-up), replacing the current "can't hand you a tree" caveat.
