## §1 — Header layout reflow (issue #3)

The conversation header had a row of buttons (Re-download · Markdown ·
PDF · Tools · Expand/Collapse · Copy as Markdown · View branches) that
on narrow viewports occluded the title / UUID / model / project line.

**Steps**

1. Open any conversation.
2. Resize the browser window narrower (drag from the right edge until
   the viewport is roughly 900–1000 px).
3. The header should reflow to **two lines** — title on top, action row
   below. **None** of the metadata (UUID, model, project name, date)
   should be hidden behind the buttons.
4. Resize back wide. Title + actions on a single row again is fine.

**Pass:** title + every action button visible at every width down to
~600 px (mobile would normally show the drawer instead).

## §3 — Selection stability across Tools toggle (issue #2)

Previously: searching, clicking a result, then toggling Tools (show /
hide tool blocks) would change which message was selected — the
selection used a numeric index into the visible-message list, and the
list shrinks/grows when tool blocks come/go.

**Steps**

1. Open a conversation that has tool blocks in it (any Claude Code
   session with `Bash` / `Read` / `Grep` calls — lots of these).
2. **Cmd+K** → search panel. Type a query that matches a specific
   message, e.g. a string you remember from the conversation.
3. Click a result card → conversation pane scrolls to that message;
   the message gets the keyboard-selected ring.
4. **Without changing selection,** click the **Tools** button in the
   header to toggle tool blocks on (or off, whichever flips).
5. **The same message must remain selected** (same ring around the
   same `data-message-uuid`). The viewport should re-anchor too — the
   selected message stays in view.
6. Toggle Tools again → still same message.

**Pass:** selection follows the message UUID, not the visible-list
index, so it survives Tools, Expand/Collapse, and per-bubble
collapse/expand.

## §4 — Search performance (issue #0)

Cold first search after `restart.sh` is unavoidably slow (the backend
walks every JSON / JSONL file once to populate caches). Warm subsequent
searches should be noticeably faster than yesterday.

**Steps**

1. **Restart** with `./restart.sh`. Cold cache.
2. **Cmd+K** → search "the" → time the first response. Expected ~5–7 s
   on this machine. Article calls this out — first call must be slow.
3. Type a different query (e.g. "fixture"). Expected ~0.8–1 s. If it
   feels noticeably faster than yesterday's run, the `FileCache` +
   per-message memo are doing their job.
4. Run the bench harness:
   ```bash
   uv run python scripts/bench_perf.py --runs 10
   ```
   Search mean should be in the ~0.9–1.1 s range (down from ~1.1–1.2 s
   pre-fix). List endpoint should still be ~2.3 s on cold cache,
   sub-millisecond once warm (the `/api/config` Bug A fix from
   yesterday).

**Pass:** warm-cache search visibly snappier than yesterday. NOT yet at
the article's "~50 ms" target — that needs an inverted index and is
called out as out-of-scope. If the warm-cache number doesn't beat
~1.5 s, log it.

## §5 — Markdown bundle export (issue #4)

New: an export mode that produces a zip with `conversation.md` plus an
`images/` directory, with image refs rewritten to relative paths so the
zip is portable to colleagues.

**Steps**

1. Settings → scroll to **Markdown Export** section.
2. **Toggle "Bundle images into a zip"** ON.
3. Pick a **dialect**:
   - **CommonMark** (default; renders in GitHub / MacDown / most
     viewers).
   - **Obsidian** (uses `![[image.png]]` wikilink syntax).
4. Open a conversation that has images (a CC session with `[Image: …]`
   markers is the cleanest test).
5. Click the **Markdown** export button in the conversation header.
6. With "Bundle images" ON you should get a `.zip` file (NOT a `.md`).
   Save it.
7. **Unzip** the file. Confirm:
   - `conversation.md` at the root.
   - `images/` directory containing each referenced image (CC cached
     PNGs).
   - Image refs in `conversation.md` are **relative** paths
     (`images/<filename>`), not `/api/...` URLs.
8. Open `conversation.md` in MacDown / VS Code preview / Obsidian /
   GitHub Gist (drop in a temp Gist) — images should render inline in
   each. (Obsidian dialect needs the Obsidian app specifically; the
   wikilink format won't render in GitHub.)
9. Toggle "Bundle images" OFF. Export Markdown again. You should get a
   plain `.md` (no zip), image refs as `/api/...` URLs (the old
   behavior — only renders while the local backend is running).

**Caveat noted by the council:** Desktop `Message.files[]` images
(claude.ai-hosted, behind sessionKey) can't be bundled without an
authenticated proxy fetch. For those, the bundled `.md` carries a
footnote `_(Desktop image attachment not bundled: <name>)_` so the
recipient can see what's missing. CC `[Image: source: …]` images bundle
cleanly because they live on local disk.

**Pass:** zip structure correct, dialect choice respected, CC images
bundled and rendering in the recipient's Markdown viewer.
