# Feature request: a scheduled background Desktop-fetch safety net

**Status:** Spec only. Decided 2026-06-07 to design now, build after the Medium
series ships. No code in this pass.

**Origin:** While reviewing the Part 3 userdoc we wrote that the MCP server
"reads only what's already on disk, so it's only as complete as your archive,"
which raised the question: should `install-watcher` also keep that archive
current on its own, so the user can't silently lose data by forgetting to
Refresh? Answer below: yes, but only for one source, and the credential
lifecycle is the entire design problem.

## 1. The motivating ask

> "We really ought to make `install-watcher` save the conversations, too. We
> need something scheduled to refresh everything, for safety."

The user wants the always-on watcher to be a true safety net: a conversation
should land in the local archive without the user having to remember to click
Refresh.

## 2. Important scoping nuance: "everything" is really just Claude Desktop

The three sources are not equally exposed, because two of them already live on
disk and are already watched.

| Source | Where it lives | Already protected? |
|---|---|---|
| **Claude Code** | conversation JSONL on disk in `~/.claude/projects/*/*.jsonl`; images in `~/.claude/image-cache/` | **Yes.** Watcher drift-scans the JSONL; the image-cache mirror copies images out before Claude Code rotates them off disk (`backend/cc_watcher.py` surface A). |
| **Claude Cowork** | session files under the Cowork root (`audit.jsonl`, `outputs/`, `uploads/`) | **Yes, for content.** Watcher watches `_live_cowork_root()` for drift (`backend/cc_watcher.py:458`, `_try_start_cowork_observer`). Content is on disk, not in a rotating cache. *(Open: confirm Cowork has no separate rotation behavior — see §6.)* |
| **Claude Desktop** | **server-side** at Anthropic; on local disk **only if fetched** | **No.** The watcher never fetches. This is the only true gap. |

So the real feature is a **scheduled incremental Desktop fetch**, not "refresh
everything." The local sources already have their net.

**The data-loss scenario it closes:** a Desktop conversation deleted
server-side — by the user, or aged-out/removed by Anthropic — is gone unless a
prior fetch had snapshotted it locally. A scheduled fetch turns the local
archive into a durable, offline, searchable snapshot that survives server-side
deletion.

## 3. Why the watcher can't just do this today (verified 2026-06-07)

Fetch needs live credentials, and **re-auth is interactive**. This is the whole
blocker.

- The fetch path returns an auth-class error on `401` / `403` / `cf-mitigated`
  and maps it to `SESSION_EXPIRED_MESSAGE` (`backend/routers/fetch.py:87`,
  `:129`, `:213`).
- The only recovery is `fetcher.playwright_capture.capture_credentials`, which
  **opens a browser and waits up to ~5 minutes for the human to log in**
  (`LOGIN_TIMEOUT`, `backend/routers/fetch.py:78`; the `capture_waiting_login`
  heartbeat in the Refresh SSE stream).
- A headless launchd / systemd / Task Scheduler job **cannot complete an
  interactive login.** Popping a browser window out of a background daemon is
  also hostile UX.

**Therefore the naive design is actively harmful:** "fetch on a timer" works
until the session key expires, then silently 401s forever. That is *worse* than
no scheduled fetch, because the user believes they have a safety net while it
quietly does nothing. **Handling expiry gracefully is the feature**, not an
afterthought.

## 4. The capability already exists — this is wiring plus a lifecycle policy

- The watcher already runs a periodic **backstop poll** every
  `SCAN_INTERVAL_SEC` (default 600s, env
  `CLAUDE_EXPLORER_CC_WATCHER_INTERVAL_SEC`) — `backend/cc_watcher.py:135`,
  `:760`. `scan_once()` (`:182`) already does the image-cache pass and the
  search-index drift pass on that schedule. A creds-gated incremental fetch is a
  third isolated pass in the same loop.
- The incremental fetch logic already exists in
  `_fetch_phase_stream` / `refresh_pipeline_stream`
  (`backend/routers/fetch.py:633`, `:1074`) and the `claude-explorer fetch`
  CLI. The new code is the **gate and the expiry policy**, not the fetch itself.

## 5. Proposed design (decide details when picked up)

Core shape — a creds-gated, log-and-skip incremental fetch on the existing
backstop poll:

1. On each backstop tick, before fetching: load credentials. If absent or
   already known-expired, **log and skip** (do not attempt, do not capture).
2. If present, run an **incremental** fetch (skip already-saved conversations).
   Never `--full-refresh` on the schedule.
3. On an auth-class error (`401`/`403`/`cf-mitigated`), **log and skip** for
   this and all subsequent ticks until creds are refreshed. **Never launch
   `capture_credentials` from the background job.**
4. **Surface the expired state to the user** so the net announces when it needs
   a hand: reuse the existing `SESSION_EXPIRED` signal and the banner pattern
   (cf. `WatcherMissingBanner`) so the app shows "Background sync paused —
   credentials expired, hit Refresh to re-authenticate." The user's next
   interactive Refresh re-captures and the schedule resumes.

Design intent: the schedule keeps the archive current **while creds are valid**,
and **fails loud, not silent** when they aren't.

## 6. Open questions for the revisit

- **Cadence.** Is the 600s backstop the right interval for a network fetch, or
  should the Desktop pass run on a coarser sub-interval (e.g. every Nth tick, or
  hourly) to avoid hammering the API? Incremental fetch is cheap (it skips
  already-saved), but it is still N HTTP round-trips.
- **Expiry-notification UX.** Banner only? OS notification (we already have a
  notification surface for the watcher)? Both? How loud is too loud.
- **Corrupt-config writer gate.** Interactive fetch is gated by
  `_refuse_if_config_corrupt`; `scan_once` is intentionally NOT gated (it must
  run to prevent image-cache loss). A scheduled fetch **writes** conversation
  files, so it MUST respect the corrupt-config gate (writing to the wrong
  `data_dir` would orphan the archive). Resolve: the fetch pass checks the
  corrupt flag and skips (logging) when set, even though the enclosing
  `scan_once` is exempt.
- **Cowork rotation.** Confirm Cowork keeps everything on disk and has no
  image-cache-style rotation; if it does, that's a separate watcher surface, not
  part of this fetch feature.
- **First-run / never-authed users.** A user who has only ever used Claude Code
  has no Desktop credentials and never will. The skip path must be silent and
  normal for them, not a nagging banner.
- **Cross-workspace.** Fetch is scoped to the current login's org
  (`backend/routers/fetch.py:106` notes cross-workspace sync is future work);
  the scheduled fetch inherits that limit.
- **Interaction with the in-app Refresh lock.** Refresh uses
  `_refresh_in_progress` + an `asyncio.Lock` and returns 409 on concurrent
  runs. The scheduled fetch must take the same lock so a background tick and a
  user-clicked Refresh can't collide.

## 7. Acceptance sketch (for when this is built)

- With valid creds: a new Desktop conversation appears in the local archive
  within one backstop interval, no user action.
- With expired creds: the background pass logs an auth-skip, does **not** open a
  browser, and the app surfaces an actionable "re-authenticate" banner. A manual
  Refresh clears it and the schedule resumes on the next tick.
- With no creds (Code-only user): the pass skips silently; no banner, no log
  spam.
- With corrupt config: the fetch pass skips (logged); the image-cache and
  search-index passes still run.
- Background tick and manual Refresh never run concurrently (shared lock).
- A scheduled tick never triggers `capture_credentials` (test: stub capture,
  expire creds, assert it is never called from the watcher path).
