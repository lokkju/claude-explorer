# Slash Commands & Tool Calls — Display + API Contract Spec

**Status:** V1-readiness spec. Phase 1 deliverable. Written **without reading
implementation code** — drawn from user goals across the session, UX.md,
the JSONL data shapes, and the part-2 article. Phase 2 audits against
implementation; fixes are a separate task.

**Scope:** This document specifies the contract between the FastAPI
backend (`GET /api/conversations/{uuid}` → Pydantic `Message`) and the
React conversation viewer for slash-command markers, tool calls,
thinking blocks, compact markers, attachments, canned responses, and
prelude markers.

**Terminology note:** The user sometimes says "MCP server return." In
this codebase that means **the FastAPI backend's JSON response**. The
separate `claude-sessions` MCP server is out of scope.

**Guiding principles:**
- *One truth, three surfaces:* viewer, copy-as-markdown, export, and now
  search inclusion are all governed by the same toggle state.
- *V1 polish:* hide noise by default, reveal via an affordance.
- *Preserve user prose:* the user's actual words (including argful
  slash-command bodies) must remain visible and searchable.
- *Backend is the source of truth; frontend is the lens.* The backend
  returns lossless, structured data; the frontend decides how to
  fold/collapse/hide based on toggles and affordances.
- *No regression:* compact-marker pre/post messages must never be
  dropped; this regression has happened before.

## Shared Pydantic field reference

Fields used across multiple cases (proposed names — Phase 2 will verify
the implementation matches). Defaults shown.

| Field | Type | Default | Set on |
|-------|------|---------|--------|
| `uuid` | `str` | — | every message |
| `role` | `Literal["user","assistant","system"]` | — | every message |
| `content` | `str \| None` | required | every message (may be empty string) |
| `timestamp` | `datetime \| None` | `None` | every message |
| `is_command_marker` | `bool` | `False` | argless slash command rows ONLY |
| `slash_command` | `str \| None` | `None` | every message carrying a slash command (argless OR argful) |
| `slash_command_args` | `str \| None` | `None` | argful slash commands only |
| `is_prelude` | `bool` | `False` | leading contiguous markers before first real user message |
| `is_canned_response` | `bool` | `False` | assistant `"No response requested."` following `/exit` |
| `is_compact_marker` | `bool` | `False` | `type:"summary"` JSONL rows |
| `compact_summary` | `str \| None` | `None` | compact markers only |
| `tool_calls` | `list[ToolUseBlock]` | `[]` | assistant messages with `tool_use` blocks |
| `tool_results` | `list[ToolResultBlock]` | `[]` | user messages with `tool_result` blocks |
| `thinking_content` | `str \| None` | `None` | assistant messages with `thinking` blocks |
| `attachments` | `list[Attachment]` | `[]` | user messages with attachments |

**Why flat booleans rather than a nested `SlashCommand` value object?**
This spec describes the EXISTING contract for V1 polish, not a redesign.
The codebase already uses flat fields; the spec must be auditable
against today's code. A future v2 may consolidate, but a breaking schema
change is out of scope for V1.

**Why server-decided `is_prelude` rather than client-computed?** The
backend already iterates the message list to attach `parentUuid` /
branching metadata; computing `is_prelude` in the same pass is cheap
and avoids the client re-deriving on every render. The prelude position
is also exported (markdown/PDF) — letting the server own the truth
keeps all four surfaces (viewer, copy, markdown bundle, PDF) consistent
without re-implementing the rule in three places.

## Case 1: Argless Slash Command (`/exit`, `/clear`, `/resume`, `/compact`)

**JSONL shape (data):** a user `message.content` STRING containing the
literal text
`"<command-name>/exit</command-name>\n<command-args></command-args>"`
(args block empty or whitespace-only).

### Backend API contract
| Field | Value |
|-------|-------|
| `role` | `"user"` |
| `content` | `""` (empty string) OR `None` — body MUST NOT contain the raw XML triplet |
| `is_command_marker` | `True` |
| `slash_command` | `"/exit"` (the command name with leading slash) |
| `slash_command_args` | `None` |

### Frontend display
- Renders as a small inline **badge**, not a chat bubble.
- Visual register: subtle pill / chip, centered or left-aligned in the
  timeline, label = `slash_command` (e.g. `/exit`). Distinct from a
  user bubble.
- Star (bookmark) icon NOT shown on markers.
- Copy icon NOT shown on markers — there is nothing semantically useful
  to copy.

### Search contract
- Argless markers are **NOT indexed** (no semantic value).
- Specifically: when populating FTS5 body text for a message, skip any
  row where `is_command_marker=True` AND `slash_command_args` is
  `None`/empty.

### Export contract
- Markdown bundle / inline / PDF: **excluded** by default. Markers are
  conversational chrome.

### Bidirectional invariants
1. **No-leak invariant:** If `is_command_marker=True`, the `content`
   field MUST NOT contain the literal substrings `"<command-name>"` or
   `"<command-args>"`. (Auditable: load any conversation, scan content
   for these substrings on rows where `is_command_marker=True`.)
2. **Slash present:** If `is_command_marker=True`, `slash_command` MUST
   start with `/` and contain no whitespace, and MUST NOT be `None`.
3. **Role:** `is_command_marker=True` implies `role == "user"`.

## Case 2: Argful Slash Command (`/plan <prose>`, `/coding <task>`, `/metaprompt <prompt>`)

**JSONL shape:** user `message.content` STRING:
`"<command-name>/plan</command-name>\n<command-args>actual prose body — possibly multiple paragraphs and even with embedded `</command-args>` text from a paste</command-args>"`.

### Backend API contract
| Field | Value |
|-------|-------|
| `role` | `"user"` |
| `content` | the **prose body** (the args text, verbatim, multi-line preserved). Equivalently the user-visible body. The raw XML wrapper MUST be stripped. |
| `is_command_marker` | **`False`** |
| `slash_command` | `"/plan"` (the command name with leading slash) |
| `slash_command_args` | the prose body (mirror of `content` for callers that want the structured field) |

**Note on duplication of `content` ↔ `slash_command_args`:** both carry
the prose. This is intentional for V1 — frontends that want a unified
text stream read `content`; tooling that wants the structured args
reads `slash_command_args`. A future migration may unify these but for
V1 they MUST agree.

### Frontend display
- Renders as a **full user bubble** with the prose body as primary
  content.
- A small **`/plan` provenance badge** appears in the bubble header (or
  as an icon adjacent to the role label) — NOT in place of the bubble.
- Bookmark + copy icons are shown — this is real user content.

### Search contract
- Fully indexed: `content` body text goes into FTS5 like any user
  message.
- The `slash_command` name itself (e.g. `/plan`) is NOT indexed (users
  search prose, not command syntax).

### Export contract
- Markdown bundle / inline / PDF: **included**. The export may show a
  command provenance prefix like `> /plan` followed by the body, but the
  body MUST appear verbatim.

### Bidirectional invariants
1. **Mutual exclusion:** `is_command_marker=True` AND
   `slash_command_args is not None and len(slash_command_args.strip()) > 0`
   MUST NOT both be true on the same message. Argful commands are
   never markers.
2. **Prelude exclusion:** `is_prelude=True` AND
   `slash_command_args is not None and len(slash_command_args.strip()) > 0`
   MUST NOT both be true. An argful command carries real user prose and
   is never a system prelude.
3. **Body preservation:** the literal text inside `<command-args>...</command-args>`
   MUST appear in `content` with whitespace and newlines preserved.

## Case 3: `/exit` + Canned Assistant Response Fold

**JSONL shape:** a user `/exit` marker (Case 1) immediately followed by
an assistant message whose content is exactly `"No response requested."`
(allowing leading/trailing whitespace).

### Backend API contract

The two messages are returned as **two separate `Message` objects**
(not pre-merged). The backend adds metadata so the frontend can fold
visually:

| Field (assistant msg) | Value |
|-----------------------|-------|
| `role` | `"assistant"` |
| `content` | `"No response requested."` (preserved verbatim) |
| `is_canned_response` | `True` |

Detection rule: `is_canned_response=True` IFF (a) `role == "assistant"`
AND (b) `content.strip() == "No response requested."` AND (c) the
immediately preceding non-prelude message has `slash_command == "/exit"`.

### Frontend display
- The canned-response assistant message is **hidden** by default in the
  viewer (no bubble rendered at all). The preceding `/exit` badge is
  sufficient signal.
- Optional: a small "(no response)" inline note attached to the `/exit`
  badge for users who want explicit confirmation. The spec leaves this
  open; the requirement is that no full assistant bubble appears.

### Search contract
- **NOT indexed.** The canned response is filler; indexing it would
  pollute searches for the word "response" or "requested."

### Export contract
- **Excluded** from all exports (markdown, PDF, copy).

### Bidirectional invariants
1. **Strict trigger:** `is_canned_response=True` requires
   `content.strip() == "No response requested."`. Any other content
   (even close variants like "no response needed") MUST set the flag
   `False`. This is brittle by design — we'd rather miss-fold than
   accidentally hide a legitimate short reply.
2. **Role:** `is_canned_response=True` implies `role == "assistant"`.
3. **Causality:** `is_canned_response=True` implies the preceding
   non-prelude message has `slash_command == "/exit"`.

## Case 4: Leading Slash-Command Preludes

**Scenario:** A conversation begins with one or more contiguous
slash-command markers (typically argless: `/init`, `/clear`, etc.) before
the first real user message. CC sometimes injects these when starting a
new session or resuming.

### Backend API contract

For each message: set `is_prelude=True` IFF
- the message is at position `0` OR every prior message in the
  conversation also has `is_prelude=True`, AND
- the message has `is_command_marker=True` (an **argless** slash
  marker — argful prose is never a prelude).

The first message that violates the rule terminates the prelude;
**no subsequent message** in that conversation may have `is_prelude=True`.

| Field | Value on prelude row |
|-------|----------------------|
| `is_prelude` | `True` |
| `is_command_marker` | `True` (always — argful is excluded) |
| `slash_command` | populated |
| `slash_command_args` | `None` |

### Frontend display
- All contiguous `is_prelude=True` messages collapse into **one
  affordance** at the top of the conversation.
- Affordance copy: e.g. `"Show N system commands ▾"` where `N` is the
  prelude count.
- Default state: **collapsed**.
- Clicking expands; each individual marker then renders as a small
  badge (Case-1 visuals).
- An expanded prelude can be collapsed again with the same control.

### Search contract
- **NOT indexed.** Preludes are system noise.

### Export contract
- **Excluded** from all exports by default.

### Bidirectional invariants
1. **Contiguity-from-start:** the set of `is_prelude=True` messages MUST
   be a contiguous prefix of the conversation. Once any message has
   `is_prelude=False`, no later message may have `is_prelude=True`.
2. **Argful exclusion:** `is_prelude=True` AND
   `slash_command_args is not None and len(slash_command_args.strip()) > 0`
   MUST NOT both hold. Argful slash commands are never preludes.
3. **Marker implication:** `is_prelude=True` implies
   `is_command_marker=True`.

## Case 5: Tool Calls (`tool_use` + `tool_result`)

**JSONL shape:** assistant messages may contain `tool_use` content
blocks (`{type:"tool_use", id, name, input}`) and the next user
message contains the matching `tool_result` blocks
(`{type:"tool_result", tool_use_id, content, is_error}`).

### Backend API contract

The backend is a **lossless pipe** for tool calls. The Pydantic
`Message` model exposes structured fields:

| Field | Value |
|-------|-------|
| `tool_calls` | `list[ToolUseBlock]` on assistant messages; default `[]` |
| `tool_results` | `list[ToolResultBlock]` on user messages; default `[]` |

`ToolUseBlock`: `{id: str, name: str, input: dict}`.
`ToolResultBlock`: `{tool_use_id: str, content: str|list, is_error: bool}`.

The API **always returns** these blocks — pagination/truncation at the
API layer is forbidden.

### Frontend display
- **Hidden by default** in the viewer.
- A single **"Tools" toggle** in the conversation header controls
  visibility. When OFF: tool blocks are not rendered. When ON: each
  tool call renders as a collapsible block nested inline at its
  position in the timeline (alongside the assistant text), styled
  distinctly (monospace, muted border).
- A separate **"Expand / Collapse All Tools"** control appears next to
  the toggle when Tools is ON.

### Search contract — *one truth, three surfaces*
- The FTS5 index contains tool-call text (always indexed).
- At **query time**, the search dispatcher applies a filter: if the
  user's `showToolCalls` preference is `False`, results whose ONLY
  match is inside a tool block are dropped from the result list. If
  `showToolCalls=True`, those results are kept.
- **Why query-time, not index-time:** the toggle is per-user and per-tab
  reactive state. An index-time filter would require a rebuild on
  toggle flip. The article (part 2) explicitly promises "with Tools
  off, a hit you couldn't see in the viewer never shows up in the
  result list either" — a runtime decision.

### Export contract
- Markdown bundle / inline / PDF / copy-as-markdown: **mirrors the
  current Tools toggle state**. If toggle is OFF at export time, tool
  blocks are stripped from the export (and the
  `This block is not supported on your current device yet.` literal
  is also stripped per UX.md §7). If ON, tool blocks are included.

### Bidirectional invariants
1. **Lossless backend:** `len(tool_calls)` on the API response MUST
   equal the count of `tool_use` blocks in the source JSONL message.
   No backend filter, ever.
2. **Pairing:** every `tool_results[].tool_use_id` MUST match a
   `tool_calls[].id` somewhere earlier in the same conversation.
3. **Search-toggle symmetry:** for the same query Q, a message M whose
   ONLY match site is inside `tool_calls`/`tool_results` MUST appear
   in search results when `showToolCalls=True` and MUST NOT when
   `showToolCalls=False`. Conversely, a message M whose match is in
   regular `content` MUST appear regardless of toggle.

## Case 6: Thinking Blocks

**JSONL shape:** assistant messages may contain
`{type:"thinking", thinking:"..."}` content blocks.

### Backend API contract
| Field | Value |
|-------|-------|
| `thinking_content` | `str \| None` — concatenated text of all `thinking` blocks in the message, separated by `\n\n` if multiple. `None` if no thinking blocks. |

Thinking text MUST NOT be concatenated into `content` — they are
distinct fields.

### Frontend display
- **Not surfaced in V1 viewer by default.** Either:
  - (a) hidden entirely with no affordance, OR
  - (b) hidden behind the same "Tools" toggle (treat thinking as
    auditing content, same register as tools).
- The spec prefers (b): a single toggle for "show internals" reduces
  cognitive load. V1 polish ships with (a) acceptable if (b) is
  scheduled.
- When shown: render as a distinct, muted collapsible block
  immediately preceding the main assistant text, labeled "Thinking" or
  similar.

### Search contract
- Indexed in FTS5 alongside content.
- Query-time filter mirrors the Tools toggle (same rule as Case 5).

### Export contract
- Mirrors the Tools toggle state at export time (same as Case 5).

### Bidirectional invariants
1. **Field isolation:** `thinking_content` text MUST NOT appear as a
   substring of `content` on the same message.
2. **Role:** `thinking_content is not None` implies
   `role == "assistant"`.
3. **Whitespace preservation:** the original line breaks within each
   `thinking` block MUST be preserved.

## Case 7: Compact Markers (`type:"summary"`)

**JSONL shape:** a CC JSONL row with top-level `type: "summary"` (not
the normal `type: "user"` or `type: "assistant"`). Sometimes called a
"compact" event — emitted when CC auto-compacts a long conversation.

### Backend API contract
| Field | Value |
|-------|-------|
| `role` | (synthetic; spec leaves to implementation — could be `"system"` or a custom value) |
| `is_compact_marker` | `True` |
| `compact_summary` | the summary text from the JSONL row |
| `content` | the summary text (also exposed via `content` for unified rendering) |

Critically: the backend is a **dumb pipe** around compact markers. All
messages BEFORE and AFTER the marker are returned unchanged. The
regression history says pre-compact messages have been dropped in the
past; this spec pins that against recurrence.

### Frontend display
- Renders as an **inline timeline divider**, e.g. a horizontal rule
  with a centered label `"— Conversation compacted —"` and the summary
  text below it in muted style. Not a chat bubble.
- The surrounding pre- and post-compact messages render normally.

### Search contract
- The `compact_summary` text is **indexed** — it's user-visible content
  and often the only retrievable description of compacted history.
- Always searchable; the Tools toggle does not gate compact summaries.

### Export contract
- **Included** in all exports as an inline divider with summary text.

### Bidirectional invariants
1. **No drop:** for any conversation containing one or more compact
   markers, the total number of `Message` objects returned by the API
   MUST equal the number of non-summary JSONL rows PLUS the number of
   summary rows. (Auditable via fixture: feed JSONL with N rows around
   one compact marker, expect N+1 messages out — not fewer.)
2. **Order preservation:** the chronological order of messages around
   a compact marker MUST be preserved (compact marker appears at its
   original position, not relocated).
3. **Tools toggle independence:** a search result whose match is in
   `compact_summary` MUST appear regardless of `showToolCalls` state.

## Case 8: Desktop Attachments (multi-row fold)

**JSONL shape (Claude Desktop):** a conversation JSON file (not JSONL)
where a user message has a `files[]` array AND/OR adjacent
attachment-only rows precede the user's text row.

### Backend API contract
| Field | Value |
|-------|-------|
| `attachments` | `list[Attachment]` on user messages; default `[]` |

`Attachment`: `{uuid: str, file_name: str, mime_type: str, size_bytes: int, file_kind: Literal["image","document"]}`.

**Folding rule:** if the source contains one or more `attachment`-only
rows immediately followed by a `text` user row from the same speaker,
the backend folds them into a single `Message` with `attachments`
populated and `content` = the text row's body.

**Orphan rule:** if attachment rows are NOT followed by a text row, the
backend emits a single `Message` with `attachments` populated and
`content = ""` (empty string, NOT `None`, so frontend can distinguish
"empty content" from "missing field").

### Frontend display
- Single user bubble per folded group.
- Attachments render as inline thumbnails (gallery) per UX.md §11/§14.
- Multiple images: two-column grid of square tiles; `+N` overflow tile
  for 6+ images.
- Click thumbnail → full-screen lightbox (UX.md §14): arrow keys
  navigate the entire conversation's images.
- Image-not-in-cache fallback tile per UX.md §11.

### Search contract
- `Attachment.file_name` is indexed (searchable by filename).
- Binary content (image bytes / PDF text) is NOT indexed in V1.
- The Tools toggle does NOT gate attachment search results —
  attachments are primary content, not tool noise (per UX.md §11
  "Image attachments are deliberately *not* gated by that toggle").

### Export contract
- Markdown bundle: every attachment is included; image refs use
  bundle paths (`images/x.png`) or wikilinks for Obsidian
  (`![[images/x.png]]`).
- Inline markdown: image refs use backend URLs.
- PDF: image bytes embedded via WeasyPrint url_fetcher.

### Bidirectional invariants
1. **User-only:** `attachments` is non-empty implies `role == "user"`.
2. **Same-speaker fold:** the backend MAY fold attachment rows into a
   text row ONLY when both rows are from the same speaker
   (`role == "user"`) and contiguous in source order.
3. **Orphan empty string:** if attachments exist but there is no
   adjacent text row, `content` MUST be `""`, not `None`.

## Cross-cutting invariants (apply to all cases)

These invariants span multiple cases and are auditable globally:

| # | Invariant | Auditable via |
|---|-----------|---------------|
| X1 | `is_command_marker=True` AND `slash_command_args is not None and slash_command_args.strip() != ""` MUST NOT both be true. (Argful is never a marker.) | iterate all messages |
| X2 | `is_prelude=True` AND non-empty `slash_command_args` MUST NOT both be true. (Argful is never a prelude.) | iterate all messages |
| X3 | The substring `"<command-name>"` MUST NOT appear in any returned `content` field. (No raw XML triplets leak.) | scan all content |
| X4 | The substring `"</command-args>"` MUST NOT appear in any returned `content` field. (No raw XML triplets leak.) | scan all content |
| X5 | `is_prelude=True` messages form a contiguous prefix; no `is_prelude=True` appears after any `is_prelude=False`. | order check |
| X6 | For a conversation with N source JSONL rows producing M API messages, M = N − (attachment-fold count) − (canned-response hide count, if hidden at backend). Compact markers do NOT reduce the count. | row-count assertion |
| X7 | Search results respect the `showToolCalls` toggle state per Case 5/6 contracts; non-tool matches are unaffected by toggle. | search-equivalence test |
| X8 | Each surface (viewer, copy-as-markdown, markdown bundle export, PDF export) honors the same Tools toggle state. "One truth, three surfaces." | export-equivalence test |

## The greedy `<command-args>` regex bug (out of band but in spec)

**Reported observation:** In session `76fe578b-…`, some later `/exit`
rows produced malformed `content` like
`'</command-args>\n<command-name>/exit</command-name>...'` for `/exit`
markers dated 2026-04-21 and 2026-04-24.

**Root cause hypothesis:** the parser uses a greedy regex
`<command-args>(.*)</command-args>` with `DOTALL`. The greedy choice
was deliberate to handle user-pasted `</command-args>` text inside a
single block. But when a **single source row contains multiple
concatenated command blocks** glued together (which CC apparently emits
on rare occasions), the greedy match spans from the FIRST `<command-args>`
to the LAST `</command-args>`, swallowing intermediate `</command-args>`
+ `<command-name>` tags as args body.

**Spec for the fix (Phase 2 will validate, not implement):**

Replace the single-shot regex with a **state-machine parser** that:
1. Finds the next `<command-name>...</command-name>` (non-greedy or
   anchored at `<command-name>`).
2. After consuming the name, finds the matching `<command-args>...</command-args>`
   using a **bounded scan**: greedy WITHIN the current block, but stops
   if it encounters another `<command-name>` tag before
   `</command-args>` (signal: a new block started, so the current args
   block actually ended at the previous `</command-args>`).
3. Continues parsing from the consumed offset to capture additional
   blocks in the same row.

This preserves the original intent (handle user-pasted
`</command-args>` text inside a single block) while correctly handling
multi-block rows.

**Spec invariant for the fix:**

- A source string containing K concatenated triplets MUST produce K
  parsed `(name, args)` tuples — not 1 mega-block.
- The first `</command-args>` followed by `<command-name>` (with
  optional whitespace between) MUST terminate the current args block.

## WWCMM (What Would Change My Mind)

I would revise this spec if any of these become true:

1. **The flat-boolean shape is replaced.** If the team adopts a nested
   `SlashCommand` value object or a `message_type: Literal[...]` enum
   in the Pydantic model, the spec's field reference table and all
   case-level field rows must be rewritten. I picked flat booleans
   because that's the existing contract; if the team migrates, this
   spec migrates.

2. **Search inclusion moves to index-time.** If a future implementation
   decides to NOT index tool/thinking blocks at all (because rebuilding
   the FTS5 index on toggle flip is too expensive), the Case 5/6 search
   contract changes from "query-time filter on result rows" to
   "permanently excluded from index." The article's promise that
   "search respects the Tools toggle" still holds either way, but the
   invariants change.

3. **The greedy-regex evidence shows it's not just a multi-block issue.**
   If Phase 2 audit finds the malformed `/exit` rows come from a DIFFERENT
   parser bug (e.g. concatenated content from two JSONL rows merged at
   ingest, not a regex problem in `<command-args>` parsing), the proposed
   state-machine fix is wrong and the spec's fix section must be rewritten.

4. **Tool-toggle search behavior is contradicted by user expectation.**
   The article says with Tools off, tool-only hits never appear. If
   user testing reveals confusion ("I turned tools off but I'm still
   finding my old tool runs"), the spec may need to flip to index-time
   filtering instead, or expose a separate "search includes tool calls"
   toggle decoupled from viewer display.

5. **Compact-marker regression turns out to be a count issue, not a
   drop issue.** If Phase 2 finds the historical regression was that
   pre-compact messages had a different `parentUuid` rewriting (and
   were not actually dropped, just visually orphaned), invariant X6
   needs reframing as a graph-integrity check rather than a count
   check.
