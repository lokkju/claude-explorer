## The §5.15 contract (do not weaken it)

`frontend/e2e/fixtures.ts` (≈ lines 620-690) installs an **auto-fixture** that, for every test, captures `console.error` / `console.warning` / uncaught `pageerror` and **throws at teardown if any are non-empty**, modulo `PROJECT_CONSOLE_ALLOWLIST` plus a **per-test** allowlist:

```ts
test('legacy noisy thing', async ({ page, consoleAssertions }) => {
  consoleAssertions.allowlist.push(/known third-party warning/)
  // ...
})
```

This rule exists to catch genuinely-swallowed errors (the 2026-05-24 settings flash bug). **The fix for Category A below is NOT to blanket-allowlist `Failed to load resource`** — that would blind the guardrail. Each fix must allowlist only the *specific, expected* error the test deliberately provokes, and the test must still assert the user-visible error UI appears.

## Fix approach

### Step 1 — Category A (the 23)

Two viable shapes; pick one and apply consistently:

**Option 1 (recommended): a tiny scoped helper.** Add to `frontend/e2e/fixtures.ts` (exported) something like:

```ts
// Allowlist a specific, expected network-layer console error for an
// error-path test. Keep the regex tight so §5.15 still catches
// UNexpected swallowed errors.
export const expectNetworkError = (
  consoleAssertions: ConsoleCapture,
  match: RegExp,
) => consoleAssertions.allowlist.push(match)
```

Then in each test, immediately after arranging the failing mock:

```ts
consoleAssertions.allowlist.push(/Failed to load resource: .*\b404\b/)         // image / fetch 404s
consoleAssertions.allowlist.push(/net::ERR_CONNECTION_REFUSED/)                  // connection-status
consoleAssertions.allowlist.push(/Failed to load resource: .*\b504\b/)          // pdf-export
consoleAssertions.allowlist.push(/Failed to load resource: .*\b401\b/)          // redownload session-expired
```

**Option 2: scope it at the mock.** If the mock-setup helpers (`mockBackend`, the cc-image/attachment 404 mocks) are shared, have the helper that installs a deliberate-failure route also push the matching allowlist entry, so any test using that mock inherits the allowlist. Cleaner if most of these go through a few shared mock helpers; check `fixtures.ts` / `fixtures-defaults` first.

**Hard rules for Step 1:**
- Allowlist ONLY the status the test provokes; never a bare `/Failed to load resource/`.
- Do NOT remove or weaken any existing user-visible assertion. Each of these tests must still assert its error UI (fallback tile, toast, dialog, friendly copy). The allowlist change is additive.
- If, while doing this, you find a test whose console error is NOT explained by its own mock (i.e. a genuinely swallowed error), STOP and treat it as a real bug — that is exactly what §5.15 is for.

### Step 2 — Category B (the 3)

Debug each in isolation (`npx playwright test e2e/<spec> --reporter=line`). For B1 also apply the Step-1 allowlist. For B2/B3 confirm whether the assertion is stale (markup/selector drift) or a real behavior regression; fix the product code if it's a real regression, the test if the contract changed intentionally. Cross-check B2 against `useBracketCompactNav` and the compact-marker component; B3 against `ManageFiltersModal`'s delete-confirm dialog.

### Step 3 — Flaky bookmarks

Add the missing settle signal before the tablist assertion in `bookmarks.spec.ts:172`. Verify by running that spec ~5× in a row.

## Verification

1. Each fixed spec passes in isolation: `npx playwright test e2e/<spec>.spec.ts --reporter=line`.
2. Full suite green: `cd frontend && npx playwright test --reporter=line` → `0 failed`. Capture the exact `N passed` line. **Do not pipe through `tail`** — the pipe masks Playwright's exit code (a `tail` exit 0 hid a real failure during the article session). Read the output file or let it print.
3. No regression in the green suites: `uv run pytest -q` (expect 1139 passed / 1 skipped) and `cd frontend && npm run test:run` (expect 538 passed).
4. Sanity: confirm the §5.15 guardrail still bites — temporarily introduce a stray `console.error('boom')` in one component, confirm a test fails, then revert.

- The other CC session shares this working tree. To avoid index races, commit each logical fix with `git commit --only -- <paths>` rather than `git add -A`.
