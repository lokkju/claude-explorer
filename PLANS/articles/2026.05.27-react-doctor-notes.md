### F1, F2 — `no-mutable-in-deps` ×2 (false positive, suppressed)

- **File:line:** `frontend/src/contexts/SearchPinContext.tsx:132`
- **Rule:** `react-doctor/no-mutable-in-deps`
- **Catch:** Linter saw `[location.pathname, location.search]` in a
  `useEffect` deps array and warned that `location.*` is a mutable
  global whose changes don't trigger re-renders.
- **Who missed it:** N/A — Council didn't miss this; it's a false
  positive by React Doctor. The `location` identifier is bound to
  `useLocation()` from react-router-dom (reactive, triggers re-renders
  on navigation), not to `window.location` (the mutable global the rule
  targets).
- **Outcome:** Inline suppression comment explaining the carve-out, so
  a future reader knows the rule was considered and rejected with
  evidence. NOT a fix; we're documenting why the linter is wrong here.
- **Beat for the article:** false-positive triage is part of the
  workflow. The story isn't "linter is always right"; it's "linter
  surfaces fast, you triage, you suppress with evidence when wrong."
  Worth one sentence — the alternative (mute the rule globally) hides
  every future true positive of the same shape.

### F7, F8, F9 — `jsx-no-constructed-context-values` ×3 (REAL, wrapped in useMemo)

- **File:line:**
  - `frontend/src/contexts/FetchPipelineContext.tsx:218`
  - `frontend/src/contexts/SourceFilterContext.tsx:36`
  - `frontend/src/contexts/KeyboardNavigationContext.tsx:227`
- **Rule:** `react-doctor/jsx-no-constructed-context-values`
- **Catch:** Every `<Provider value={{...}}>` constructs a new object
  literal on every render of the parent. That breaks referential
  equality, which fires the entire subscriber graph of the context.
- **Who missed it:** ALL THREE Council members, across multiple review
  passes including the 2026-05-22 → 2026-05-23 search-typing-lag perf
  hunt that *specifically wrote* this invariant into project
  `CLAUDE.md` after the postmortem. The invariant is stated as a hard
  P0 rule in `PLANS/POSTMORTEM-search-typing-lag-2026-05-22.md` and the
  project CLAUDE.md "Performance Work" section #2. Despite that, three
  new Provider sites shipped without `useMemo` wraps over the next
  five days. **The Council reviewed each commit and missed each one.**
  React Doctor caught all three in one Oxlint pass.
  - **Why the Council missed them:** likely a recall problem. The
    invariant lives in CLAUDE.md (which the Council reads at session
    start), but it's one rule out of ~50, and the Council tends to
    focus its attention on the diff hunks under direct review rather
    than re-checking every Provider call site in the touched files.
    A deterministic linter has perfect recall on this exact rule.
- **Outcome:** Wrapped each Provider value in `useMemo` with explicit
  deps lists. KeyboardNavigationContext is the riskiest because
  `selectedMessageIndex` changes on every keystroke during message
  navigation; the project's invariant #1 says context-of-churning-
  provider in a list-rendered component re-renders every row, and
  this Provider's value includes ~30 fields that any future row
  component might subscribe to.
- **Beat for the article:** **This is the headline catch.** The
  invariant is documented. The postmortem is in the repo. The Council
  reads CLAUDE.md every session. *Three new violations still shipped
  in five days.* React Doctor — running for less than a second per
  file — caught all three.
  - The lesson isn't "the Council is bad"; it's that LLMs and static
    analysis have complementary recall profiles. LLMs win on the
    contested-judgment calls (the Decision Records in §4.1-4.7);
    deterministic rule engines win on the exhaustive-coverage calls
    (every Provider, every effect dep, every key-on-array-index).
    Run both.
  - **Parallel to receipt 4.8 (`/security-review` PDF HTML injection):**
    same shape, different tool. The Council shipped a missing
    `escape_html()` on three sites; `/security-review` caught it
    deterministically. The Council shipped three inline Provider
    values; React Doctor caught them deterministically. **Same lesson
    repeating across tools.**

### F10, F11, F12 — `no-direct-state-mutation` ×3 (false positive, suppressed)

- **File:line:**
  - `frontend/src/components/conversation/ConversationList.tsx:459` (post-filter clamp-to-top)
  - `frontend/src/components/conversation/ConversationList.tsx:552` (keyboard-nav pre-scroll)
  - `frontend/src/components/conversation/ConversationList.tsx:594` (URL deep-link scroll)
- **Rule:** `react-doctor/no-direct-state-mutation`
- **Catch:** Rule saw `scrollEl.scrollTop = X` where `scrollEl` is a
  `useState` slot, and warned that direct property assignment on
  state values is silently dropped (React only re-renders on a new
  reference).
- **Who missed it:** N/A — Council didn't miss this; it's a false
  positive. `scrollEl` is `useState<HTMLElement | null>(null)` — the
  *state slot* holds a DOM element reference. We're not mutating
  state; we're using the standard browser `Element.scrollTop` API to
  drive imperative scroll position. The state slot exists only so
  the virtualizer effect rebinds when the element appears.
- **Outcome:** Inline suppression per call site with the same kind
  of rationale used for the `react-hooks/incompatible-library`
  suppression already in the same file. NOT a fix; documenting why
  the rule is wrong here.
- **Beat for the article:** the false-positive sibling of F1-F2.
  Static analysis can't tell that the value in a state slot is
  a DOM element with its own mutable contract; pattern-matching
  on `<stateVar>.<prop> = X` flags both real bugs and intentional
  imperative-DOM. Triage cost: one careful re-read of the surrounding
  code per call site. Cheap because the code already had long block
  comments documenting why we bypass `virtualizer.scrollToIndex` and
  manipulate `scrollTop` directly.

### F13 — `no-render-in-render` ×1 (false positive, suppressed)

- **File:line:** `frontend/src/routes/ConversationPage.tsx:1413`
- **Rule:** `react-doctor/no-render-in-render`
- **Catch:** `renderBubbleRow(message, deps)` called inside JSX
  rather than instantiated as a component (`<BubbleRow ... />`).
  Rule says: extract to a named component for proper reconciliation.
- **Who missed it:** N/A — Council didn't miss this; extracting
  would be WORSE here. `renderBubbleRow` is already a top-level
  module function. Wrapping it in a real `<BubbleRow />` component
  to get reconciliation parity would require a `React.memo` wrap
  with custom comparator, but the `deps` object carries fields
  whose identity churns every render (`compactMarkerByUuid`,
  `conversation`, `messages` array, three callbacks). Every row
  would re-render anyway and we'd pay an extra memo-comparator
  cost per row, per render, for 4051 visible bubbles. The
  inner components (`<MessageBubble>`, `<CompactMarker>`) are
  already memoized — that's where the actual perf savings live.
- **Outcome:** Inline suppression with the multi-paragraph rationale
  above.
- **Beat for the article:** the "rule's heuristic is right in
  general, wrong for this specific architecture" class. Extract-to-
  component is the right move 95% of the time; the 5% case is a
  hot path where the props object can't be stabilized and the
  inner children are already memoized. Worth one paragraph,
  paired with F10-F12 as the false-positive cluster from this
  pass. Same triage discipline: read the code, write the rationale,
  suppress; don't blindly apply the recommendation.

### F14-F20 — `exhaustive-deps` ×7 (mixed: 3 ref-in-cleanup false-ish positives + 4 already-suppressed-under-a-different-rule-name)

- **File:line:**
  - `frontend/src/components/ConnectionStatus.tsx:79` (ref-in-cleanup)
  - `frontend/src/routes/ConversationPage.tsx:359` (ref-in-cleanup)
  - `frontend/src/hooks/useUnmountSafeTimer.ts:34` (ref-in-cleanup)
  - `frontend/src/components/filters/ManageFiltersModal.tsx:645` (already had `react-hooks/exhaustive-deps` disable)
  - `frontend/src/components/search/SearchPanel.tsx:155` (already had `react-hooks/exhaustive-deps` disable)
  - `frontend/src/contexts/FilterContext.tsx:292` (already had `react-hooks/exhaustive-deps` disable)
  - `frontend/src/routes/ConversationPage.tsx:869` (already had `react-hooks/exhaustive-deps` disable)
- **Rule:** `react-doctor/exhaustive-deps`
- **Catch:** Two distinct issues bucketed under one rule:
  1. Three effects whose cleanup function reads
     `<someRef>.current`. Rule message: "ref value will likely
     have changed by the time this cleanup runs; copy to a local
     variable inside the hook callback and use that in cleanup."
     The classic React warning.
  2. Four effects with deliberately-narrowed dep arrays that
     already carried a `react-hooks/exhaustive-deps` ESLint
     suppression. React Doctor uses its own rule namespace
     (`react-doctor/exhaustive-deps`) so the existing suppression
     didn't cover it.
- **Who missed it:**
  1. For the ref-in-cleanup three: the Council shipped each as
     a `useEffect` with empty deps that reads `<ref>.current`
     in cleanup. Following the rule literally would break the
     behavior: capturing the ref at effect-run time would
     snapshot `null` because the effect runs on mount BEFORE
     any timer / AbortController is created. The intent is
     always "clean up whichever resource is currently live at
     unmount." The Council reviewed each but didn't flag the
     latent footgun shape — these effects work today, but the
     shape would bite if the effect ever gained non-empty deps.
  2. For the four already-suppressed: a deduplication oversight.
     The project has accumulated `react-hooks/exhaustive-deps`
     disables with thorough rationales over multiple Council
     passes; React Doctor's rule has identical semantics but a
     different name, so each disable needed a sibling line for
     the new namespace.
- **Outcome:**
  1. Three ref-in-cleanup sites: inline `react-doctor/exhaustive-deps`
     suppression with rationale ("cleanup deliberately reads
     `<ref>.current` at unmount time; capturing at effect-run
     would snapshot `null`").
  2. Four already-suppressed sites: added sibling
     `react-doctor/exhaustive-deps` disable lines pointing back
     to the existing `react-hooks/exhaustive-deps` rationale.
- **Beat for the article:** two beats in one rule family.
  1. **The same rule under two namespaces** — when a project
     ships under two static-analysis tools with overlapping
     rule coverage, suppression comments don't carry across
     namespaces. A trivial-looking diff (add a sibling disable
     line) but one that surfaces only when you actually run the
     second tool. Cheap to fix, easy to miss without it.
  2. **The ref-in-cleanup shape is a real latent bug shape
     that React's own docs warn about** — and the Council
     shipped three of them in the codebase. They work today
     because the deps are empty so the effect only runs once,
     but if anyone later adds a dep to one of those effects
     without noticing the cleanup, the ref read becomes a
     race. Documenting WHY each shape is safe-as-written
     hardens the code against future drift. The Council
     could have caught these in principle but didn't, the
     same way it missed the inline `<Provider value={{...}}>`
     literals.

### F53 — Virtualizer-scroll-recovery in ConversationPage (2026-05-28)

- **File:line:** `frontend/src/routes/ConversationPage.tsx:680-704`
  (auto-scroll effect, post-fix line range)
- **Source:** the 2026-05-28 keyboard-nav smoke test (the run that
  produced receipt F52's "ready to push" verdict), NOT a React Doctor
  finding. The smoke test deferred it as "real UX edge case, not a
  Council regression." Today's session closes it.
- **Catch:** the auto-scroll effect looked up the target element by
  UUID in `messageRefs.current`. With the virtualizer in play
  (`@tanstack/react-virtual` over 600+ rows on long CC sessions),
  any jump that targets an off-screen row finds `undefined` in the
  ref map and silently no-ops. `Alt+>` (jump-to-last) and `Alt+<`
  (jump-to-first) both hit this: the target hasn't been mounted yet,
  so `scrollIntoView` is called on nothing. The selection state
  moved, but the viewport stayed where it was. Selection ring
  disappeared from view until the user scrolled by hand.
- **Who missed it:** no individual reviewer; this is a coverage gap
  in the test suite. The existing `Alt+<` and `Alt+>` test only
  checked that the URL stayed stable and the app didn't crash. The
  user-observable "did the viewport actually follow the selection"
  contract wasn't pinned. The smoke walk surfaced it during real
  keystrokes; automated tests had never run against a fixture long
  enough that the virtualizer unmounted off-screen rows.
- **Outcome:** when the ref lookup misses, fall back to
  `virtualizer.scrollToIndex(visIdx, { align: 'center' })`. The
  virtualizer scrolls without needing the target mounted; the row
  mounts via the normal path once the viewport reaches it. New
  Playwright spec `keyboard-nav-alt-jump-virtualizer-recovery.spec.ts`
  pins the round-trip Alt+> -> Alt+< contract with a 600-message
  fixture. RED -> GREEN in one cycle.
- **Beat for the article:** **automated tests can pass while UX is
  broken when the fixture isn't shaped like the production case.**
  The pre-existing `Alt+<`/`Alt+>` test ran against a synthetic
  short fixture where the first message was always mounted, so the
  ref lookup never missed and the bug never fired. The 600-message
  virtualization fixture had been used elsewhere (for DOM-count
  budget assertions) but had not been paired with keyboard nav.
  Crossing the two — long fixture + keyboard nav — reproduced the
  bug deterministically in 2 seconds. Generalizes to: **when adding
  perf-driven indirection (virtualization, lazy mounting, suspense),
  every UX flow that touches the indirection needs at least one
  test against a fixture large enough to exercise it.** The unit
  test budget is the wrong axis to grow; the integration-test budget
  is.
