## Fix strategy

Both sites need EITHER (a) two disables stacked correctly so each runner's directive is immediately above its target, OR (b) the effect extracted into a tiny shared hook that's exempt from the rule. The textbook "move the comment immediately above the diagnostic" can't satisfy BOTH runners simultaneously when each requires the disable on the line directly above the diagnostic; only one comment can occupy that slot.

### Preferred fix — Path (a): single combined comment

Try a single comment line that lists BOTH directives:

```ts
// react-doctor-disable-next-line react-doctor/no-adjust-state-on-prop-change -- ... | eslint-disable-next-line react-hooks/set-state-in-effect -- ...
setState('checking')
```

Some lint runners parse multi-directive lines correctly; others don't. Verify after applying:
- React Doctor: `npm run lint:react:diff` → expect 0 errors at that line.
- ESLint: `npm run lint` → expect 0 errors at that line.

If EITHER runner still reports the error, fall back to Path (b).

### Fallback fix — Path (b): extract a shared hook

Both sites are the same pattern: "when an open/isOpen prop flips true, run a side effect to seed local state from a stored source." Extract:

```ts
// frontend/src/hooks/useSeedOnOpen.ts
export function useSeedOnOpen<T extends unknown[]>(
  open: boolean,
  setters: (() => void)[],
  deps: T,
) {
  // ... single useEffect, single set of disable comments owned here
}
```

The two consumers (`FetchDialog`, `MarkdownExportDialog`) replace their hand-rolled effect with a `useSeedOnOpen(...)` call. Both lint runners' disables live in the hook ONCE; no displacement possible.

Decide between (a) and (b) by testing (a) first — if a single combined comment works, ship that (~10 LOC change). If not, ship (b) (~50 LOC including the hook + two call-site refactors + a vitest for the hook).

## Skill / agent hardening

The pattern this plan addresses — re-aligning a disable directive for runner X displaces an adjacent disable directive for runner Y — is not yet codified in `llm-council-coding.md`. Add a one-line addition to its "Disambiguating the right tools" section:

> When re-aligning a `<runner>-disable-next-line` directive, verify no OTHER runner's disable directive was sitting between the old position and the new position. If it was, that other runner's suppression is now misaligned. Re-run every configured lint runner after any disable-comment move.

This is the symmetric pair to the existing rule that says each lint runner needs its own disable. Existing rule: don't forget the other runner's disable. New rule: don't displace the other runner's disable.
