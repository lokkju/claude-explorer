# Search Scope Propagation — Spec

**Date:** 2026-05-14
**Status:** Draft → implementation
**Owners:** DS/ML Engineer (Claude Code) + Python Expert (Gemini-3-Pro) + CTO (Opus)
**Origin:** User feature request — title search and full-text search both ignore the sidebar's broader scope, so the right-pane search and the sidebar list disagree on which conversations exist.

## Direct quote (user)

> Just as the full-text search honors the Tools toggle, both the title search and full-text search should honor the active filter (if any) and the conversations dropdown (All Conversations…). And the Workspaces dropdown.

## TL;DR

Bring the sidebar list, the title search, and the full-text search (right pane) under one sidebar-scope contract. **What's not visible in the sidebar list MUST NOT appear in either UI search surface either.** MCP `list_sessions` gets a smaller invariant — see §3.

The composition is:

```
visible(conv) := active_filter(conv) AND source(conv) AND workspace(conv) AND pin_scope(conv) AND tools_toggle(message)
```

All five compose by intersection. Order doesn't matter. Each surface (sidebar list, title search, full-text search, MCP `list_sessions`) evaluates the SAME predicate on the SAME conversation set.

---

## 1. What "scope" means (ground-truth from current code)

Five composing filters, each already implemented somewhere; the gap is that title-search and full-text-search ignore (1), (3), and the workspace half of (4):

| # | Filter | Source of truth | Where it's applied today | Where it MUST also apply |
|---|--------|-----------------|--------------------------|--------------------------|
| 1 | **Active filter graph** (atoms+groups, hide / show-only) | `FilterContext.filtersState` (frontend); `filterEngine.applyActiveFilter` | Sidebar list only (client-side in `ConversationList.tsx:76`) | Title search, full-text search |
| 2 | **Source dropdown** (`all`/`CLAUDE_AI`/`CLAUDE_CODE`) | `SourceFilterContext.sourceFilter` | Sidebar list, full-text search (`/api/search?source=`), MCP `list_sessions(source=)` | Title search (frontend-side; already comes from `useConversations` which already honors source — so it's actually fine; we keep it explicit in tests) |
| 3 | **Workspace dropdown** (`organization_id`) | `SourceFilterContext.organizationId` | Sidebar list (`/api/conversations?organization_id=`); fed to `useConversations` | Title search, full-text search, MCP `list_sessions` |
| 4 | **Pin scope** (`conversation` or `project` pin) | `SearchPinContext.pinScope` | Full-text search (`/api/search?conversation_uuid=` or `?project_path=`); sidebar dimming only (does NOT exclude — by design) | (unchanged — pin already wired into full-text) |
| 5 | **Tools toggle** (`showToolCalls`) | `SettingsContext.showToolCalls` | Full-text search (`/api/search?include_tool_calls=`) and per-message rendering | (unchanged) |

Pin scope (4) is interesting: it deliberately *dims* but does not *hide* in the sidebar (`outOfScope` opacity-40 rows still render). Search treats pin as a hard filter. We keep that asymmetry — the user opted into "pin = narrow search to this scope", and the sidebar dim is a *navigational* hint. Search-pin scope already narrows further within the filtered set per the existing implementation; we MUST NOT break that.

**Composition rule:** all five are intersected. An empty intersection returns no results. Pin scope (4) narrows further within (1)+(2)+(3), it never broadens past them — this matches today's behavior and stays.

## 2. Backend contract (`/api/search`)

### What `/api/search` accepts today

From `backend/routers/search.py`:

```
q                    required, ≥1 char
source               "all" | "CLAUDE_AI" | "CLAUDE_CODE"   (already correct)
context_size         "snippet" | "full"
sort, sort_order     existing
conversation_uuid    pin-scope (single conv)               (already correct)
project_path         pin-scope (CC project)                (already correct)
bookmarks            comma-separated UUIDs                 (already correct)
include_tool_calls   bool                                   (already correct)
```

### What's missing

1. **No workspace filter.** `/api/search` does not accept `organization_id`. The sidebar list filters by `organization_id` via `/api/conversations?organization_id=`, so search and list disagree.
2. **No active-filter resolution.** The active-filter graph lives entirely client-side (`filterEngine.ts`). The backend has no concept of it.

### Decision: hybrid contract

**For workspace (3):** add a new `organization_id` query param to `/api/search`, plumbed all the way through `search_conversations` → `search_index.query` → SQL `WHERE` clause (or, on the linear-scan fallback, post-filter on `conv.get("organization_id")`).

  - Rationale: workspace is a single string; trivially pushable to backend; matches how `/api/conversations` already does it. Symmetric API surface.
  - Schema bump? The FTS5 index currently stores `source` and `project_path` as UNINDEXED columns but NOT `organization_id`. We add it as UNINDEXED, bump `SCHEMA_VERSION` (4 → 5), and let the lifespan task rebuild. Index size impact is negligible (one short string per row, UNINDEXED so no inverted-index cost).

**For active filter (1):** pass the **resolved set of `conversation_uuids`** that pass the active filter graph. Frontend computes this from the cached conversation list (which it already holds for the sidebar).

  - Rationale considered and rejected — sending the filter graph itself:
    - Glob/regex evaluation on the backend would duplicate `filterEngine.ts` in Python. Two implementations of the same predicate is exactly the bug we'd be creating.
    - The frontend already has the full conversation list in memory (the sidebar's `useConversations` call). Computing the passing-UUIDs set client-side is one `applyActiveFilter` loop over an array we already have — call it µs-scale.
  - Why MCP doesn't get this param:
    - MCP callers don't know the user's filter names. The filter graph is private to the user's preferences blob.
    - The "one truth" invariant is preserved by *intersection* on the common subset (source, project, organization_id). See §3.

### Transport: GET + POST hybrid

**Council finding (2026-05-14, both Gemini-3-Pro and Python Expert):** CSV-in-GET is unsafe at scale. 1500 UUIDs × 37 chars ≈ 55 KB query string blows past h11/uvicorn defaults (8 KB) AND past proxies/gateways. Even with an "omit if filter is no-op" shortcut, the "almost-all" case (filter excludes 1 of 1500) still ships 1499 UUIDs. Plus SQLite's `SQLITE_MAX_VARIABLE_NUMBER` (often 999) means a naive `IN (?, ?, ..., x1500)` clause errors on older builds.

**Decision:** the search endpoint supports BOTH:

1. **GET `/api/search?...`** — existing query-string contract; unchanged. Accepts `organization_id` (new) and `conversation_uuids` (new, CSV). The CSV param is intended for small scopes (manual curl, bookmarks-style ≤ ~100 UUIDs).
2. **POST `/api/search`** — new variant. Accepts a JSON body with the same fields as GET (`q`, `source`, `context_size`, `sort`, `sort_order`, `conversation_uuid`, `project_path`, `bookmarks`, `organization_id`, `conversation_uuids` as an actual JSON array, `include_tool_calls`). The frontend defaults to POST whenever it has a `conversation_uuids` list to send.

Both methods delegate to the same `search_conversations()` internals. The Pydantic body model + the Query params produce the same internal kwargs.

**New backend params (shared shape):**

```
organization_id      str | None              (matches /api/conversations)
conversation_uuids   list[str] | None        (UI passes the post-filter set;
                                              MCP does not use this. GET
                                              accepts as CSV; POST accepts
                                              as a JSON array.)
```

Both AND-compose with existing scope filters. `conversation_uuid` (singular, pin scope) remains "most-specific wins" — when present, it overrides `project_path`, `bookmarks`, AND `conversation_uuids`. `organization_id` always ANDs.

**Empty-set semantics:** if `conversation_uuids` is an empty list `[]` (the user has an active filter that excludes everything), return `[]` immediately. `None` (param absent) means "no constraint". `bookmarks` already does this same split.

### Backend implementation

#### Linear-scan path

The linear-scan path (`_search_via_linear_scan`) already iterates conversations and gates each one against `source`/`project_path`/`bookmarks`. We add the two new gates symmetrically:

```python
if organization_id is not None and conv.get("organization_id") != organization_id:
    continue
if conversation_uuids is not None and conv.get("uuid") not in conversation_uuids:
    continue
```

(`conversation_uuids` typed as `set[str] | None` here for O(1) lookup.)

#### FTS5 fast path — TEMP TABLE join, NOT post-filter

**Council finding (Python Expert, 2026-05-14):** post-filtering the FTS5 result set in Python after `LIMIT 5000` is a CORRECTNESS BUG identical in shape to the documented `include_tool_calls=False` drift. If the active filter excludes 95% of conversations, the top-5000 by bm25 could all be filtered out while real matches sit at rank 5001+. Index limit + post-filter = silent drops.

**Decision:** push the `conversation_uuids` gate into SQLite using a TEMP TABLE join. The pattern:

1. Per-query, on the read connection: `CREATE TEMP TABLE IF NOT EXISTS allowed_conv(uuid TEXT PRIMARY KEY)`.
2. `DELETE FROM allowed_conv` then `executemany("INSERT INTO allowed_conv VALUES (?)", [(u,) for u in conversation_uuids])`.
3. Augment the existing query: `... WHERE messages MATCH ? AND conv_uuid IN (SELECT uuid FROM allowed_conv) [AND ...] LIMIT ?`.
4. bm25 ordering preserved; LIMIT applies AFTER the conv_uuid filter, so we get the top-5000 ranked hits *within the allowed set*.

The TEMP table is per-connection (threading.local read conn), so DROP+CREATE per call is safe across threads and idempotent within a thread. SQLite's TEMP-table semantics make this near-zero-cost; the `PRIMARY KEY` gives O(log n) lookup for the JOIN.

`organization_id` goes into SQL directly — single UNINDEXED column equality. FTS5 SCHEMA_VERSION bumps 4 → 5 to add the column; the lifespan task rebuilds non-blockingly (confirmed: `backend/main.py:253` uses `asyncio.to_thread`).

The title-sweep SQL (`SELECT DISTINCT conv_uuid FROM messages WHERE title LIKE ?`) gets parallel gates for `organization_id` and the TEMP table join.

## 3. MCP contract (`list_sessions`)

### Current signature

```python
list_sessions(
    query: str | None,
    source: str | None,          # "CLAUDE_AI" | "CLAUDE_CODE"
    project: str | None,         # substring, case-insensitive on project_name
    limit: int = 20,
    offset: int = 0,
)
```

### Decision: ADD `organization_id`; DO NOT ADD `active_filter`

**Add `organization_id`** — the workspace filter is a stable backend concept (UUID), mirrors `/api/conversations` and the new `/api/search?organization_id=` param. MCP callers (other Claude sessions) can reasonably know their own org UUID. Symmetric with the UI.

**Do NOT add `active_filter`** — the user's filter names ("Crons", "scratch", whatever) are private to that user's preferences blob. An MCP caller has no way to know them or resolve them. The filter graph itself is also not stable across users. Exposing it would either:

- (a) require MCP callers to learn a private vocabulary — friction with no payoff;
- (b) require the MCP server to read `~/.claude-explorer/preferences.json` and resolve the active filter from there — possible, but the value-add over "the human is already filtering in the UI" is near zero.

**Why the asymmetry is fine:** the LLM Council's "one truth, three surfaces" invariant is preserved by *intersection*, not by *full symmetry*. The MCP surface offers a SUBSET of the UI's filters — `source` + `project` + (new) `organization_id`. Any query the UI can express, MCP can express via passing the same `source` and `organization_id`. The active-filter case is a **UI convenience overlay** on top of that common subset.

**Regression guard (test-encoded):** MCP `list_sessions(source=X, project=Y)` and `/api/search?source=X&project_path=…` must return the same conversation set for the same query — that's the parity invariant for the common subset.

## 4. UI propagation

### Single source of truth

The sidebar already owns:
- `useSourceFilter().sourceFilter`
- `useSourceFilter().organizationId`
- `useFilters().filtersState` (the active-filter graph)

The right-pane SearchPanel currently subscribes to `useSourceFilter().sourceFilter` (and `useSearchPin()`, `useSettings().showToolCalls`). We extend it to also subscribe to `useSourceFilter().organizationId` and `useFilters().filtersState`.

### How we compute `conversation_uuids` client-side

`SearchPanelContext` already has access to the source filter and pin scope. It does NOT today have access to the conversation list or to the filter graph. We wire this in:

1. **Get the conversation list.** `useConversations({ source, organization_id })` returns the post-source, post-workspace list. (This is exactly what the sidebar uses.)
2. **Apply the active filter.** Use `applyActiveFilter(conv.name, filtersState)` — same predicate the sidebar uses in `ConversationList.tsx:76`. Get the passing set as `string[]`.
3. **Pass to `useSearch`.** Either:
   - (a) pass the array as a new `conversationUuids: string[]` field on the existing `scope` argument; OR
   - (b) include it in the `useSearch` queryKey + the `api.search` URLSearchParams.

We go with (a) — the `scope` object is already the "narrowing knobs for search" parameter bundle. Adding `conversationUuids` to it keeps the existing shape pattern.

### The "no-op short-circuit" optimization

```ts
const passingUuids = useMemo(() => {
  if (!conversations) return undefined
  // Active filter check is the same predicate the sidebar uses.
  const passing = conversations.filter(c => applyActiveFilter(c.name, filtersState))
  // No active filter OR filter passes everything → don't constrain.
  // Note: this short-circuit only affects whether we send the param;
  // it does NOT mask the "almost all" case (1499/1500). For that we
  // rely on the POST transport — sending 1499 UUIDs over GET CSV is
  // unsafe; over a POST JSON body it's fine. See §2 Transport.
  if (passing.length === conversations.length) return undefined
  return passing.map(c => c.uuid)
}, [conversations, filtersState])
```

`undefined` means "no constraint" (param absent); `[]` means "filter excludes everything" (backend short-circuits to empty results); a non-empty array means "constrain to these UUIDs".

### Title search

The title-search input lives in `Sidebar.tsx` and filters via `useConversations({ search: query })` (a client-side `name`/`project_path` substring filter inside `useConversations`). Today, that hook already takes `source` and `organization_id`. The active-filter is applied AFTER the hook returns, in `ConversationList.tsx:76`.

**Title-search scope correctness** is already structurally correct: title search filters the SAME list the sidebar renders. The user-visible bug is that **the title input's results in the sidebar list match expectations**, but full-text-search results don't — because full-text-search goes to a different endpoint that doesn't see the same filters.

We will add a test that pins the title-search behavior (substring on `name` OR `project_path` of conversations that pass source + workspace + active-filter), so we don't accidentally regress it when refactoring.

### Auto re-run on scope change

`useSearch` already uses a `queryKey` that includes every variable that affects the query. Adding `conversationUuids` and `organization_id` to that key means a sidebar filter change automatically re-fires the search. **No manual re-issue needed.** This satisfies the spec invariant "toggling a filter off restores hidden results without manual re-issue".

## 5. Invariants (ready for tests)

**I1. Visibility-set parity.** For any query `q`, the set of conversations returned by `/api/search?q=q&{scope}` equals `{conv | conv is visible in sidebar list under {scope}}`. ("Visible" means rendered, not dimmed.)

**I2. UI–MCP common-subset parity.** For any `(q, source, organization_id)` that both surfaces accept, `/api/search?q=&source=&organization_id=&project_path=P` returns the same conversation set as `list_sessions(query=q, source=source, project=basename(P), organization_id=organization_id)` (modulo MCP's `project` being a substring match on the project NAME, while `/api/search?project_path=` is an exact match on the project PATH — see invariant I7).

**I3. Active-filter sidebar↔search parity.** If an active filter excludes conversation `C`, then `C` does not appear in either the sidebar list OR any search surface results, for any query.

**I4. Restoration.** Toggling an active filter from "filter X is active" to "no filter" does NOT require a manual re-search. The search re-fires automatically because its queryKey changes.

**I5. Pin-scope narrows-further.** With both an active filter AND a conversation/project pin, results are the intersection. The pin never broadens past the filter.

**I6. Tools-toggle scope unchanged.** The `include_tool_calls=false` semantics are message-level, not conversation-level; they remain orthogonal to all four conversation-level scope filters.

**I7. MCP project-substring vs path-exact.** `list_sessions(project="foo")` continues to match project NAMES by substring (existing contract); we do NOT change this. Tests document the difference where it matters.
